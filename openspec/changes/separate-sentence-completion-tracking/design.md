## Context

当前服务端学习状态里，`practice_sentence` 既承担“用户已经进入句子练习”的阶段推进作用，又通过 `practiced_sentence_count` 间接被消费成“句子已有完成结果”的近似信号。与此同时，前端刚刚完成了一轮解释层收口，已经把“进入句子练习”“句子完整复现”“完成当前场景练习”区分开来。

这造成了一个新的不一致：

- 前端 UI 解释：进入练习 != 句子完成
- 服务端状态语义：`practice_sentence` 仍会推进 `practiced_sentence_count`
- `today` / `continueLearning` / scene session done 仍依赖这套旧服务端近似信号

如果不继续往下收口，项目会进入“前端解释是新的，服务端真相还是旧的”的状态。

## Goals / Non-Goals

**Goals:**

- 让服务端显式区分“进入句子练习”“句子完成”“场景练习完成”三类学习结果。
- 调整场景聚合与 dashboard 聚合逻辑，保证 `today`、`scene` 和服务端统计消费同一套完成语义。
- 让 session done、scene done 与变体解锁建立在更准确的句子完成/场景练习完成之上，而不是建立在“做过一次练习”之上。
- 为兼容历史记录提供安全回退方案，避免旧数据直接失真。

**Non-Goals:**

- 本次不新增新的练习题型或练习 UI。
- 本次不调整 review 策略或推荐算法。
- 本次不重构整套学习数据库模型，只处理当前已确认混用的字段和聚合口径。
- 本次不扩大到 chunks、audio 或缓存链路。

## Decisions

### 1. 把“进入句子练习”和“句子完成”视为两类独立事件

原因：
- `practice_sentence` 只能证明用户已进入句子练习，不足以证明该句已经达到 `complete` 里程碑。
- 如果继续复用一个字段，会让后续 done 判定和 today 聚合始终带有误差。

备选方案：
- 继续保留现有字段，只在前端解释层做换算。
  不采用，因为服务端仍会持续产出带歧义的聚合结果。

### 2. 场景 done 继续依赖场景练习完成，但不能再依赖“已进入句子练习”这一弱信号

原因：
- scene done 是更高层级的闭环状态，必须建立在比 `practice_sentence` 更强的信号上。
- 这样才能让 continue learning 和 today 入口的状态推进真正可信。

备选方案：
- 完全取消 `practiced_sentence_count` 对场景 session 的影响。
  暂不直接采用，因为兼容期仍可能需要保留“已经进入过句子练习”的阶段信息。

### 3. 兼容期优先增加更明确的句子完成解释层，而不是直接删除旧字段

原因：
- 现有数据库和历史数据已经写入 `practiced_sentence_count`、`practice_sentence`。
- 直接删除旧字段风险大，且会影响旧记录读取。

备选方案：
- 一步到位清理旧字段。
  不采用，因为迁移和回退成本太高。

### 4. `today` / `continueLearning` 聚合必须基于新的服务端句子完成语义重新解释

原因：
- 这两个入口是用户最容易感知状态跳变的地方。
- 如果聚合层不跟着一起调整，链路真相依然会分裂。

备选方案：
- 只改 scene，不动 today。
  不采用，因为聚合入口正是这类语义漂移最容易暴露的地方。

## Risks / Trade-offs

- [风险] 历史记录没有显式“句子完成”数据，可能需要在兼容期用练习快照或聚合规则推导。
  -> 先把新语义字段或解释层补齐，旧数据按保守方式回退，避免把“已开始”误判成“已完成”。

- [风险] 会同时影响服务层、API 层、today 聚合和 scene 页，回归面较大。
  -> 把测试拆成服务层、API、today、scene 四类，并先覆盖最关键的状态升级场景。

- [风险] done 判定调整后，部分用户历史进度可能在短期内显示得更保守。
  -> 在设计上优先保证语义准确，必要时补兼容说明或迁移脚本。
