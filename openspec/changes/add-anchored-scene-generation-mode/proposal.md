# 变更提案：为生成场景增加句子锚点模式

## Status
draft

## 任务分类
Spec-Driven Change

## Why
当前“生成场景”只支持用户输入一个泛化的情境描述，再生成一个可练习的小场景。  
这条链路能覆盖“我想练咖啡店点单”“我想练礼貌拒绝”这类意图，但无法稳定承接另一类高频需求：

- 用户先记住一句印象深刻的英语台词或表达
- 希望 AI 围绕这句原话生成一个短场景
- 希望这句原话成为本次场景的记忆锚点，而不是只被模型“理解意思后大概带过”

当前问题不在于“完全不能生成”，而在于语义边界不清：

- 前端入口仍然把输入统一表述为“场景方向”
- API 只有 `promptText`，没有声明用户是在给“情境描述”还是“锚点句子”
- 生成 prompt 没有要求“必须围绕该句组织场景”或“必须原样保留该句”
- 相关语义目前没有正式 spec，后续实现、测试和文档都容易继续漂移

用户价值：

- 让“看到一句印象深刻的英语 -> 立即生成可练习小场景”成为一条稳定路径
- 降低用户把零散台词转成练习素材的成本
- 让当前场景生成功能更贴近真实英语学习和记忆迁移场景

## What Changes
- 在现有“生成场景”入口内增加模式切换，而不是新开独立入口
- 增加两种明确模式：
  - `context`：按情境生成
  - `anchor_sentence`：按句子生成场景
- 在 `anchor_sentence` 模式下，为生成链路增加“句子锚点”约束：
  - 输入主句必须作为本次生成的核心句
  - 生成结果必须包含该句原文或系统允许的明确变体策略
  - 场景仍需保持短小、自然、可练习
- 扩展场景生成请求契约、prompt 契约和最小测试基线
- 为该能力补齐正式 spec delta，避免“只是 UI 切换，实际语义不稳定”

## Stability Closure
### In This Round
- 收口当前场景生成入口“只有一个自由 prompt，语义过宽”的边界漂移
- 收口“句子驱动生成”没有正式契约、只能靠实现猜语义的问题
- 收口后续测试无法判断“是否必须保留锚点句”的缺口
- 明确该能力继续复用现有场景生成主链路，不拆新页面、不并行制造第二套入口

### Not In This Round
- 不处理“同一句锚点句一次生成多个场景变体”
  - 原因：这会扩大到生成批量策略和结果筛选，超出本轮最小方案
- 不处理“自动从电影台词出处、情绪标签、人物关系中抽取结构化元数据”
  - 原因：当前最小能力只需要先稳定承接用户输入的一句英文
- 不处理“锚点句生成后自动进入表达保存、重点句标记、复习计划联动”
  - 原因：会扩大到学习资产沉淀与 review 信号设计，不适合和本轮一起推进

### Risk Tracking
- 延后原因：优先先把入口模式、API 契约和生成保证做稳定
- 风险记录位置：本 change 的 `design.md`、`tasks.md`

## Scope
### In Scope
- Scenes 页面生成 sheet 的模式切换与输入文案分流
- 场景生成请求参数增加模式语义
- 服务端生成 prompt 按模式分支
- `anchor_sentence` 模式下的最小结果保证
- 相关交互测试、请求 schema 测试或服务测试补齐
- 对应 OpenSpec delta

### Out of Scope
- 独立“台词卡片”页面或新的生成入口
- 批量生成多个候选场景
- 电影来源识别、人物识别、情绪自动标注
- 新的持久化字段或数据库表改造

## Impact
影响的规范：
- 新增 capability：`scene-generation-contract`

影响的模块：
- `src/components/scenes/generate-scene-sheet.tsx`
- `src/lib/utils/scenes-api.ts`
- `src/lib/server/request-schemas.ts`
- `src/app/api/scenes/generate/route.ts`
- `src/lib/server/scene/generation.ts`
- `src/lib/server/prompts/scene-generate-prompt.ts`
- Scenes 页面相关测试

是否涉及 API 变更：是  
是否涉及前端交互变化：是  
是否影响缓存策略：轻微影响，主要体现在生成缓存 key 需纳入模式语义  
是否影响测试基线：是  
兼容性：向后兼容，旧调用默认视为 `context` 模式  

风险点：
- 如果只加 UI 切换，不补 API 和 prompt 约束，能力会继续模糊
- 如果要求“必须原句保留”过死，可能降低部分生成自然度，需要先定义最小可接受边界
- 如果模式命名和文案不清，用户仍可能混淆“情境生成”和“句子生成场景”
