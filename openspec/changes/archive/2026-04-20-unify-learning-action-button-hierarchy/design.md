## Context

当前仓库已经有统一按钮基础类：

- `APPLE_BUTTON_STRONG` 对应主按钮
- `APPLE_BUTTON_BASE` 对应次按钮
- `Button` 组件也有 `app-button-primary / secondary` 语义

但学习主链路里并没有稳定执行这套层级。当前已暴露出的漂移点主要有三类：

1. 主动作漂移
   - `lesson` 里的“加入复习”曾用局部主色
   - `chunks` 里的“开始复习”是主按钮
   - `scenes` 里的“生成场景”又是另一套主色入口
2. 次动作漂移
   - `保存到表达库 / 保存并加入复习 / 保存句子 / 添加到表达库` 在不同页面混用了白底、浅色底、无边框小圆按钮
3. 页面特例不断叠加
   - 一些页面直接内联 class，而不是复用已有主次按钮语义，导致 AI 或维护者很难判断是否属于“主动作”

这次改动不改变学习流程语义，只收口视觉层级，因此更适合做“跨模块但低业务风险”的 UI 规范统一。

## Goals / Non-Goals

**Goals:**
- 为学习主链路建立可执行的按钮层级规则：
  - 主动作：统一使用主按钮视觉
  - 次动作：统一使用白底次按钮视觉
  - 图标按钮：保持 icon-only，但不得伪装成主动作
- 把当前已经漂移的学习链路按钮统一回同一套层级。
- 让后续开发在看到“加入复习 / 开始复习 / 生成场景 / 保存到表达库”时，能够直接根据动作语义选择样式，而不是按页面记忆。
- 补充最小测试，防止回漂。

**Non-Goals:**
- 不对后台、营销页、设置页做整站视觉翻修。
- 不修改按钮文案、业务跳转、提交逻辑或状态机。
- 不在本轮引入新的 design token 或统一所有非按钮控件。

## Decisions

### 1. 以“动作语义”而不是“页面归属”决定按钮样式

决定：
- 主动作统一收敛为主按钮样式，覆盖：
  - `开始复习`
  - `加入复习`
  - `生成场景`
  - 与当前学习步骤强绑定的唯一主要 CTA
- 保存、补录、加入库等非立即推进学习主步骤的动作统一为次按钮样式，覆盖：
  - `保存到表达库`
  - `保存并加入复习`
  - `保存句子`
  - `添加到表达库`

原因：
- 同一动作类型在跨页面切换时必须给用户稳定预期。
- 这比“某页一直蓝、某页一直白”更符合闭环产品的操作语义。

备选方案：
- 继续按页面分别定义。放弃，因为会继续漂移。
- 统一所有按钮都改主色。放弃，因为会丢失主次层级。

### 2. 优先复用现有 `app-button-primary / secondary` 语义，不新增新 token

决定：
- 本轮优先把局部手写 class 改回已有按钮体系，必要时只做少量补充 class 组合。

原因：
- 现有基础类已经足够表达主次动作。
- 新增一套 token 会扩大范围，超出本轮最小收口。

备选方案：
- 新建 `learning-primary-button`、`learning-secondary-button` 等新类。暂不采用，除非实现时发现现有基础类无法承接。

### 3. 只收口学习主链路页面，不扩散到 admin / marketing

决定：
- 本轮实现范围锁定 `scenes / scene / lesson / chunks / review / today` 相关学习界面。

原因：
- 用户当前指出的问题集中在学习闭环。
- 后台和营销页虽然也有按钮，但不属于学习动作语义，贸然统一会扩大验证成本。

### 4. 用交互测试锁住关键按钮层级

决定：
- 不做全站截图测试。
- 通过现有 interaction tests 补 class 断言和按钮存在性断言，锁住关键页面的主次按钮语义。

原因：
- 当前仓库已有稳定的 interaction 测试结构。
- 用最小断言就能避免“改着改着又回到局部特例”。

## Risks / Trade-offs

- [风险] 某些页面看起来“更像营销 CTA”，改成统一主按钮后视觉会变得更克制
  → Mitigation：只收学习主链路，且以“动作语义”优先，避免误改展示型按钮。

- [风险] 一些局部页面用的是手写 class，统一时容易漏改
  → Mitigation：先按关键词检索主次动作入口，再用 interaction tests 锁住已知页面。

- [风险] 仅改学习主链路，短期内后台/营销页仍可能保持不同按钮风格
  → Mitigation：在 proposal / tasks 中明确这是本轮刻意不收范围，而不是遗漏。

## Migration Plan

1. 先在 OpenSpec 中定义学习动作按钮层级规则。
2. 盘点学习主链路中已知漂移按钮，按“主动作 / 次动作 / 图标动作”分类。
3. 逐页替换为统一样式：
   - scene detail
   - lesson detail
   - chunks focus detail
   - chunks 手动录入
   - scenes 生成入口
4. 补最小 interaction tests。
5. 记录 `docs/dev/dev-log.md`，如需要再补对应实现文档说明。

## Open Questions

- 是否需要把 `today` 和 `review` 中所有“去做回忆 / 继续学习”按钮也显式纳入同一轮校验。
- 如果实现时发现 `app-button-primary / secondary` 仍不足以表达学习按钮层级，是否要在本 change 内补一个共享 helper，而不是继续内联 class。
