## ADDED Requirements

### Requirement: Review 调度必须保守消费变体改写与目标表达覆盖信号
系统 MUST 允许新增的变体改写完成状态与完整输出目标表达覆盖结果参与待复习项排序和下一次复习节奏细调，但必须继续保留 `again / hard / good` 的主方向。新增信号不得单独主导全部调度结果。

#### Scenario: due 列表包含未命中目标表达的条目
- **WHEN** 多条表达同时到期，且某条最近一次完整输出为未命中目标表达
- **THEN** 系统 MUST 能将该条目标记为需要补完整输出或目标表达覆盖的调度关注点
- **AND** 该关注点 MUST 可以让条目相对普通中性条目更优先出现
- **AND** 页面 MUST 使用服务端提供的稳定解释，而不是自行拼接原始字段

#### Scenario: due 列表包含未完成变体改写的条目
- **WHEN** 多条表达同时到期，且某条最近一次 review 未完成变体改写
- **THEN** 系统 MUST 能将该条目解释为仍缺少迁移训练
- **AND** 系统不得把它与已经完成变体改写且完整输出覆盖目标表达的条目视作同等训练深度

#### Scenario: 调度器处理历史空值
- **WHEN** due 列表中存在缺少新增正式信号的历史记录
- **THEN** 系统 MUST 对历史空值采用保守中性解释
- **AND** 不得把缺字段历史记录错误推断为未完成迁移或未命中目标表达

### Requirement: Review 下一次复习节奏必须区分目标表达覆盖结果
系统 MUST 在计算下一次复习时间时区分完整输出已覆盖目标表达、完整输出未覆盖目标表达和完整输出未开始，但该细调 MUST 服从最终 feedback 的主方向。

#### Scenario: 用户提交 good 但完整输出未命中目标表达
- **WHEN** 用户提交 `good`，但本次完整输出未覆盖目标表达
- **THEN** 系统不得把该条目安排到与已覆盖目标表达且输出信心高的条目同等远的下一轮
- **AND** 系统 MUST 保留 `good` 的整体正向反馈含义

#### Scenario: 用户提交 hard 或 again 且未完成迁移训练
- **WHEN** 用户提交 `hard` 或 `again`，且本次未完成变体改写或完整输出目标表达覆盖
- **THEN** 系统 MUST 保守收紧下一次复习节奏
- **AND** 页面解释 MUST 指向输出或迁移训练不足，而不是只展示笼统失败
