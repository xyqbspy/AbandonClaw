## ADDED Requirements

### Requirement: Review 变体改写阶段正式化后必须保持训练降级边界
系统 MUST 在普通表达 review 的变体改写阶段记录用户是否完成固定方向改写，但不得把未完成改写直接等同于本次 review 失败。变体改写正式信号只能说明用户是否进入迁移训练，最终队列推进仍 MUST 由最终 feedback 和正式调度规则共同决定。

#### Scenario: 用户完成变体改写后提交 review
- **WHEN** 用户在普通表达 review 中选择固定改写方向并提交非空改写草稿
- **THEN** 页面提交 MUST 带上变体改写完成状态和对应固定方向 id
- **AND** 后端 MUST 能把该状态记录为本次 review 事件的正式信号
- **AND** 页面不得要求该改写通过 AI 评分后才能继续最终 feedback

#### Scenario: 用户跳过或未完成变体改写
- **WHEN** 用户未完成变体改写但继续走到最终 feedback
- **THEN** 系统 MUST 把该状态解释为迁移训练未完成
- **AND** 不得直接把它解释为本次 review 失败
- **AND** 页面 MUST 继续提供参考、回退或最终反馈入口

### Requirement: Review 完整输出阶段必须区分完成状态与目标表达覆盖
系统 MUST 在完整输出阶段继续记录用户是否完成输出，并额外区分完整输出是否确定性覆盖目标表达。页面 MUST 把目标表达覆盖解释为输出训练信号，而不是语法、自然度或完整质量评分。

#### Scenario: 用户完整输出包含目标表达
- **WHEN** 用户提交的完整输出经过确定性归一化后包含目标表达
- **THEN** 系统 MUST 记录完整输出已完成
- **AND** 系统 MUST 记录目标表达覆盖为已命中
- **AND** 后续页面说明 MUST 避免把该结果称为 AI 质量评分

#### Scenario: 用户完整输出未覆盖目标表达
- **WHEN** 用户提交了非空完整输出但未确定性覆盖目标表达
- **THEN** 系统 MUST 记录完整输出已完成
- **AND** 系统 MUST 记录目标表达覆盖为未命中
- **AND** 页面 MUST 将其解释为“本轮还没把目标表达用进完整输出”的训练提示

#### Scenario: 用户未填写完整输出
- **WHEN** 用户未填写完整输出就提交最终 feedback
- **THEN** 系统 MUST 记录完整输出未开始
- **AND** 系统 MUST 记录目标表达覆盖为未开始
