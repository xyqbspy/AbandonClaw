## MODIFIED Requirements

### Requirement: 任何改动都必须评估测试影响
任何实际改动在提交前 MUST 评估测试影响，包括已有测试是否需要更新、是否需要补充回归测试，以及是否会引入未覆盖风险。当改动涉及认证入口、关键 API handler 或已有失败的默认测试基线时，维护者 MUST 在完成实现前恢复可依赖的绿色基线，并为受影响入口增加直接回归测试。

#### Scenario: 默认单元测试命令已存在失败用例
- **WHEN** 维护者准备继续补充同一链路的新测试或修复入口实现
- **THEN** 他必须先修正或同步已有失败的现有测试，使受影响的默认测试命令回到可依赖状态
- **AND** 不应在已知测试失败的基线上继续累加新回归用例

#### Scenario: 改动认证入口或关键 API route
- **WHEN** 维护者修改 `middleware`、学习续接 API、复习提交 API 或其对应的入口规则
- **THEN** 他必须为该入口增加或维护直接的自动化测试，覆盖参数校验、重定向/响应结果以及关键 service 入参透传
- **AND** 不能仅依赖更高层的页面或手工回归来间接保护这些入口

#### Scenario: 完成关键测试补强后准备收口
- **WHEN** 维护者完成 `middleware` 或高优 API handler 测试补强并准备结束本次改动
- **THEN** 他必须执行受影响的默认测试命令和新增的定向测试命令
- **AND** 必须在变更记录或说明文档中留下本次回归范围和验证结果
