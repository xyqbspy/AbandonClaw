## Context

当前仓库的测试链路分为两层：`*.test.ts` 通过 Node `--test` 运行纯逻辑与 handler 规则测试，`*.test.tsx` 通过 jsdom 运行页面与组件交互测试。现状上，交互测试整体可用，但单元测试基线已经出现漂移，说明部分实现更新后没有同步维护对应断言；同时，`middleware`、`review` 与 `learning` 的关键 API 入口仍缺少直接测试，一旦认证分支、参数校验或 service 透传规则变化，风险只能依赖页面级回归间接暴露。

这次变更涉及页面入口、中间件与 API handler 三个层次，属于跨模块的测试补强。它不改变用户功能契约，但会改变仓库对“哪些关键入口必须被自动化验证”的维护要求。

## Goals / Non-Goals

**Goals:**
- 恢复当前单元测试基线，使默认测试命令重新具备可依赖性。
- 为 `middleware` 增加认证、重定向与安全路径约束测试，直接保护页面入口行为。
- 为 `review` 与 `learning continue/progress/start/pause` 等高优 API handler 增加入口测试，保护参数校验、服务调用透传与错误返回。
- 让这次补强的回归范围和执行命令在仓库文档中可追踪。

**Non-Goals:**
- 不在本次引入覆盖率平台、CI 门槛或新的测试框架。
- 不对全部 API route 做一次性补测，只处理学习与复习链路中最关键的一批入口。
- 不修改任何用户可见业务流程或页面文案，只修正与当前实现不一致的测试预期。

## Decisions

### 1. 先恢复失败单测，再扩入口测试
选择先修复 `review-page-messages` 相关失败，而不是直接新增更多测试。原因是当前 `test:unit` 不全绿，会让后续补充的用例无法提供清晰信号。相比一次性扩大量测试，先恢复基线能让每一步补强都具备可验证的增量结果。

备选方案是先补 `middleware` 或 API handler 测试，再在最后一起处理失败用例；这个方案会让实施过程中始终带着已知红灯，降低回归判断效率，因此不采用。

### 2. 中间件测试聚焦“规则输出”，避免耦合 Next 内部实现
`middleware` 的测试会优先验证未登录访问、已登录访问 auth 页面、`returnTo` 校验和重定向目标，而不是复制 Next.js 内部的请求对象细节。这样可以把断言集中在仓库自己的认证规则，减少未来 Next 版本变动带来的脆弱性。

备选方案是通过更高层的页面集成测试间接覆盖中间件，但这会让失败定位模糊，也不利于保护独立的跳转分支，因此不采用。

### 3. API handler 测试按“入口归一化 + 服务透传 + 错误分支”组织
对 `review` 与 `learning` 的 route 测试不会追求覆盖所有服务细节，而是锁住三类行为：请求体与 route 参数的归一化、调用 service 时的关键入参、异常/校验失败时的响应码与响应体。这与仓库现有 `scene parse/import` 和 `expression-clusters` handler 测试风格保持一致，维护成本更低。

备选方案是直接把更多逻辑下沉到 service 再只测 service，但本次目标是保护现有入口边界，不做接口重构，因此不采用。

### 4. 文档只补最小回归说明，不扩大为全量测试手册重写
当前 `test.md` 已经承担测试约定说明，本次只补本次变更涉及的关键链路和命令，避免在一次测试补强里顺带重写整份文档，扩大变更面。

## Risks / Trade-offs

- [风险] `middleware` 在 Next.js 环境对象上有额外隐式依赖，测试桩不完整会导致假阳性或假阴性。 → 缓解：优先复用最小可运行的 `NextRequest` 构造方式，只断言仓库自定义规则输出。
- [风险] API handler 测试如果过度断言内部调用细节，后续 service 重构会造成高维护成本。 → 缓解：只锁关键参数透传和外部响应，不锁实现步骤。
- [风险] 补测试时发现现有实现本身存在隐藏问题，会扩大本次范围。 → 缓解：范围仅限恢复当前行为与补入口保护；若发现真实行为变更需求，先回写 proposal/design 再继续。
- [风险] 测试文案文件已有编码痕迹，直接修改可能放大乱码问题。 → 缓解：所有编辑统一按 UTF-8 处理，只改必要文件并在验证中显式执行相关测试。

## Migration Plan

本次无需数据迁移或部署迁移。实施顺序为：先修复现有失败单测，再补 `middleware` 测试，再补 `review`/`learning` 的高优 API handler 测试，最后更新测试说明与 `CHANGELOG.md`。如新增测试暴露既有实现缺陷且超出本次范围，则停止继续扩展并回到提案阶段调整范围。

## Open Questions

- `learning` 入口测试首批是否只覆盖 `continue/progress/start/pause`，还是同时纳入 `complete/practice`。当前提案默认只覆盖前一组高频入口。
- `test.md` 的乱码是否属于本次顺手修复范围。当前默认只补最小内容，不做整份文档清理，除非实施中影响阅读与维护。
