## Context

当前项目的运行时缓存已经分散在多条链路中：

- `scenes` 列表通过 `scene-list-cache` 做内存 + IndexedDB 缓存，并在页面加载和 hover 时做详情预取。
- `scene detail` 通过 `scene-cache` 做详情持久化，并在进入详情后继续预取相关场景。
- `chunks` 列表通过 `phrase-list-cache` 做查询维度缓存，当前行为是先展示缓存，再继续联网刷新。
- 音频通过 `tts-api` 维护请求去重、短时 URL 缓存和浏览器 Cache API 持久缓存，并通过 `audio-warmup` / `resource-actions` 做句子与 chunk 的预热。

问题在于这些链路的策略并不一致：

- `scenes` 列表和 `scene detail` 在命中新鲜缓存时会直接停止联网，导致缓存 TTL 同时承担“离线可用”和“是否继续校验新鲜度”两种职责。
- `chunks` 列表则已经采用“先展示缓存，再联网刷新”的模式，与 `scenes` 行为不一致。
- API 层只有部分 GET 接口显式声明 `Cache-Control`，前端 `fetch` 又没有完全统一，浏览器默认缓存语义会混入自定义缓存层。
- 场景音频的句子和 chunk 已有预热，但场景整段循环音频没有纳入相同链路。

本次 change 需要先把这些链路收束成统一规则，再进入实现阶段。

## Goals / Non-Goals

**Goals:**

- 统一 `scenes`、`scene detail`、`chunks` 列表的缓存新鲜度原则。
- 明确前端自管缓存与 HTTP 响应缓存的职责边界。
- 为场景整段循环音频补齐预热目标和接入点。
- 在 proposal 阶段提前定义回归测试要求，避免实现时只修局部。

**Non-Goals:**

- 本次不引入新的缓存基础设施或第三方库。
- 本次不修改服务端业务字段结构、数据库表或学习状态模型。
- 本次不扩大到 `today`、`review`、`progress` 全量缓存重构，只处理已查明受影响的 `scenes/chunks/audio` 链路。

## Decisions

1. 场景列表与场景详情采用统一的 `stale-while-revalidate` 风格策略。
   - 决策：命中本地缓存时允许立即渲染，但不得因为缓存仍在 TTL 内就直接跳过网络刷新。
   - 原因：用户首先需要“快”，但学习状态、变体入口、标题/副标题也必须保持较新的服务端状态。
   - 备选方案：继续保留“TTL 内不联网”。
   - 不选原因：会让缓存 TTL 直接变成用户可见数据的陈旧上限，现有 7 天窗口过大。

2. 将“缓存保留时长”和“是否触发后台刷新”拆开管理。
   - 决策：TTL 只决定缓存是否可作为本地回退数据，不再决定是否停止联网校验。
   - 原因：离线能力与 freshness 是两个不同目标，耦合后会导致策略难以调优。
   - 备选方案：仅缩短 TTL，不改刷新规则。
   - 不选原因：即便缩到 1 天或 1 小时，依然会存在命中缓存后不刷新导致的陈旧窗口。

3. 对用户态 GET 接口统一声明缓存职责。
   - 决策：用户个性化数据接口应显式声明响应缓存语义，并让前端 `fetch` 在需要时明确 `cache` 策略，避免依赖浏览器默认行为。
   - 原因：当前一部分接口显式 `no-store`，另一部分默认处理，会放大调试复杂度。
   - 备选方案：只改前端，不改接口响应头。
   - 不选原因：职责边界仍然不清晰，后续继续容易出现“同类接口不同语义”。

4. 场景循环音频预热接入现有 `tts-api` 持久缓存，不新增单独缓存层。
   - 决策：复用 `ensureSceneFullAudio` / `prefetchSceneFullAudio` 和浏览器 Cache API。
   - 原因：现有音频缓存已经支持持久化、去重和清理面板，继续复用最简单。
   - 备选方案：新增场景级音频离线缓存模块。
   - 不选原因：复杂度高，且会与现有 TTS 缓存重复。

## Risks / Trade-offs

- [后台刷新频率增加] -> 通过请求去重、空闲调度和受控预取上限避免无意义重复请求。
- [列表在缓存渲染后再被网络结果更新，可能引发轻微闪动] -> 需要保留稳定排序和最小化状态替换策略。
- [统一 `Cache-Control` 后可能暴露既有测试对默认缓存行为的隐式依赖] -> 在实现阶段补齐受影响的交互测试和 API 测试。
- [补整段场景音频预热可能增加首屏后空闲带宽占用] -> 只在合适入口、空闲时机和非弱网条件下触发。

## Migration Plan

- 第一步：收敛 `scenes` 列表与 `scene detail` 的刷新策略，先保证缓存命中后仍会进行后台刷新。
- 第二步：统一相关 GET 接口的响应头和客户端 `fetch` 缓存语义。
- 第三步：把场景整段循环音频纳入既有预热链路，并限制触发时机。
- 第四步：补充交互测试、缓存测试和音频缓存回归测试。
- 回退策略：若新策略引起明显抖动或请求过量，可单独关闭后台刷新或场景整段音频预热，不影响既有缓存读写结构。

## Open Questions

- `scene list` 与 `scene detail` 的具体 TTL 是否统一，还是按列表/详情分别设定更短窗口，需要在实施阶段结合现有测试和体验再定。
- 场景整段音频预热更适合接在列表 hover、进入详情首屏稳定后，还是仅在用户已触发句子音频后再升级预热，需要在实现时做入口取舍。
