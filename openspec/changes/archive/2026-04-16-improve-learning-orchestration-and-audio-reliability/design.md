## Context

当前项目已经具备较完整的学习主链路和后端治理基线，但还存在三个体验与生产化层面的缺口：

1. `today` 的任务入口虽然已经能聚合 continue learning、review 和概览数据，但优先级解释和任务文案还不够稳定，容易出现“为什么先推荐这个”的可解释性不足。
2. TTS 链路已经有多层缓存、预热与回退，但真实 HTTP baseline 中仍出现过单次 `500`，而 `scene full` 播放失败时的降级体验、批量重生成的并发边界和异常记录还不够收口。
3. 第三阶段已补发布前 checklist，但 `next.config.ts` 还没有最小安全头配置，生产化收口仍偏文档约定，缺少运行时基线。

这次 change 不再扩展数据权限或接口治理范围，而是把推荐编排、音频可靠性和最小安全收口做成一组低侵入、可验证的增强。

## Goals / Non-Goals

**Goals:**

- 为 `today` 建立稳定的任务优先级与解释规则，减少推荐结果的偶然性和不可解释性。
- 为 TTS 链路补齐 scene full 失败降级、批量重生成并发边界和上游异常记录要求。
- 在不引入重型依赖的前提下补齐最小安全头，并把相关发布前检查纳入维护流程。
- 为上述改动准备最小测试和文档更新入口。

**Non-Goals:**

- 不在本次 change 中重写 today 推荐引擎或引入复杂评分系统。
- 不引入新的 TTS 供应商，也不做完整熔断框架。
- 不把前端缓存或音频链路整体迁移到 Service Worker / PWA 架构。
- 不在本次 change 中扩展数据库模型或再开新的大规模治理阶段。

## Decisions

1. `today` 编排只做“规则显式化 + 文案解释增强”，不做复杂推荐算法。
   - 原因：当前最缺的是稳定规则和可解释性，而不是更复杂的排序模型。
   - 替代方案：直接引入新的权重评分系统。
   - 不选原因：范围过大，且会放大主链路回归风险。

2. TTS 可靠性优先做“降级体验 + 边界控制 + 异常记录”，不更换现有链路主体。
   - 原因：现有 `Storage + 浏览器缓存 + 预热` 方案已经能支撑主路径，问题集中在失败时的用户体验和运维可见性。
   - 替代方案：引入第二套 TTS 提供商或完整断路器。
   - 不选原因：会明显扩大成本和实现复杂度。

3. 最小生产化安全收口以 `next.config.ts` 安全头和 checklist 为主。
   - 原因：当前缺口是“没有系统化 headers 基线”，这类配置收益稳定、侵入小。
   - 替代方案：直接引入更重的安全网关或平台能力。
   - 不选原因：超出当前项目阶段。

4. 文档更新优先复用现有说明，不新增重复语义文档。
   - `today` 规则优先更新现有 today 相关 spec / docs。
   - 音频链路继续收口到 `audio-tts-pipeline.md`。
   - 发布检查继续收口到现有 release readiness checklist。

## Risks / Trade-offs

- [Today 规则过于保守] → 先只显式化当前稳定优先级，避免一次引入复杂评分。
- [音频降级过多会让体验显得“退化”] → 只在 scene full 或上游失败时降级，并保留明确提示。
- [安全头配置误伤现有资源加载] → 先上最小基线头，避免直接启用高强度 CSP。
- [文案解释与真实后端状态脱节] → 将解释文案绑定到稳定字段与现有 today contract，而不是临时前端状态。

- [批量重生成并发提高后压到上游] → 采用有界并发，不做无限并发重生成。
- [发布清单继续散落] → 只允许在现有 checklist 和 dev-log 中收口，不再新增并列文档。
