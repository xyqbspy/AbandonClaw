# 设计说明：add-shared-loading-ellipsis

## Status

draft

## Current Flow

- 项目当前通过 `LoadingContent`、`LoadingButton`、`LoadingOverlay`、`LoadingState` 承担大部分 loading 展示
- 这些组件只接受静态 `loadingText`
- 各页面如果想显示“生成中...”之类的文案，只能自己传固定字符串

scene 练习页的“重新生成题目”目前也没有独立、统一的 loading 展示策略。除此之外，场景生成、导入、句子保存、后台批量动作、复习页进入场景等入口仍然大量使用静态的“生成中...”或“进入场景中...”字符串。

## Problem

- loading 文案目前是静态的，等待感偏弱
- 缺少统一的“动态省略号”实现，后续谁要用都容易再造一套
- “重新生成题目”这类动作没有统一的“正在生成中”视觉反馈
- 同类长等待动作目前体验不一致，用户在不同页面会看到不同风格的 loading 文案

## Decision

### 1. 动态省略号做成公共组件

- 在共享 loading 组件旁新增一个轻量公共文案组件
- 输入基础文本，例如“正在生成中”
- 输出带动态省略号的展示，例如视觉上在 `.`、`..`、`...` 之间轮流

### 2. 优先兼容现有 LoadingContent 体系

- 这套组件要能直接被 `LoadingContent` / `LoadingButton` 作为 `loadingText` 使用
- 不引入重依赖，不额外拆复杂动画系统
- 动画主要依赖 CSS 或极轻量渲染逻辑，保证可复用、易维护

### 3. scene 练习重生成先接入

- 在“重新生成题目”触发后，菜单项或对应按钮区域展示统一的“正在生成中”动态文案
- 保持现有行为不变，只提升等待反馈

### 4. 扩到首批高频长等待入口

- 第一批优先接入这些用户可感知、等待时间相对明显的入口：
  - 场景生成 sheet 的提交按钮
  - 场景导入 dialog 的导入按钮
  - 句子编辑 sheet 的保存按钮
  - 场景后台动作区的更新、生成、强制生成、删除
  - 复习页进入场景的按钮
- 这一批都沿用现有基础文案，只把省略号展示统一到公共组件，不改业务逻辑和交互时序

## Risks

- 如果只做字符串拼接，无法保证动画节奏统一
- 如果直接在每个使用点写定时器，维护成本会迅速扩散
- 如果动效实现不注意无障碍，可能影响屏幕阅读器重复播报
- 一次性铺得太广容易引入测试噪音，因此优先只覆盖长等待、高频、已有公共 loading 入口的场景

## Validation

- 公共 loading 组件能稳定渲染基础文本和动态省略号
- scene 练习“重新生成题目”会展示“正在生成中”统一文案
- 首批接入入口在 loading 时会展示动态省略号，不再停留在静态 `...`
- 现有 loading 组件行为不回归
