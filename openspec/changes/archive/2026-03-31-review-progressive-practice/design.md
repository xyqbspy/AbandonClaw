## Context

当前 `review` 页已经具备统一舞台和基础阶段推进，但普通表达复习仍主要停留在“回忆参考句 + 自评”层，场景回补则偏向单题补做。新的需求希望把单次复习拆成更清晰的四类交互：

- 熟悉度判断与输出信心
- 微回忆练习
- 变体改写
- 场景完整输出

这不是单纯 UI 扩展，而是会同时影响：

- 页面状态机和 CTA 推进顺序
- 普通表达与场景回补的统一舞台模型
- 失败后的降级策略
- 哪些训练结果写入后端、哪些暂时只是前端 TODO

如果这些边界不先写清，实施时很容易出现前端交互很丰富，但后端仍只收到粗粒度 `again/hard/good`，导致学习语义和页面体验再次脱节。

## Goals / Non-Goals

**Goals:**
- 定义 `review` 的递进式训练模型，让单次复习能从识别自然推进到输出。
- 约束不同掌握度下的训练门槛，避免所有复习项都走一条过重的完整流程。
- 明确各阶段与后端记录的关系，允许分阶段接入后端能力，但禁止伪造正式学习完成信号。
- 为后续统计、调度和 AI 反馈留出稳定扩展位。

**Non-Goals:**
- 本次 proposal 不直接决定最终的 AI 评分算法或自然度评分公式。
- 不要求在第一期就完成所有后端落库、统计和推荐改造。
- 不把 `review` 改成新的独立学习系统；它仍属于现有学习闭环中的复习巩固环节。

## Decisions

### 1. 用“阶段 + 分层路径”替代“所有题都走满全流程”

`review` 将统一抽象出四类阶段信号，但每条复习项不一定全部执行：

- `recognition_check`
- `micro_recall`
- `variant_rewrite`
- `full_output`

页面需要根据当前复习项类型、历史掌握度、是否来自场景回补等因素，决定本次训练走到哪一层。这样可以避免简单表达也被迫进入过重的完整输出任务。

备选方案是所有复习项固定执行 A/B/C/D 四步。这个方案实现简单，但会显著拉长复习时长，也会降低高频复习的可持续性，因此不采用。

### 2. 把“正式回写信号”和“前端 TODO 占位”显式分开

不同阶段产生的结果强度不同：

- 熟悉度/自信度更适合作为调度补充信号
- 微回忆和变体改写更像过程信号
- 完整输出才接近高置信度产出能力信号

因此设计上必须允许：

- 前端先落交互
- 后端能力不足的阶段先以 TODO 占位
- 只有已经接通正式契约的阶段结果才真正影响学习状态或复习调度

备选方案是统一把所有阶段结果压缩回旧的 `again/hard/good`。这个方案兼容最简单，但会丢失新交互的核心价值，因此不采用。

### 3. 失败处理必须设计成“训练降级”，而不是单次判错

递进式复习里，失败不是终点，而是回到更低提示强度的训练路径。设计上需要明确：

- 完整输出失败后是否降级为 chunk 提示
- 变体改写失败后是否回到原句微回忆
- 场景回补失败后是否回到原场景继续练

这样 `review` 才会保持训练产品属性，而不是变成轻量考试页。

### 4. 后端计划先作为 change 内显式任务管理

如果实施阶段确认需要新增字段、事件或聚合能力，应继续纳入同一个 change 的 implementation tasks，而不是在页面里先偷偷写本地状态替代正式语义。这样能保证：

- 前端交互和后端信号同一套文档维护
- `today` / dashboard / future analytics 不会再次消费不清晰的复习状态

## Risks / Trade-offs

- [流程变重] → 通过分层路径控制不同复习项的训练深度，避免所有题都执行完整链路。
- [前后端语义脱节] → 明确区分正式回写信号与 TODO 占位，并要求后端接入时同步更新 spec 和测试。
- [评估标准不稳] → 第一阶段先聚焦稳定的流程与信号边界，自然度评分、AI 点评等高波动能力后置。
- [页面状态机复杂度上升] → 优先把阶段映射、门槛和失败降级放进 selectors/logic，而不是直接堆在 `page.tsx`。
