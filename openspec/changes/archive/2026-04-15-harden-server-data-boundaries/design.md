## Context

当前主链路的后端接口已经具备第一阶段入口治理能力：`middleware.ts` 会下发 `requestId`，高成本接口已有基础限流和来源校验，未知错误也已经统一收口。但从链路文档和现有实现来看，真正影响数据正确性的风险仍集中在服务端内部：

- `scene / learning / review / phrases` 的部分用户态读写仍通过 `service role` 客户端访问，实际权限边界依赖 service/repository 中的手工过滤，而不是最小权限模型。
- `review submit`、`learning` 状态推进和短语批量保存等写接口会改变学习进度、调度状态或用户资源集合，但当前缺少统一的幂等键、版本检查或条件更新。
- 第一阶段的限流是进程内窗口实现，只适用于单实例或低流量环境，无法作为长期线上基线。
- 参数校验虽然已有通用 helper，但不同 route 的接入程度不一致，高风险写接口仍存在分散校验逻辑。

这次设计面向第二阶段：先收紧主链路写入的“权限边界”和“一致性保护”，再把第一阶段限流升级成真正可上线的共享能力。

## Goals / Non-Goals

**Goals:**

- 为用户态接口建立明确的数据访问边界，减少对 `service role` 的依赖，并保留少量明确白名单场景。
- 为关键写接口建立最小一致性保护，确保重复点击、重试和多端并发时不会破坏学习/复习状态。
- 将高成本接口限流升级到共享存储承载的实现，并保留第一阶段已有的错误契约。
- 收口高风险写接口的参数校验方式，优先形成统一 schema 驱动入口。
- 为第二阶段改动规划最小回归测试和压测基线。

**Non-Goals:**

- 不在本轮设计中一次性迁移所有服务端查询到全新仓储框架。
- 不在本轮设计中引入完整熔断、分布式追踪或 Sentry/OTel 平台能力。
- 不在本轮设计中处理字段级加密、密钥管理或全面合规体系。
- 不要求第一步就完成所有 Supabase RLS 设计；允许按主链路优先级逐步迁移。

## Decisions

### 1. 按“用户态接口优先收紧，后台任务白名单保留”处理 `service role`

原因：

- 当前最大的安全纵深问题不是完全没有鉴权，而是用户态接口在服务端内部绕开了最小权限边界。
- 直接全量禁用 `service role` 风险过大，会同时影响后台预热、批量运维和跨用户聚合场景。

备选：

- 方案 A：一次性全面禁用 `service role`，所有调用强制改为用户上下文。
- 方案 B：先按接口类型切分，用户态读写优先迁移，后台任务保留白名单。

本次选择方案 B，因为更符合主链路优先和最小风险落地。

### 2. 并发保护优先采用“幂等键 + 条件更新/版本检查”而不是锁服务

原因：

- `review`、`learning` 和 `phrases` 的问题核心是重复提交和状态竞争，不一定需要先引入分布式锁。
- 幂等键和条件更新更容易与现有 route/service 结合，也更容易写测试。

备选：

- 方案 A：引入 Redis 锁或数据库 advisory lock 统一保护所有关键写入。
- 方案 B：按接口语义分别使用幂等键、唯一约束、条件更新或版本字段。

本次选择方案 B，因为复杂度更低，且更适合渐进迁移。

### 3. 共享限流在第二阶段升级，但保持第一阶段 helper 契约

原因：

- 第一阶段已经有统一的 `rate-limit.ts` 入口，升级时最好保持 route 调用方式不变，只替换底层存储。
- 这样可以降低高风险接口回归范围，不需要重新改每个 route 的错误处理。

备选：

- 方案 A：新建完全不同的限流接入方式。
- 方案 B：保留现有 helper API，仅将状态存储迁移到共享后端，并补充配置和 fallback。

本次选择方案 B。

### 4. 参数校验按“高风险写接口优先”统一到 schema 驱动

原因：

- 当前最容易出现不一致的地方在学习、复习、短语批量写入和高成本生成接口。
- 如果一开始强推全量迁移，会把第二阶段范围拉得过大。

备选：

- 方案 A：一次性统一全部 API 参数校验。
- 方案 B：先收口关键写接口和高成本接口，再逐步扩散。

本次选择方案 B。

## Risks / Trade-offs

- [用户态接口迁移时遗漏隐式 `service role` 依赖] -> 先盘点 `scene / learning / review / phrases` 的调用链和白名单，再分批实施。
- [幂等设计不当导致正常重复操作被误拒绝] -> 每个接口先按业务语义定义幂等窗口和去重键，并补直接 route 测试。
- [共享限流引入新基础设施依赖] -> 保留进程内 fallback，仅将共享方案用于生产默认路径。
- [参数校验统一引起旧前端请求被拒绝] -> 先从已有文档和测试覆盖的高风险接口开始，保持错误码和错误结构稳定。

## Migration Plan

1. 盘点用户态接口中的 `service role` 使用点，区分可迁移与必须保留的后台白名单。
2. 为 `review`、`learning`、`phrases` 定义关键写接口的一致性策略和测试基线。
3. 升级 `rate-limit` 底层存储并接入配置，保证现有 route 层契约不变。
4. 为高风险写接口统一参数校验入口，并补最小压测脚本。
5. 完成定向回归测试、文档更新和变更记录。

回滚策略：

- 数据边界收紧按模块逐步接入，任何模块出现问题都可以回退该模块的 client 选择策略。
- 共享限流升级保持 helper 契约不变，可快速退回进程内实现。
- 幂等和条件更新按接口逐个落地，异常时可单独回退某个接口的保护逻辑。

## Open Questions

- `phrases save-all` 更适合幂等键还是数据库唯一约束驱动的批量 upsert 去重？
- `learning progress` 是否需要引入显式版本字段，还是只用条件更新时间与阶段状态推进保护？
- 共享限流的存储介质是直接选现有基础设施，还是在 proposal 通过后再做最小选型确认？
