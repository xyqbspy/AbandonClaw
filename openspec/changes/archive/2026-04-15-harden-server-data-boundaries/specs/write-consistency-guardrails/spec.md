## ADDED Requirements

### Requirement: 关键写接口必须具备幂等或条件更新保护
任何会推进学习阶段、写入复习结果、批量保存短语或改变用户态资源集合的关键写接口 MUST 在重复请求、网络重试或多端同时提交时保持结果一致。系统 MUST 为这类接口提供幂等键、条件更新、版本检查或等效的一致性保护，而不能把每次重复请求都当成新的有效写入。

#### Scenario: 客户端重复提交同一 review 结果
- **WHEN** 同一用户在短时间内重复提交同一条 review 结果
- **THEN** 系统 MUST 至多写入一次正式复习结果
- **AND** 不得因为重复提交而错误推进多次调度或 summary 更新

#### Scenario: 多端同时推进同一 scene 学习状态
- **WHEN** 同一用户从两个终端同时提交同一 scene 的学习状态推进请求
- **THEN** 系统 MUST 按定义好的阶段推进规则只接受合法更新
- **AND** 不得把较旧状态覆盖为较新状态之前的值

### Requirement: 一致性保护必须保持主链路可恢复
关键写接口在触发幂等命中、版本冲突或条件更新失败时 MUST 返回受控结果，并保持前端可以继续恢复当前学习或复习链路，而不能产生不透明的内部错误。

#### Scenario: 写接口命中幂等保护
- **WHEN** 客户端提交的关键写请求命中已存在的幂等结果
- **THEN** 系统 MUST 返回可处理的受控结果
- **AND** 前端 MUST 能继续停留在当前链路或读取最新状态，而不是被迫中断

#### Scenario: 写接口发生版本冲突
- **WHEN** 关键写请求因条件更新或版本检查失败而无法落库
- **THEN** 系统 MUST 返回可识别的冲突结果
- **AND** 不得把底层数据库错误直接暴露给客户端
