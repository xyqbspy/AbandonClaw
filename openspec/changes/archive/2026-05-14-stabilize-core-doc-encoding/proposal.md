# Proposal: 收紧核心文档编码与乱码检查自保护

## 背景

本轮暴露的不是业务能力问题，而是维护链路稳定性缺口：

- `CHANGELOG.md` 的文件字节内容实际是干净 UTF-8，但终端文本预览出现了乱码假象，容易诱导维护者误判并在干净文件上继续叠加“修乱码”操作。
- `scripts/check-mojibake.ts` 把一批乱码片段直接写在源码里，因此只能把自身加入忽略名单，导致“乱码检查器自己带乱码也能通过”。
- 当前 stable rule 已要求核心入口文档保持可读 UTF-8，但还没有明确约束“乱码检查脚本本身不得通过自我忽略规避检查”，也没有明确“诊断乱码时要先区分文件损坏和终端显示问题”。

## 本轮目标

1. 修复 `scripts/check-mojibake.ts` 的自保护缺口，让检查器源码本身保持干净可读。
2. 为乱码模式与忽略样例改成编码安全表达，避免继续把可疑乱码字面量写进检查器源码。
3. 补一条最小测试，验证常见乱码片段仍能被识别，且检查器源码不再依赖忽略自身。
4. 将“核心文档编码诊断”和“乱码检查器不得自我豁免”写入 `project-maintenance` stable spec 与维护手册。

## 本轮收口项

- 收紧 `text:check-mojibake` 的实现边界。
- 补充维护规则：先验证文件字节是否为干净 UTF-8，再决定是否重写文档。
- 明确核心维护脚本自身也属于编码稳定性保护范围。

## 明确不收项

- 不做全仓历史文档乱码大扫除。
- 不引入新的第三方编码检测依赖。
- 不改业务页面、学习链路或已有产品能力。

## 风险与延后

- 历史未触碰文档如果仍存在旧乱码，本轮不会一次性全部清理；后续若再次触碰，再按最小范围收口。
- 终端显示链路的宿主环境差异无法仅靠仓库代码完全消除，因此本轮重点放在“先验证文件字节、再决定是否重写”的规则与工具层兜底。
