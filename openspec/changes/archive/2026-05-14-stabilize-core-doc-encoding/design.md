# Design: 核心文档编码稳定性收口

## 问题拆解

当前问题包含两层：

1. 诊断层
   - 维护者看到终端输出乱码时，可能会把显示问题误判为文件内容损坏。
   - 对 `CHANGELOG.md` 这类正式发布文档而言，这种误判的风险很高，因为误修可能污染真实发布记录。

2. 工具层
   - `scripts/check-mojibake.ts` 直接包含多段乱码字面量。
   - 为避免脚本误报自己，当前实现把 `scripts/check-mojibake.ts` 放进忽略名单。
   - 这使得检查器失去自检能力，形成稳定缺口。

## 设计决策

### 决策 1：乱码模式改成编码安全表达

将可疑乱码片段从“源码里直接写字面量”改成“由 code point 组合生成”的形式。

原因：

- 检查脚本源码保持干净 UTF-8。
- 不再需要通过忽略脚本自身来规避检查。
- 保留现有高置信度模式匹配策略，不引入复杂检测器。

### 决策 2：为检查器补最小单测

增加一个轻量 Node test，至少覆盖：

- 常见高置信度乱码片段仍能被识别。
- 检查脚本源码本身不再落入需要被忽略的脏文本集合。

原因：

- 这是维护脚本，不适合只靠人工目测。
- 单测成本低，但能有效避免未来回退。

### 决策 3：把“先验证字节、再决定是否重写”写入稳定规则

在 `project-maintenance` stable spec 和维护手册中明确：

- 发现文档疑似乱码时，必须先用 UTF-8 感知的读取方式验证文件内容。
- 如果文件内容本身干净，只是终端显示异常，则应修诊断链路或记录环境差异，而不是重写干净文件。
- 乱码检查脚本及其规则文本不得通过忽略自身来通过检查。

原因：

- 这次问题说明“编码稳定性”不仅是文件保存规则，也包括诊断流程本身。
- 正式发布文档比一般源码更需要防止误修。

## 本轮收口项

- `scripts/check-mojibake.ts`
- `scripts/check-mojibake.test.ts`
- `openspec/specs/project-maintenance/spec.md`
- `docs/dev/project-maintenance-playbook.md`
- `docs/dev/dev-log.md`

## 明确不收项

- 不重写 `CHANGELOG.md` 全历史内容。
- 不修复未触碰的历史 archive 或历史 dev 文档。
- 不引入自动转码器或文档格式化流水线。

## 风险

- 乱码片段模式仍是启发式，不可能覆盖所有编码损坏形式。
  - 缓解：继续沿用高置信度策略，后续发现新模式再最小补充。
- 某些终端宿主的显示链路仍可能出现假乱码。
  - 缓解：stable rule 明确“先验证文件字节内容”，避免把显示问题误当文件损坏。
