## Context

当前项目的接口链路大致是：

- `middleware.ts` 负责页面与部分 API 的登录边界。
- route / handler 负责请求体解析、参数校验和调用 service。
- `src/lib/server/api-error.ts` 负责把异常映射成接口响应。
- `src/lib/server/upstream.ts` 负责上游超时控制。
- 各模块在 route、handler、service 中零散输出 `console.warn/error/log`。

现状已经具备一些基础保护，但还没有统一的接口治理层：

- 高成本接口大多依赖“已登录才能访问”，但没有统一限流。
- 请求缺少统一 `requestId`，日志是散点式的，问题排查依赖人工拼接上下文。
- 受保护写接口没有统一 `Origin`/来源校验。
- 未知内部错误虽然会走 `fallbackMessage`，但没有统一把追踪信息带回响应，也没有稳定的日志结构。

这次设计的目标不是一次性搭完整平台，而是建立第一阶段的最小治理基线，并限定接入范围，避免大面积改动业务实现。

## Goals / Non-Goals

**Goals:**

- 为请求建立统一 `requestId`，并让错误响应与日志都能复用该标识。
- 为高成本接口增加统一限流基线，优先保护 AI / TTS / 解析接口。
- 为受保护写接口增加最小同源来源校验。
- 收口未知内部错误的对外响应与日志记录方式。
- 先在高风险接口落地，保持变更低侵入、易回归。

**Non-Goals:**

- 不实现依赖 Redis/网关的分布式限流。
- 不一次性迁移全部 route 到统一框架。
- 不在本次变更中完成 service-role 权限收紧和数据库层纵深防护。
- 不在本次变更中引入完整 OpenTelemetry、Sentry 或熔断体系。

## Decisions

### 1. 采用“公共 helper + 高风险接口先接入”的最小落地方式

原因：

- 当前接口已经有 `route -> handler -> service` 分层，但并不完全统一。
- 如果先强推全量中间件化或全量改造，范围会过大，审批和回归成本都高。

备选：

- 方案 A：先上全局 API 框架包装器，再迁移全部接口。
- 方案 B：先抽出公共治理 helper，只接入高风险接口。

本次选择方案 B，因为更符合“最小可维护方案”。

### 2. requestId 由 middleware 生成，并向后续响应透传

原因：

- middleware 是请求进入应用后最稳定的统一入口。
- 由这里生成 `x-request-id`，可同时覆盖页面与 API，并能让后续 route/handler 读取。

备选：

- 在每个 route 内单独生成。
- 在 logger 调用时惰性生成。

放弃原因：

- route 内生成会导致规范不一致。
- 惰性生成不利于串联 middleware 与 route 日志。

### 3. 限流第一阶段采用进程内窗口限流

原因：

- 当前目标是建立治理基线，不引入新的基础设施依赖。
- 进程内限流足够保护单实例开发/预览和低流量环境，也便于先验证阈值。

备选：

- 直接引入 Redis/Upstash 做分布式限流。

放弃原因：

- 引入新依赖和运维复杂度，不适合第一阶段。

### 4. 写接口来源校验采用最小同源策略

原因：

- 当前鉴权主要依赖 cookie/session。
- 对受保护写接口增加 `Origin` 校验，可以在不引入完整 CSRF token 的前提下，先补最小攻击防护。

备选：

- 直接上完整 CSRF token 方案。

放弃原因：

- 当前范围过大，会波及前端调用契约和表单流程。

### 5. 未知内部错误继续对外返回通用错误，但补充 requestId

原因：

- 保持现有 `AppError` 契约不变，减少业务回归。
- 对未知错误统一返回受控文案，并附带 `requestId`，兼顾安全与排障。

备选：

- 对所有错误都返回统一通用文案。

放弃原因：

- 会破坏已有显式业务错误契约，影响前端提示和测试。

## Risks / Trade-offs

- [限流阈值过严] -> 先只接入高成本接口，并采用保守阈值；通过测试样本和手动回归校准。
- [同源校验误伤服务端调用] -> 仅先接入受保护写接口，并允许明确的同站来源；保留最小兼容判断。
- [requestId 接入不一致] -> 先让 `api-error` 和重点 route 统一使用，再逐步扩散。
- [进程内限流不适用于多实例] -> 在 spec 中明确这是第一阶段基线，后续有流量压力时再升级。

## Migration Plan

1. 新增请求上下文、日志和限流 helper。
2. 在 `middleware.ts` 注入 `x-request-id`。
3. 改造 `api-error`，统一返回 `requestId`。
4. 先接入高成本接口与核心写接口。
5. 补 route 级测试、middleware 测试与文档记录。

回滚策略：

- 若某项治理误伤正常请求，可按模块回退单个 helper 接入点。
- `requestId` 透传与错误响应收口可独立保留，不依赖限流或来源校验。

## Open Questions

- 第一阶段是否把 `/api/review/submit` 和 `learning/*` 全部纳入来源校验，还是先仅覆盖高风险写接口。
- 限流阈值是否需要按“开发/生产”环境分开配置。
- 是否在本 change 内同步定义日志字段最小集合，还是只先要求存在 `requestId` 与错误上下文。
