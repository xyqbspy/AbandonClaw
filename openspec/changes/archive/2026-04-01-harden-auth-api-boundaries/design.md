## Context

当前认证与接口保护链路分散在三个层面：

- 页面侧登录/注册页会读取 `redirect` 参数并在客户端跳转。
- `middleware` 负责一部分页面和 API 的登录态兜底。
- 各 API route 再按需调用 `requireCurrentProfile()`、`requireAdmin()` 或直接进入外部模型调用。

现状的问题不是“完全没有保护”，而是边界不一致：

- `middleware` 已经拦截不安全的 `redirect` 目标，但登录/注册页仍保留更弱的客户端校验。
- 一部分高成本接口没有统一受保护，匿名请求仍可能触发模型或重型解析。
- 模型调用底层缺少统一超时和失败保护，上游慢请求会直接占住 API 处理时间。
- `/api/me` 在登录后热路径存在重复认证/资料查询，增加了不必要的 I/O。

这次变更横跨认证页面、`middleware`、API route 和模型客户端，属于明显的跨模块安全/性能收口。

## Goals / Non-Goals

**Goals:**

- 统一登录重定向的安全规则，确保前后端只接受站内安全路径。
- 为高成本接口建立一致的访问控制，避免匿名流量直接消耗外部模型资源。
- 在模型调用层提供最小可用的超时与错误收敛，降低接口被上游拖慢或放大错误的风险。
- 让 API 保护策略更一致，减少“新增 route 时忘记补鉴权”的概率。
- 优化登录后热路径的重复 auth/profile 查询。
- 为上述边界补上直接自动化测试。

**Non-Goals:**

- 不重做整套登录体验或 Supabase 认证方案。
- 不在本次 proposal 中引入复杂的全局网关、WAF 或新第三方限流基础设施。
- 不处理与认证无关的普通业务接口重构。
- 不承诺一次性覆盖所有历史接口，只聚焦已识别的高风险入口与统一规则。

## Decisions

1. 站内重定向校验抽为共享规则，而不是继续分别维护页面版和 `middleware` 版。  
原因：当前真实问题就是前后端规则漂移。共享 helper 能把修复收敛成一处规则和一组测试。  
备选方案：只修页面局部判断。  
放弃原因：短期能止血，但后续仍容易再次分叉。

2. 对高成本接口优先采用“默认要求登录，例外显式声明”的策略。  
原因：这些接口一旦开放给匿名请求，风险同时体现在安全、成本和性能上。先统一走 `requireCurrentProfile()` 或等价保护，能最快收口主要风险。  
备选方案：保留匿名访问，仅加输入校验。  
放弃原因：不能解决资源滥用和成本放大。

3. 模型调用层只补最小失败保护，不在本次引入复杂熔断体系。  
原因：当前最急迫的是超时和错误收敛，优先级高于完整的队列、熔断或集中配额系统。  
备选方案：一次性上全局重试、熔断、并发池。  
放弃原因：实现面过大，会拖慢本次安全收口。

4. API 保护分两层：`middleware` 做粗粒度兜底，route/service 保留细粒度权限判断。  
原因：只靠 `middleware` 不够细，只靠 route 又容易漏；两层并存更稳。  
备选方案：只扩大 `middleware` 白/黑名单。  
放弃原因：无法替代管理员权限、用户归属等业务级校验。

5. `/api/me` 以“单次拿 user，再基于该 user 获取 profile”为目标收敛查询。  
原因：它是登录成功后的热路径，减少一次到两次 auth I/O 能直接改善稳定性和响应时间。  
备选方案：维持现状，只做外层缓存。  
放弃原因：会掩盖重复查询问题，且对首个请求帮助有限。

## Risks / Trade-offs

- [高成本接口改为必须登录后，匿名体验会收紧] → 在 proposal 和 spec 中明确属于预期行为变化，并为必要的匿名例外单独列白名单。
- [统一 `redirect` 规则后，历史上依赖宽松路径判断的边缘跳转可能失效] → 只接受明确站内路径，并补覆盖常见合法/非法输入的测试。
- [新增超时后，慢上游会更早失败] → 同步收敛错误响应并保留可观察日志，优先保障服务不被拖死。
- [扩大 `middleware` 保护范围后，部分原先“靠 route 自己处理”的接口可能提前返回 401] → 在 spec 中把这类行为视为正式边界收紧，并补未登录回归测试。

## Migration Plan

1. 先实现共享重定向校验与测试，确保登录页、注册页、`middleware` 行为一致。
2. 再逐步收紧高成本接口访问控制，并同步补 handler/route 测试。
3. 为模型客户端增加超时与错误收敛，确认受影响接口的失败分支稳定。
4. 优化 `/api/me` 热路径查询，补针对 route 的回归测试。
5. 最后扩充 `middleware` API 兜底规则并跑受影响测试集合。

回退思路：

- 若某个高成本接口必须短期保留匿名访问，应以显式 allowlist 方式回退，而不是回到“默认全部放行”。
- 若模型超时阈值设置过紧，可只调整阈值，不回退统一错误收敛结构。

## Open Questions

- 哪些高成本接口在产品上必须支持匿名访问，如果有，接受的保护方式是静态限流还是轻量配额。
- 统一超时阈值是按 provider 固定值处理，还是按接口类型区分。
- 是否要在本次实现中一并加入 `Origin` 校验，还是先把它保留为下一轮安全收口。
