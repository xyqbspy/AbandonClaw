## Context

当前 `review` 链路分成两部分：

- 普通表达复习：`/api/review/due` 调 `getDueReviewItems()`，直接从 `user_phrases` 里取 `review_status in (saved, reviewing)` 且到期的表达。
- 场景回补复习：同一个接口额外调 `getDueScenePracticeReviewItems()`，从 `user_scene_practice_attempts` 联表 `scenes` 生成回补项。

这意味着“普通表达复习”和“场景回补”对场景存在性的约束并不一样：

- 场景回补使用 `inner join scenes`，来源场景失效时项目本身不会出现在列表里。
- 普通表达复习只保留历史 `source_scene_slug`，拉列表时不会校验该 slug 当前是否仍然可见。

页面层现在只要看见 `sourceSceneSlug`，就会直接渲染“查看原场景”按钮并跳转 `/scene/{slug}`。如果这个场景已被删除、当前用户不可见，或表达记录里留下的是历史失效 slug，就会落到“场景不存在”。

## Goals / Non-Goals

**Goals:**
- 明确 `review` 普通表达复习的来源契约，避免团队误以为它默认只来自“已完成场景”。
- 定义“查看原场景”入口的可用性判断和降级行为，避免直接把用户送进 not found。
- 为 `review` 页面补一份专项维护文档，固定数据关系、场景关系和后续变更边界。
- 为后续是否引入“仅完成场景进入 review”保留显式演进空间，而不是在 bugfix 中隐式更改策略。

**Non-Goals:**
- 本次不默认重构整个 `review` 队列排序策略。
- 本次不默认引入新的场景完成度过滤规则。
- 本次不默认增加 AI 或推荐层能力。

## Decisions

### 1. 把“来源规则澄清”和“失效跳转兜底”放进同一个 change

原因：
- 当前 bug 的根本原因不是单点跳转错误，而是来源契约和展示契约脱节。
- 只修前端按钮而不写明来源规则，后续很容易再次出现误判。

备选方案：
- 只修复按钮跳转：短期快，但无法回答“review 到底该来自哪些场景”的维护问题。
- 直接改成只允许完成场景进入 review：范围过大，属于独立产品行为变更。

### 2. 普通表达复习继续以“已保存且到期的表达”为当前真实契约

原因：
- 这是现有后端 `getDueReviewItems()` 的真实行为，也是当前 `today` / `review` 已经在消费的逻辑。
- 直接在这个 change 中改成“仅已完成场景”会触及复习进入时机、统计口径和学习节奏，不应夹在 bugfix 中偷偷改变。

备选方案：
- 提案里直接把来源改成“仅已完成场景”：会扩大到新的产品策略评审。

### 3. “查看原场景”需要显式降级，不再默认假设 slug 永远有效

候选实现方向：
- 服务端在 due review 接口里额外返回 `sourceSceneAccessible`。
- 或前端按保守规则，仅在确认来源场景仍有效时展示入口。

当前设计倾向：
- 优先在服务端聚合层输出稳定字段，减少前端猜测。
- 若实施时发现成本过高，可采用前端降级文案 + 点击前校验作为阶段一方案，但必须在文档里注明限制。

## Risks / Trade-offs

- [风险] 若只做前端降级，不补服务端可访问性字段，仍可能存在边界状态不一致
  → Mitigation：在设计与文档中明确阶段一限制，并补回归测试。

- [风险] 团队可能继续把“来源澄清”误解成“要把 review 只绑定已完成场景”
  → Mitigation：在 proposal、spec 和维护文档里明确把这两件事拆开。

- [风险] 历史 `sourceSceneSlug` 数据可能已经存在失效值
  → Mitigation：把“历史来源场景失效”定义为合法降级场景，而不是异常状态。
