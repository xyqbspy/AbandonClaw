## Context

当前 `review` 递进式练习已经分出了：

- 微回忆
- 熟悉度 / 输出信心
- 变体改写
- 完整输出
- 最终反馈

但正式后端仍主要围绕：

- due review 列表
- `again / hard / good` 提交
- 场景回补的正式练习记录

这意味着新的训练层级虽然已经能改善前端体验，却还没有稳定进入服务端学习语义。继续维持现状会带来几个问题：

- 无法区分“眼熟”和“能主动输出”
- 无法把高价值的完整输出能力信号纳入调度
- `today` / dashboard 如果要展示更细粒度复习进度，会缺少稳定字段来源
- 后续很容易再次把前端局部状态误当成正式学习结果

## Goals / Non-Goals

**Goals:**
- 明确 `review` 递进式练习里哪些阶段结果值得成为正式后端信号。
- 约束新信号与现有 `again / hard / good`、场景回补、学习统计之间的关系。
- 定义这些信号在服务端、聚合层和 `today`/dashboard 中的消费边界。
- 形成可逐步实施的数据模型和迁移计划。

**Non-Goals:**
- 这次设计不直接确定 AI 自然度评分算法或 chunks 命中率算法细节。
- 不要求所有前端阶段都在第一版就正式落库。
- 不把 `review` 信号和 scene 主学习信号混成同一套字段。

## Decisions

### 1. 只把“可稳定解释且对调度有价值”的阶段升级为正式信号

不是每个前端阶段都值得直接落库。当前更合理的正式信号候选是：

- `recognition_state`
- `output_confidence`
- `full_output_status`

而变体改写更适合作为中间训练态，除非后续评估规则稳定，否则先不强制变成正式长期字段。

这样做可以避免把大量高噪声草稿数据直接写进长期学习态。

### 2. 新信号必须和最终 feedback 并存，而不是替代

`again / hard / good` 当前已经承担调度主入口，短期内不应该被完全替换。更合理的做法是：

- 保留现有最终反馈信号
- 把新训练信号作为补充维度
- 再由服务端决定如何组合影响调度

这样兼容成本更低，也更容易逐步上线。

### 3. 聚合层只消费“解释清晰的正式信号摘要”

`today` / dashboard 不应直接消费所有 review 原始事件，而应使用服务端聚合后的摘要字段，例如：

- 当前复习项是否达到主动输出级别
- 今日 review 中有多少项完成了完整输出
- 某条复习是否仍停留在识别级别

这样可以减少前端页面之间各自解释事件的风险。

### 4. 采用“事件记录 + 摘要聚合”双层设计

推荐把 review 正式信号拆成两层：

- 事件层：记录单次阶段结果
- 摘要层：记录当前复习项或当天聚合后的稳定状态

事件层适合后续分析、回放和策略演化；摘要层适合 `today` / dashboard / review 列表快速消费。

## Risks / Trade-offs

- [字段过多、语义发散] → 只先升级少数高价值信号，避免把所有前端阶段都直接落库。
- [和现有 review 调度冲突] → 保留 `again / hard / good` 作为兼容主信号，新信号先做补充维度。
- [today / dashboard 解释再次漂移] → 强制通过聚合摘要字段消费，不允许页面直接读取原始事件自定义解释。
- [迁移历史数据困难] → 对新增信号按“仅新记录生效，历史记录保守空值”设计，不强求一次性回填全部历史事件。
