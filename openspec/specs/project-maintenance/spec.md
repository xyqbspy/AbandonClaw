## Purpose

定义仓库长期维护时必须遵守的基础规则，确保维护者可以快速理解项目、按 OpenSpec 流程推进改动，并为每次实际变更留下可追踪的记录。

## Requirements

### Requirement: 项目维护必须有统一入口文档
仓库 MUST 提供一份面向维护者的项目说明，覆盖产品主链路、核心目录职责、改动入口和回归检查点。

#### Scenario: 维护者首次接手项目
- **WHEN** 维护者第一次进入仓库
- **THEN** 他可以通过固定文档快速理解 `today`、`scene`、`chunks`、`review`、`progress` 的关系
- **AND** 他可以定位主要代码目录和服务入口
- **AND** 他可以知道修改某一条学习链路后应该回归哪些测试

### Requirement: 非微小改动应优先经过 OpenSpec
涉及功能行为、数据流、状态流、缓存策略、测试链路、维护规范或跨页面 UI 一致性的改动，维护流程 MUST 优先通过 OpenSpec 形成可审阅的 proposal、spec 与 tasks。

#### Scenario: 准备统一多个详情页的底部动作表现
- **WHEN** 维护者要统一多个页面之间的 footer、按钮、icon 或交互表达
- **THEN** 应先通过 OpenSpec 记录这次统一的目标、基准和影响范围
- **AND** 不应只把决策留在临时聊天记录中

### Requirement: 每次实际改动后必须维护 changelog
仓库 MUST 维护根目录 `CHANGELOG.md`，用于记录每次实际改动的日期、范围、说明与验证结果。

#### Scenario: 完成一次代码或文档变更
- **WHEN** 维护者完成一次实际改动
- **THEN** 必须同步更新 `CHANGELOG.md`
- **AND** 记录本次变更涉及的模块
- **AND** 记录已执行的验证或未验证风险

### Requirement: 实施前必须先评估完整功能链路
维护者在开始实现前 MUST 先评估目标改动所在的完整功能链路，不能只按局部表象或既有惯性直接下手。

#### Scenario: 准备修改某个页面或组件
- **WHEN** 维护者准备修改页面、组件、接口或状态逻辑
- **THEN** 应先识别该改动前后的上游触发、当前承接和下游结果
- **AND** 不应在未理解完整链路前直接提交局部实现

### Requirement: 任何改动都必须评估功能连续性
任何实际改动在实施前 MUST 评估功能连续性，避免只修局部而导致链路断裂、状态失配或交互回退。

#### Scenario: 修改单个局部实现
- **WHEN** 维护者只改动一处局部实现
- **THEN** 仍应检查相关入口、详情、提交、回写、跳转或回退链路是否保持连续
- **AND** 如果存在链路断裂风险，应先补充设计说明或拆分任务

### Requirement: 任何改动都必须评估测试影响
任何实际改动在提交前 MUST 评估测试影响，包括已有测试是否需要更新、是否需要补充回归测试，以及是否引入未覆盖风险。当改动触及认证入口、关键 API handler 或已存在失败的默认测试基线时，维护者 MUST 先恢复可依赖的绿色基线，再为受影响入口补充直接回归测试。

#### Scenario: 默认单元测试命令存在失败用例
- **WHEN** 维护者准备继续补充同一链路的新测试或修复入口实现
- **THEN** 必须先修正或同步已有失败测试，使受影响的默认测试命令恢复到可依赖状态
- **AND** 不应在已知测试失败的基线上继续累积新的回归用例

#### Scenario: 改动认证入口或关键 API route
- **WHEN** 维护者修改 `middleware`、学习续接 API、复习提交 API 或其对应的入口规则
- **THEN** 必须为该入口增加或维护直接自动化测试，覆盖参数校验、重定向或响应结果，以及关键 service 入参透传
- **AND** 不能仅依赖更高层页面测试或手工回归来间接保护这些入口

#### Scenario: 完成关键测试补强后准备收尾
- **WHEN** 维护者完成 `middleware` 或高优 API handler 测试补强并准备结束本次改动
- **THEN** 必须执行受影响的默认测试命令和新增的定向测试命令
- **AND** 必须在变更记录或说明文档中留下本次回归范围和验证结果

### Requirement: 认证与高成本接口改动必须先审计边界与失败保护
维护流程 MUST 在实现任何认证入口、登录跳转规则、接口访问边界或高成本外部调用改动前，先审计请求入口、访问控制、失败保护与回归测试范围。维护者不得只修补单个 route 或页面判断，而忽略对应的中间件、客户端跳转、服务端调用和异常路径。

#### Scenario: 准备修改登录跳转或认证入口
- **WHEN** 维护者准备调整登录页、注册页、认证中间件或相关重定向规则
- **THEN** 必须先核对客户端与服务端是否使用同一套安全校验规则
- **AND** 必须把非法 `redirect`、默认回退和合法站内路径纳入回归测试

#### Scenario: 准备修改会触发外部模型的接口
- **WHEN** 维护者准备调整会触发外部模型调用或重型解析的 API route
- **THEN** 必须先确认该接口的访问边界、超时策略、错误收敛方式和受影响测试
- **AND** 不得在未审计匿名访问成本和失败保护前直接扩大接口暴露面

#### Scenario: 准备调整高成本或敏感 API route
- **WHEN** 维护者准备修改会触发模型调用、批量重生成、重型解析或用户资料热路径的 API route
- **THEN** 必须先盘点 middleware 边界、route 内权限、输入规模限制、错误收敛和测试覆盖
- **AND** 不得只修补单个 route 的局部判断而忽略完整请求链路

### Requirement: 项目维护文档必须覆盖组件分层与组件库说明
仓库 MUST 在维护入口文档中明确组件分层规则，并提供一份可持续维护的组件库说明，帮助维护者快速判断组件归属与复用边界。

#### Scenario: 维护者评估组件是否应进入公共层
- **WHEN** 维护者发现某个组件被多个 feature 共同使用，或准备新建一类可复用组件
- **THEN** 他必须能够通过维护文档与组件库说明判断该组件应进入公共层还是继续留在 feature 内
- **AND** 文档必须同时覆盖推荐做法和不建议抽公共的反例

### Requirement: 维护流程必须识别并治理超重页面与容器组件
仓库 MUST 在维护流程中识别明显超重的页面和 feature 容器，并优先通过拆分降低单文件职责复杂度。

#### Scenario: 维护者发现核心入口文件持续膨胀
- **WHEN** 维护者发现某个页面或 feature 容器已经同时承担大量状态、派生逻辑、动作编排和视图渲染
- **THEN** 必须先评估是否需要拆分为 hook、logic、section component 或局部容器
- **AND** 应在维护文档或 OpenSpec change 中写清拆分边界与回归计划

### Requirement: 高状态密度页面必须优先按数据流与局部交互边界拆分
仓库 MUST 在维护 `review`、`scenes` 这类同时承载缓存、阶段状态、手势或多弹层装配的页面时，优先按数据流、局部交互和 overlay 装配边界拆分，而不是继续把所有行为保留在单个 `page.tsx` 中。

#### Scenario: 维护者继续扩展 review 或 scenes 页面
- **WHEN** 维护者发现 `review/page.tsx` 或 `scenes/page.tsx` 同时承担数据刷新、局部状态机和大块 JSX 装配
- **THEN** 必须先评估是否拆成页面级 data hook、interaction controller 或局部 overlay/component
- **AND** 应在 OpenSpec change 中写清本轮拆分优先处理的职责块和对应回归范围

### Requirement: 触摸交互必须兼容浏览器被动监听约束
仓库 MUST 在实现下拉刷新、拖拽或其它触摸移动交互时，避免在被动监听上下文里直接调用 `preventDefault()`，以免产生浏览器运行时告警或导致默认滚动阻止失效。

#### Scenario: PullToRefresh 处理下拉手势
- **WHEN** 用户在支持页面顶部执行下拉刷新手势
- **THEN** 系统必须在不触发被动监听告警的前提下维持当前刷新阈值与 `app:pull-refresh` 事件契约
- **AND** 不得因为监听修复改变现有启用路径或下拉触发结果

### Requirement: 重入口后续拆分必须优先继续收口页面动作与分支装配
仓库 MUST 在处理 `chunks/page.tsx`、`lesson-reader.tsx` 这类已经过一轮减重但仍然明显过重的入口时，优先继续拆分页面动作编排、section 装配和分支控制，而不是先拆零散展示原子或直接做公共化。

#### Scenario: 维护者继续治理 chunks 或 lesson reader
- **WHEN** 维护者发现 `chunks/page.tsx` 或 `lesson-reader.tsx` 仍同时承担多组局部动作、分支装配和大块 JSX
- **THEN** 必须先评估是否拆成页面动作模块、selection/controller、section 组件或 sheet 装配模块
- **AND** 不应为了减行数优先抽离不稳定的 shared 组件或零散原子组件
