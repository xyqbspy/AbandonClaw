# 测试策略

本文档用于约束项目在 AI 协作开发、重构迭代和功能演进阶段的测试策略。

目标：
- 让测试成为“行为约束”和“回归保护”，而不是单纯的绿灯工具
- 在允许重构、删功能、改测试的前提下，避免糊弄式通过测试
- 让 AI 在处理测试失败时先分析，再修改

---

## 链路文档与测试联动

若改动涉及主链路、状态流转、回写规则、恢复逻辑：
- 先检查对应 feature-flow 是否仍然成立
- 若链路已变化，先更新文档，再判断测试应保留、重写还是删除
- 验收测试应优先保护 feature-flow 中定义的核心规则

当前默认链路文档目录：

- `docs/feature-flows/today-recommendation.md`
- `docs/feature-flows/scene-training-flow.md`
- `docs/feature-flows/session-resume.md`
- `docs/feature-flows/review-writeback.md`

## 0. 测试分层

项目中的测试分为三层：

### 1. 验收测试（最高优先级）
用于保护核心产品规则、用户路径、关键状态流转。

示例：
- Today 推荐优先级
- Session 恢复逻辑
- Scene 完成判定
- Review 回写状态
- Expression 阶段推进规则

要求：
- 优先表达业务语义，而不是实现细节
- 不应轻易删除或弱化
- 若要修改，必须说明对应的产品变化依据

---

### 2. 业务级集成测试
用于保护模块之间是否真正串联起来。

示例：
- Scene -> Session -> Review 主链路
- 创建 Expression Item 后 Today 推荐变化
- 某学习动作是否正确写入记录层

要求：
- 优先验证模块协作
- 避免过度依赖内部实现细节
- 在重构期仍应尽量保留主链路覆盖

---

### 3. 单元测试
用于保护纯逻辑、纯函数、状态转换、小范围局部行为。

示例：
- evidence 判定函数
- selector
- parser / formatter
- 状态计算函数

要求：
- 可以随重构调整
- 不应承担完整产品正确性的证明责任
- 若过度依赖实现细节，应重写或降级

---

## 1. 当前阶段策略

### 重构 / 演进阶段
当前项目允许：
- 修改单元测试
- 删除已废弃功能对应测试
- 将旧实现细节测试改写为更稳定的行为测试

但必须满足：
- 先说明测试原本保护什么行为
- 说明该行为是保留、替换还是废弃
- 不允许为了通过测试而机械弱化断言
- 不允许用 mock 掩盖真实问题

### 稳定阶段
当项目进入相对稳定阶段后：
- 优先保留现有业务语义测试
- 未记录需求变更时，不应随意修改测试语义
- 涉及核心规则的测试修改，应附变更依据

---

## 2. 测试失败处理原则

遇到测试失败时，先做归因，再决定修改代码还是测试。

必须先判断失败属于哪类：

- A. 代码实现有 bug，测试是对的
- B. 功能/需求已变化，测试已过时
- C. 正在重构，测试依赖旧实现细节，应改写
- D. 测试设计有问题（断言错误、mock 不合理、耦合过深）
- E. 历史遗留低价值功能/测试，应删除

默认原则：
- 若测试保护的是当前仍有效的业务语义，优先修改实现
- 若业务语义已变化，允许同步修改测试
- 若测试只是旧实现细节，优先改写为行为测试
- 若功能已废弃，允许删除对应测试，但必须记录理由

---

## 3. 禁止事项

以下行为默认禁止：

- 为了通过测试直接删除关键断言
- 无解释地放宽严格断言
- 用 mock 掩盖真实行为问题
- 不分析失败原因，直接批量改单测
- 在未说明原因的情况下删除测试文件
- 把实现细节测试误当成业务语义测试来强保留

---

## 4. AI 修改测试的输出要求

当 AI 处理测试失败时，应先输出：

1. 失败的是哪个测试
2. 该测试原本保护什么行为
3. 它属于：
   - 业务语义
   - 模块行为
   - 实现细节
   - 历史遗留行为
4. 建议处理方式：
   - 修改实现
   - 修改测试
   - 重写测试
   - 删除旧测试
5. 原因
6. 风险
7. 影响范围

未完成上述分析前，不应直接批量修改测试。

---

## 5. 新功能测试建议

默认建议顺序：

1. 先写验收标准
2. 再写业务级测试
3. 最后补单元测试

不要只靠单元测试证明功能完成。

推荐每个关键功能至少补一份 `acceptance checklist`，例如：

```md
功能：Today 推荐逻辑

验收标准：
- 有未完成 session 时，优先展示继续当前训练
- review 积压超过阈值时，优先推荐 review
- 新 scene 不应覆盖未完成 session
- scene 完成后应推荐下一个合理动作
6. 测试执行策略

默认只运行最小相关测试：

当前模块测试
直接受影响的页面测试
直接受影响的集成测试

默认不跑全量测试，除非：

用户明确要求
局部失败提示存在连锁影响
修改涉及主链路或公共能力
7. 删除功能时的测试策略

当删除低价值/废弃功能时：

可以同步删除对应测试
但必须说明：
删除的是哪个功能
为什么该功能已废弃或被替代
是否影响主链路
是否需要新增替代性的测试保护新行为

若删除导致用户可感知行为变化，应同步补充文档或 spec 说明。

8. 核心规则白名单（建议维护）

对于项目中的核心规则，建议维护白名单。
除非明确要求变更，否则不得擅自改变其语义。

例如：

Today 推荐优先级
Scene / Session / Review 主链路
Expression 阶段定义
学习证据模型
关键状态流转规则
9. 推荐 Prompt 模板

处理测试失败时，可使用以下提示：

测试失败了。当前项目处于重构和功能调整阶段，允许改测试，但不要为了通过而机械弱化测试。

请先：
1. 说明失败测试原本保护什么行为；
2. 判断这是代码 bug、测试过时、实现细节耦合，还是旧功能应删除；
3. 优先保留核心业务语义；
4. 如果改测试，改成表达“当前正确行为”的测试；
5. 如果删测试，说明对应功能为何已废弃；
6. 禁止只删断言、放宽断言、加 mock 糊过去。

先输出分析和建议，再执行修改。
