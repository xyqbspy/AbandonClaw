# 变更接入模板

接需求时先填最小块。只有当任务不是 Fast Track，或处理中暴露主链路 / 状态流 / 数据流 / 缓存 / 权限 / 稳定规则风险时，才继续补后面的详细检查。

## 0. 最小填写块

- 任务分类：
- 输出语言：默认中文；不得输出纯英文回答；代码、命令、API 名和错误原文可保留英文。
- 问题定位：
- 涉及模块：
- 是否影响主链路：
- 北极星判断：
- 最小改动方案：
- 风险与影响范围：
- 最小测试方案：
- 是否需要更新文档：
- 是否需要同步产品/技术总览：
- 本轮收口项 / 明确不收项：

## 1. 任务分流速查

| 类型 | 适用场景 | 收尾标准 |
| --- | --- | --- |
| Fast Track | UI / 样式 / 文案、lint / type / import、局部测试、小范围不改语义重构 | 最小相关测试通过，必要文档最小同步即可 |
| Cleanup / Removal | 删除废弃入口、重复逻辑、旧状态、旧测试或旧文档 | 说明删除依据和影响范围；若改变用户行为，升级 Spec-Driven |
| Spec-Driven | 改业务行为、主链路、状态流、数据流、API、数据模型、权限、缓存、跨页面一致性或维护规范 | proposal / design / spec delta / tasks / 文档 / stable spec / archive 按完成态收尾 |

职责入口保持分层：

- `AGENTS.md`：强制红线和任务分流。
- `docs/README.md`：告诉你应该读哪类文档。
- `openspec/specs/project-maintenance/spec.md`：长期稳定契约。
- `docs/dev/project-maintenance-playbook.md`：执行清单和日常维护说明。
- 本文件：接需求阶段的问题分析骨架。

## 2. 上下文预算

- 必读上下文：
- 明确不读上下文：
- 补读触发条件：
- 可能升级条件：

默认规则：

- Fast Track：只读入口规则、相关文件、必要索引和最小测试上下文。
- Cleanup：读删除对象、引用方和影响范围。
- Spec-Driven：先用 `docs/README.md` 和索引定位，再按 feature-flow、domain-rules、stable spec、system-design、代码递进阅读。

## 3. 完整链路检查

非微小改动需要回答：

- 这次需求是否服务产品北极星：让每一次场景学习，都沉淀为用户在未来真实场景中能回忆、能使用、能迁移的表达资产？
- 它推进的是进入学习、语境理解、表达沉淀、未来回忆、真实使用或场景迁移，还是只是增加功能入口？
- 上游入口是什么？
- 当前节点承担什么职责？
- 下游会写回哪里？
- 失败、回退、空态会落到哪里？
- 是否影响 `today -> scene -> chunks -> review` 中的某一段？

## 4. 稳定性收口检查

- 这次需求是否暴露旧规则漂移、重复语义、缺失文档、缺失测试、边界不清或旧兼容语义未收口？
- 本轮必须顺手收口哪些项？
- 哪些项明确不在本轮处理？
- 不处理的原因是什么？
- 延后风险记录在哪里？

建议直接填写：

- 本轮收口项：
- 明确不收项：
- 延后原因：
- 风险记录位置：

## 5. 测试影响检查

- 哪些已有测试会直接受影响？
- 是否需要更新旧测试？
- 是否需要新增回归测试？
- 如果本轮不补测试，剩余风险是什么？

## 6. OpenSpec 判断

- 是否应该先建 change？
- 如果建 change，change id 是什么？
- proposal 覆盖哪些影响范围？
- delta spec 修改哪个 capability？

## 7. 收尾要求

- 是否更新 `docs/dev/dev-log.md`？
- 是否需要同步 `docs/meta/product-overview.md` / `docs/meta/technical-overview.md`？
- 是否需要同步 stable spec？
- 是否需要 archive？
- 是否运行 `pnpm run maintenance:check`？
- 若本次完成态将直接进入 `main` 且存在用户可感知变化，是否更新正式 `CHANGELOG.md`？
