# 变更接入模板

在开始任何非微小改动前，先用这份模板快速过一遍。

## 0. 任务分流速查

先用这张表做第一轮判断，避免小改动误套大流程：

| 类型 | 适用场景 | 收尾标准 |
| --- | --- | --- |
| Fast Track | UI / 样式 / 文案、lint / type / import、局部测试、小范围不改语义重构 | 最小相关测试通过，必要文档最小同步即可 |
| Cleanup / Removal | 删除废弃入口、重复逻辑、旧状态、旧测试或旧文档 | 说明删除依据和影响范围；若改变用户行为，升级 Spec-Driven |
| Spec-Driven | 改业务行为、主链路、状态流、数据流、API、数据模型、权限、缓存、跨页面一致性或维护规范 | proposal / design / spec delta / tasks / 文档 / stable spec / archive 按完成态收尾 |

职责入口保持分层：
- `AGENTS.md`：强制红线和任务分流。
- `docs/README.md`：告诉你应该读哪类文档。
- `openspec/specs/project-maintenance/spec.md`：长期稳定契约。
- `docs/dev/project-maintenance-playbook.md`：执行清单和日常维护说明。
- 本文件：接需求阶段的问题分析骨架。

## 1. 变更摘要

- 变更名称：
- 目标模块：
- 变更类型：
- 功能行为
- 数据流 / 状态流
- 缓存
- 测试
- 跨页面 UI 一致性
- 维护规范

## 2. 完整链路检查

- 上游入口是什么？
- 当前节点承担什么职责？
- 下游会写回哪里？
- 失败、回退、空态会落到哪里？
- 这次改动会不会影响 `today -> scene -> chunks -> review` 中的某一段？

## 2.1 上下文预算检查

- 本轮任务属于 Fast Track、Cleanup 还是 Spec-Driven？
- 本轮必须读取的最小上下文是什么？
- 哪些历史 archive、旧 proposal、dev-log 或无关模块明确不读？
- 如果处理中发现主链路、状态流、数据流、缓存、权限或稳定规则风险，需要补读哪些文档？
- 是否需要因为补读结果重新升级任务类型？

建议直接按下面格式填写：

- 必读上下文：
- 明确不读上下文：
- 补读触发条件：
- 可能升级条件：

## 3. 功能连续性检查

- 如果只改局部，这条链路会不会在别处断掉？
- 是否存在状态不同步风险？
- 是否存在按钮、文案、禁用态、跳转路径不一致风险？
- 是否需要拆成多步改动，避免一次改太多？

## 4. 测试影响检查

- 现有哪些测试会直接受影响？
- 现有测试是否需要更新？
- 是否需要新增回归测试？
- 如果这次不补测试，剩余风险是什么？

## 5. 稳定性收口检查

- 这次需求是否同时暴露旧规则漂移、重复语义、缺失文档、缺失测试、边界不清或旧兼容语义未收口？
- 如果暴露了，本轮必须顺手收口哪些项？
- 哪些项明确不在本轮处理？
- 不处理的原因是什么？
- 如果延后，风险记录到哪里？

建议直接按下面格式填写：

- 本轮收口项：
- 明确不收项：
- 延后原因：
- 风险记录位置：

## 6. OpenSpec 判断

- 这次是否应该先建 change？
- 如果建 change：
- change 名称是什么？
- proposal 要覆盖哪些影响范围？
- delta spec 要改哪条 capability？

## 7. 收尾要求

- 是否更新 `docs/dev/dev-log.md`
- 若代码已合并 `main` 且存在用户可感知变化，是否更新正式 `CHANGELOG.md`
- 是否记录验证结果
- 是否满足 archive 前检查条件
