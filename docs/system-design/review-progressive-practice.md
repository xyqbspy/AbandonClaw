# Review 递进式练习维护说明

## 1. 文档目的

这份文档专门说明 `review` 页里新的递进式练习流怎么组织、当前哪些阶段已经接到正式后端、哪些还只是前端 TODO 占位，以及失败后应该如何降级而不是直接判错结束。

它和 `docs/system-design/review-source-mapping.md` 的分工不同：

- `review-source-mapping.md` 解释复习项来自哪里、和原场景是什么关系
- 本文解释单次复习内部怎么从识别推进到输出

## 2. 当前普通表达复习的阶段

当前普通表达复习按以下阶段推进：

1. `micro_recall`
2. `confidence_check`
3. `variant_rewrite`
4. `full_output`
5. `feedback`

对应前端主页面：

- `src/app/(app)/review/page.tsx`

对应阶段映射逻辑：

- `src/app/(app)/review/review-page-selectors.ts`

## 3. 各阶段的目标

### 3.1 微回忆 `micro_recall`

目标：

- 先不直接暴露表达本体
- 让用户只根据语境和释义，在脑中先主动提取

当前做法：

- 页面先显示语境与翻译
- 用户点击后再展开参考句

这一步主要解决“看见就认识，但自己提不起来”的问题。

### 3.2 熟悉度 / 输出信心 `confidence_check`

目标：

- 区分“只是眼熟”与“已经有主动输出信心”

当前前端信号：

- 眼熟，能认出来 / 还是陌生
- 能主动说出来 / 还需要提示

这一步目前主要用于本次复习内部路径，不直接回写正式后端字段。

### 3.3 变体改写 `variant_rewrite`

目标：

- 让用户把同一表达迁移到新对象、新视角或新时态

当前前端内置的改写方向：

- 改成对自己说
- 改成对同事说
- 改成昨天发生的事

这一步当前仍是前端本地草稿，用来承接后续更细的命中率、自然度和 AI 反馈。

### 3.4 完整输出 `full_output`

目标：

- 不再依赖填空
- 直接输出一整句或两整句

这是当前递进流里最接近真实输出能力的阶段。

当前仍先保留为本地草稿，不直接生成新的正式学习统计字段。

### 3.5 结果判断 `feedback`

目标：

- 把前面几步训练的结果收口成当前仍然稳定的复习提交动作

当前正式后端回写仍沿用：

- `again`
- `hard`
- `good`

也就是说：

- 前面的递进式阶段用于训练和补充判断
- 真正正式影响当前复习队列的，仍是最后这一步提交

## 4. 当前哪些能力是正式信号，哪些是 TODO

### 4.1 当前正式接通的能力

- 复习队列加载
- 原场景可访问性判断
- 普通表达 `again/hard/good` 提交
- 熟悉度 `recognition_state` 正式记录
- 输出信心 `output_confidence` 正式记录
- 完整输出状态 `full_output_status` 正式记录
- 场景回补的正式记录与完成信号

### 4.2 当前明确仍是 TODO 的能力

- 变体改写的命中率和自然度评估
- 完整输出的 chunks 命中率与缺失点分析
- 基于递进阶段结果的更细粒度调度策略

新增正式字段、聚合摘要和 `today` 消费规则，见：

- `docs/domain-rules/review-practice-rules.md`

其余未接通能力在页面上仍只能作为训练位和说明存在，不能伪造成已经写入正式学习状态。

## 5. 场景回补与普通表达的关系

场景回补暂时仍保留现有正式链路：

- `recall`
- `practice`
- `feedback`

原因是它已经和 `learning-api`、practice run / attempt / complete 链路打通。

所以当前第一期的策略是：

- 普通表达先扩展成递进式训练流
- 场景回补继续保持正式记录稳定
- 两者仍共用同一个 `review` 舞台和进度入口

## 6. 失败后的降级策略

当前递进流不把“高强度阶段没写好”直接等价成复习失败。

第一期约定：

- 用户如果在变体改写阶段卡住，可以先展开参考，再继续写本地草稿
- 用户如果在完整输出阶段卡住，仍可以回看上一步改写结果和参考句
- 场景回补失败时，继续沿用“回到原场景继续练”的正式链路

后续如果要加入更明确的自动降级逻辑，例如：

- 完整输出失败自动回退到改写
- 改写失败自动回退到微回忆

需要继续走新的 OpenSpec change 或扩充当前规范。

## 7. 后续维护注意点

- 如果把熟悉度、自信度、改写或完整输出接入正式后端，必须同步更新 OpenSpec spec、测试和这份文档
- 如果 `today`、dashboard 或其他聚合页要显示新的复习阶段结果，必须先定义稳定字段来源
- 如果改写方向从固定三种扩展成动态生成，也要先评估评估逻辑和失败降级是否会一起变化

## 8. 建议回归点

- 普通表达：从微回忆一路走到结果判断
- 普通表达：未完成熟悉度选择时不能继续下一阶段
- 普通表达：未填写改写草稿时不能进入完整输出
- 普通表达：未填写完整输出草稿时不能进入结果判断
- 场景回补：继续保持原有正式提交与刷新行为
