# UI 风格指南

## 1. 目标

这份文档用于约束新增页面、功能入口和跨页面 UI 调整时的风格判断，避免每次从局部页面重新发明按钮、卡片、布局和颜色。

它不替代组件库说明，也不定义完整设计系统。维护时先用这份文档判断 UI 方向，再回到对应组件和 stable spec。

## 2. 适用范围

优先适用于：

- 新增页面或主要功能入口
- 新增跨页面可复用组件
- 调整学习主链路里的主要动作、保存动作或危险动作
- 调整详情页、sheet、modal、toolbar、空态、loading 等通用 UI 结构
- 发现同类按钮、卡片、列表或动作区在多个页面出现明显漂移

不必默认适用于：

- 单个页面内的小文案修正
- 不改变语义的局部 spacing 微调
- 测试、类型、lint 或 import 修复

这些小修仍按 Fast Track 处理；如果处理中发现跨页面规则漂移，再升级。

## 3. 页面布局原则

新增页面先判断页面角色，再决定布局：

- 工作台页面：用于反复操作和扫描，例如 `today`、`chunks`、`admin`。应优先保持信息密度、清晰分区和稳定工具栏，不做营销式 hero。
- 学习流程页面：用于推进一个学习任务，例如 `scene`、`lesson`、`review`。应突出当前任务、下一步动作和反馈，不堆无关卡片。
- 详情或浮层：用于围绕一个表达、句子或任务查看上下文。应复用既有 sheet、detail block、footer action 和音频动作结构。
- 管理页面：用于维护数据和排查问题。应优先使用统一列表、筛选、菜单和 admin 操作按钮。

页面 section 不默认做成大卡片。只有单个重复项、详情浮层、modal 或明确需要框定的工具区才使用 card。

## 4. 页面骨架模板

这些模板是默认结构，不是强制布局。新增页面时优先从相近模板出发，再按业务需要增减。

### 工作台页面

适用于 `today`、`chunks`、`admin` 这类需要反复扫描和操作的页面。

推荐骨架：

1. 页面标题区：说明当前页面对象和一个主要结果，不放大段说明。
2. 摘要区：展示最关键的 1-3 个状态或统计。
3. 工具区：筛选、搜索、排序、视图切换等操作集中放置。
4. 主内容区：列表、表格、任务卡或工作流主体。
5. 局部反馈区：empty / error / loading 贴近对应内容，不把整页打断。

默认避免：

- 顶部做大 hero。
- 多个互相竞争的主 CTA。
- 每个 section 都包 card，导致信息密度过低。

### 学习流程页面

适用于 `scene`、`lesson`、`review` 这类推进学习任务的页面。

推荐骨架：

1. 当前任务区：用户此刻要读、听、回忆或作答的内容。
2. 上下文辅助区：提示、翻译、来源、例句或表达说明。
3. 主动作区：推进当前任务的唯一主要动作。
4. 次动作区：保存、跳过、查看详情、加入复习等辅助动作。
5. 结果反馈区：本题、本轮或本场景完成后的反馈和下一步。

默认避免：

- 在主任务旁放太多并列操作。
- 用 toast 代替任务完成后的页面反馈。
- 把保存类动作做得比当前学习推进动作更强。

### 详情或浮层

适用于表达详情、句子详情、来源场景回看、确认弹框和编辑浮层。

推荐骨架：

1. 标题或焦点对象：当前正在看的表达、句子、任务或操作对象。
2. 核心信息区：翻译、解释、例句、来源或状态。
3. 扩展信息区：相似/对照表达、使用场景、历史结果。
4. 动作 footer：主动作、次动作、危险动作保持固定位置和层级。
5. 关闭或返回：清楚表达退出当前上下文的路径。

默认避免：

- 在浮层内部再嵌套完整页面结构。
- 让危险动作和主动作并排且视觉相近。
- 打开详情后丢失来源上下文或回退路径。

### 管理页面

适用于 `/admin` 下维护数据、音频、缓存、业务事件的页面。

推荐骨架：

1. 页面标题区：说明维护对象。
2. 筛选和刷新区：让维护者快速缩小范围。
3. 列表或面板区：展示可操作对象和状态。
4. 操作区：统一使用 admin 操作按钮和确认流程。
5. 结果/错误区：保留 requestId、失败原因或可排查信息。

默认避免：

- 把 admin 页面做成学习页风格。
- 危险动作无确认。
- 维护信息只靠 toast 闪过。

## 5. 视觉语言

当前产品默认是学习工作台风格，不是营销页或内容杂志风格。

默认视觉口径：

- 保持清爽、克制、可扫描，不用大面积装饰背景。
- 信息层级靠标题、分区、间距和动作层级表达，不靠大量颜色堆叠。
- 一个页面只保留一个最强主动作区域，避免多个大按钮同时抢焦点。
- 同一类信息在不同页面使用相近结构，例如摘要、空态、loading、详情 block、动作 footer。
- 图标用于辅助识别动作，不替代必须理解的文本；纯图标按钮必须有可访问名称或 tooltip。

默认避免：

- 新增单页专属大渐变、局部主色或装饰背景。
- 把运营式 hero 用在工作台或学习流程页。
- 用大卡片嵌套小卡片制造层级。
- 为了“更显眼”把多个次动作做成主按钮。
- 在同一页面混用多套圆角、阴影、边框厚度和按钮高度。

## 6. 间距、密度和容器

页面密度按使用场景决定：

- 工作台页面优先可扫描：列表、筛选、摘要和操作区要紧凑但不拥挤。
- 学习流程页优先专注：当前句子、表达或题目要明显，辅助信息不抢主任务。
- 详情浮层优先上下文完整：内容、解释、例句、动作 footer 要分区清楚。
- 管理页优先批量操作：筛选、表格、菜单和危险动作要位置稳定。

容器规则：

- 页面主体优先用固定最大宽度或已有布局容器，不在新页面随意扩大/缩小内容宽度。
- toolbar、filter、summary、content、footer action 要各自承担清晰职责。
- card 用于重复项、详情、modal 或明确工具区；页面大 section 不默认包成 card。
- 不做 card 套 card；如果需要层级，优先用标题、分隔、列表或局部 block。

## 7. 状态反馈

新增 UI 必须考虑这些状态，不只做 happy path：

- loading：使用已有 loading 文案、skeleton 或按钮 loading 状态，不在页面里重复写一套动画。
- empty：说明当前为什么为空，并给出一个合理下一步；不要只显示“暂无数据”。
- error：错误文案要可恢复，优先提供重试、返回或替代路径。
- disabled：必须能看出为何不可用，避免把主要动作静默置灰。
- success：只反馈用户刚完成的动作，不用 toast 替代页面状态更新。

如果状态会跨页面复用，优先提取 shared 组件或 helper；如果只服务单页，可以留在 feature 内。

## 8. 通用 UI 模式

### 工具栏

工具栏用于承载搜索、筛选、排序、视图切换和批量动作。

- 搜索和筛选靠左，主要操作靠右。
- 移动端允许折叠为纵向排列或更多菜单。
- 不把业务说明、状态反馈和主要内容混进工具栏。

### 空态

空态必须回答三个问题：

- 当前为什么没有内容？
- 用户现在能做什么？
- 这个状态是否正常？

空态优先使用已有 empty-state 或相近 shared 组件；如果新增空态文案，要和页面主目标一致，不写泛泛的“暂无数据”。

### 确认弹框

确认弹框用于危险动作、不可逆操作或会造成明显状态变化的操作。

- 标题说明动作对象。
- 正文说明影响范围。
- 主确认按钮使用危险层级或明确动作文案。
- 取消动作保持弱层级。

### 列表项和任务项

列表项优先保持稳定扫描结构：

- 左侧或顶部放主对象。
- 中间放摘要、状态或来源。
- 右侧或底部放少量动作。
- 不在每个列表项里放完整详情。

### 表单和编辑

表单优先明确字段语义和保存结果：

- 标签不可只靠 placeholder。
- 保存动作和取消动作要区分主次。
- 校验错误贴近字段展示。
- 长文本编辑要考虑移动端高度和按钮可见性。

## 9. 响应式与可访问性

新增页面至少检查桌面和移动端两个断点：

- 移动端不应依赖 hover 才能发现主要动作。
- 主按钮、图标按钮和更多菜单要有足够点击面积。
- 长文本、英文表达、中文解释和按钮文案不能溢出容器。
- footer action、悬浮按钮、sheet 底部动作不能遮挡主要内容。
- 纯图标按钮必须有 `aria-label`、可访问名称或 tooltip。
- 表单、筛选、tab、segmented control 要能通过键盘或语义查询定位。

涉及 pointer / touch / wheel / 拖拽 / 滑动 / 下拉刷新 / 悬浮按钮时，还要遵守运行时防护要求：`releasePointerCapture` 前确认 capture 状态，`preventDefault()` 前确认事件可取消。

## 10. 动作层级

新增按钮先按动作语义分类，不按页面归属分类。

- 主动作：推进当前任务的唯一主要动作，例如开始学习、继续学习、开始复习、生成场景、提交本轮结果。
- 次动作：辅助主任务或沉淀资产，例如保存到表达库、保存句子、添加到表达库、保存并加入复习。
- 危险动作：删除、清空、重置、断开关系等不可轻易恢复的动作。
- 图标动作：播放、暂停、循环、关闭、更多、展开、复制等工具动作。

规则：

- 同一动作语义跨页面保持同一主次层级。
- 不为单个页面临时创造新主色、特殊边框或一次性按钮底色。
- 学习主链路按钮优先遵守 `learning-action-button-hierarchy`。
- 详情页 footer 动作优先遵守 `detail-footer-actions`。
- 音频动作优先使用 `src/components/audio` 的统一组件。
- admin 操作优先使用 `AdminActionButton`、`AdminLoadingActionButton` 或 `AdminConfirmActionButton`。

## 11. 组件归属

新增组件先看是否已有可复用组件，再判断落点：

- 纯 UI primitive -> `src/components/ui`
- 跨 feature 稳定复用 -> `src/components/shared`
- 音频动作 -> `src/components/audio`
- 管理后台子域复用 -> `src/components/admin`
- feature 私有业务组件 -> `src/features/<feature>/components`
- 页面级路由和数据编排 -> `src/app/*`

不要因为“样式看起来像”就抽公共。只有输入输出稳定、交互职责稳定、脱离原 feature 后名称仍清晰，才适合进入公共层。

具体组件分层以 [component-library.md](/d:/WorkCode/AbandonClaw/docs/system-design/component-library.md) 为准。

## 12. 样式写法约束

新增 UI 时优先顺序：

1. 复用已有业务组件或 shared 组件。
2. 复用 `src/components/ui` 的 variant。
3. 复用已有局部 helper 或封装。
4. 最后才写页面局部 class。

注意：把局部 class 收到 `today-page-styles.ts`、`review-page-styles.ts`、`scene-page-styles.ts` 这类文件，只是页面族内部样式收口，不是公共样式抽象。它适合用来减少 JSX 噪音和防止同一页面继续漂移，但不应该被当成最终设计系统。

需要公共化时，优先抽稳定 UI 语义，而不是抽 class 字符串：

- 底层一致性用 design tokens 表达，例如颜色、圆角、阴影、字体、间距。
- 跨页面稳定结构用 shared 组件或组件 variant 表达，例如空态、信息块、状态 pill、固定 footer action。
- 单个 feature 的组合视觉继续留在 feature-private styles，直到出现真实跨页面复用需求。

升级判断：

- 只有一个页面或一个 feature 使用：保留局部 styles。
- 多个组件都在同一 feature 内使用：优先 feature-private styles 或 feature 内组件。
- 2-3 个页面或 feature 出现相同 UI 语义，且 props 与交互职责稳定：再考虑 shared 组件。
- 只是颜色、圆角、阴影看起来相似：不抽 shared，先记录为候选。

禁止默认做法：

- 为单页手拼一套新的主按钮、次按钮或危险按钮。
- 用新的局部主色替代既有主次层级。
- 因为局部页面好看而破坏跨页面一致性。
- 把强业务容器直接迁到 `shared` 只为了目录整齐。
- 在多个页面复制同一组复杂 class，而不是提取稳定组件或 helper。

允许局部 class 的情况：

- 只服务当前页面的一次性布局。
- 不改变全局动作层级。
- 不形成跨页面重复。
- 不绕过已有组件的交互状态、可访问性或 loading / disabled 表达。

## 13. 常见反模式

新增或调整 UI 时，看到下面情况要先停一下：

- 页面看起来更完整，但首要任务更不明显。
- 新页面为了“有设计感”加入了和学习任务无关的装饰。
- 一个区域里同时出现多个同等强度 CTA。
- 保存、删除、继续、返回等动作视觉层级接近，用户需要猜。
- 同类 loading / empty / error 在不同页面各写一套。
- 组件抽到 shared 后，名称变空泛，props 仍然携带某个 feature 的内部状态。
- 移动端只能靠 hover、宽表格或横向拥挤布局才能操作。
- 页面为了局部效果绕过已有按钮、sheet、音频动作或 admin 组件。

## 14. 新增页面检查清单

新增页面或主要功能入口前，至少回答：

- 这个页面是工作台、学习流程、详情浮层还是管理页面？
- 是否可以套用上面的默认页面骨架？
- 视觉上是否仍是学习工作台风格，而不是营销页或装饰型页面？
- 首要动作是什么，是否已经有对应主按钮层级？
- 保存、沉淀、加入复习等次动作是否沿用现有层级？
- 是否需要复用 shared / audio / admin 组件？
- 是否引入了新的 card、toolbar、sheet、modal 或空态模式？
- loading、empty、error、disabled、success 状态是否都有落点？
- 移动端是否仍能发现主要动作，且按钮文案不溢出？
- 是否只是局部样式，还是会影响跨页面一致性？
- 是否需要更新 `component-library.md` 或对应 stable spec？

## 15. 什么时候必须走 OpenSpec

出现以下情况时，不按普通 Fast Track 直接做：

- 新增公共 UI 能力或通用组件。
- 改变跨页面按钮主次层级。
- 改变详情页 footer、sheet、modal 或音频动作的统一结构。
- 改变组件公共化边界或 feature-to-feature 依赖关系。
- 引入新的页面级布局范式。
- 批量统一多个页面的视觉表达。

局部文案、小范围 spacing、单页不改语义的样式修复，可以继续按 Fast Track 处理。
