## 2. 30 秒讲完这个项目是什么

这是一个用 Next.js + Supabase 做的英语学习全栈应用。

它不是普通的内容展示网站,而是围绕一条学习主线做闭环:**今日任务 → 场景学习 → 把表达存下来 → 复习 → 看进度**。场景给真实语境,把里面有价值的表达存下来变成自己的资产,复习把这些表达拉回主动回忆,今日任务把所有学习状态汇总起来告诉用户今天该做什么。

我做得比较深的部分包括:学习状态的服务端管理、表达资产的关系维护、文字转语音(后面简称 TTS)的音频生成和缓存链路、前端多层缓存和预加载、高成本接口的限流和治理、注册入口的安全防护、权限边界的逐步收紧,以及把 AI 能力真正融进学习流程而不是做成孤立的演示。

> **30 秒电梯版**:"我做了一个英语学习应用,核心想解决的不是'看懂内容',而是'学过的表达能不能在未来真实场景里用出来'。所以围绕场景学习、表达沉淀、复习训练做了一条闭环,前端用 Next.js 后端用 Supabase,做了比较深的音频缓存链路、AI 能力嵌入和后端接口治理。"

## 3. 2 分钟讲架构

项目是单仓库全栈结构 —— 前端页面、后端接口、数据库迁移都在同一个仓库里。

核心目录大致是:

- `src/app`:页面和后端接口的入口
- `src/features`:按业务分类组织的页面级组件
- `src/components`:公共 UI 组件
- `src/lib/server`:服务端业务逻辑、权限、数据访问
- `src/lib/cache`:前端缓存和预加载
- `supabase/sql`:数据库的表结构、权限规则和迁移历史

架构思路一句话总结:

> 页面层尽量薄,只负责"该展示什么、该派发什么事件";业务判断都下沉到下面的逻辑层;跨页面共享的状态尽量放服务端管;前端缓存负责让体验流畅;Supabase 提供账号体系、数据库和文件存储。

## 4. 为什么这样选

### 4.1 为什么用 Next.js + Supabase 单仓全栈

这个项目还在快速成型阶段,学习主线、复习、表达管理、音频、AI 这些功能需要频繁互动。单仓全栈的好处:

- **页面和接口一起改**,不用前后端分两个仓库联调
- **服务端的聚合逻辑可以贴着页面入口写**,今日任务、进度页这种"汇总展示"的页面写起来直接
- **Supabase 直接提供账号、数据库、文件存储**,不用自己搭一套
- **数据库结构、权限规则和业务代码都能在同一个仓库里追**

它不是最终架构。等以后 TTS、推荐、AI 生成这些独立能力负载变大,可以再拆出来成独立服务。

### 4.2 为什么没用重型的全局状态管理(比如 Redux)

这个项目真正需要长期保存的不是前端 UI 状态,而是**学习进度、复习信号、表达资产、音频资源** —— 这些都要跨页面、跨会话、跨设备保存。把这些都塞进一个前端的大仓库,关掉浏览器就没了,解决不了问题。

所以做法是:

- 页面内的临时交互,用 React 自己的 state 处理
- 派生出来给页面展示的数据,放专门的"展示逻辑"文件里
- 能恢复、能聚合、能追踪的核心状态,放服务端管
- 列表和详情的体验优化,用本地缓存兜底

这比硬塞一个 Redux 更贴合业务实际。

### 4.3 为什么页面要薄

项目有一条很重要的约定:**页面只负责"编排",不吞业务**。

举个例子:场景详情页不是普通的展示页,它要同时管:数据加载、视图切换、学习状态同步、音频播放、表达保存、练习生成、变体学习。这些全塞进一个页面文件,很快就会变成几千行没人敢动。

所以项目把复杂度拆成几层:

- **展示逻辑层**:负责"页面要展示什么数据"的派生计算
- **判断逻辑层**:纯函数,只做判断和状态转换(比如"这个表达算不算熟了")
- **动作调度层**:负责"用户点了按钮之后,该按什么顺序执行哪些操作"
- **数据写入层**:负责真正去调接口、改数据
- **组件层**:只管展示和交互

面试可以说:这个项目不是按技术名词堆文件,而是按"页面职责"拆复杂度。

## 5. 技术亮点一:学习状态的服务端闭环

学习不是在前端点一下"完成"按钮就算完成,而是**服务端维护一套完整的状态机**。

关键动作有四个:

- **开始**:用户进入一个场景,标记开始学习
- **进度**:用户在场景里停留、阅读、切换内容时,增量回写进度
- **暂停**:用户离开页面时记录暂停
- **完成**:做完练习或者变体学习,推进到完成态

这套设计的价值:

- 用户关掉浏览器再回来,能从上次的位置继续
- 今日任务页能告诉用户"上次没学完的还有这个"
- 进度页能统计长期学习量
- 复习页能从真实学习行为里拿信号,而不是凭空猜
- 学习进度不是临时 UI 状态,而是可以长期沉淀的数据

如果面试官问"这和普通学习页面有什么区别":

> 普通页面只记录用户当前看到哪里。这个项目把学习行为做成服务端状态机,之后的"今日推荐""复习训练""进度统计"都消费同一份状态。这样不管用户在哪个设备、什么时间回来,状态都是连续的。

## 6. 技术亮点二:表达资产模型

这个项目不是只让用户读完一个场景就完事,而是**把场景里的表达沉淀成"个人资产"**。

一个表达大致会经历这样的生命周期:

1. 用户在场景里遇到一个真实表达(比如 "I'm just trying to figure things out")
2. 把它存下来
3. 在表达管理页里补全翻译、例句、用法说明、语义焦点
4. 生成相似表达和对比表达
5. 把相关表达组织成"表达簇"
6. 进入复习和今日任务的长期循环

值得讲的点是:**"表达管理页"不是收藏夹,而是表达的工作台**。它让"学过一个场景"变成"积累了一组能复用的表达"。

面试可以这样说:

> 我把表达从"一次性内容消费"里抽出来,做成可以补全、关联、复习和迁移的资产层。场景是输入入口,表达管理是资产沉淀层,复习是长期回忆训练。

## 7. 技术亮点三:文字转语音(TTS)的音频链路

音频链路是项目里很值得讲的一块,因为它**不是简单的"请求一个音频网址来播"**。

完整链路是这样:

1. 服务端拿到文本(或分好段的文本),调用微软的 Edge TTS 生成音频
2. 把音频上传到 Supabase 的文件存储里
3. 服务端给前端返回一个**临时下载链接**(有效期是有限的)
4. 前端拿到链接后做几层缓存:内存里记一份网址、浏览器本地存一份音频文件
5. 之后再播放同一段音频,优先用本地的,不再请求服务端
6. 即使临时链接过期了,本地存的音频还能继续播

这个设计同时兼顾了:

- 第一次播放可用
- 第二次播放更快
- 减少重复生成音频(TTS 本身慢且收费)
- 弱网环境也有兜底
- 高成本的 TTS 接口不会被反复打爆

> **30 秒电梯版**:"音频在我们这是高成本资源 —— 生成慢、收费、用户会反复听。所以我把它做成稳定资源:服务端按内容算一个钥匙,文件存到 Supabase 上;客户端做了内存缓存、浏览器本地音频缓存、避免重复请求几层,即使下载链接过期了,本地音频还能继续播。"

### 7.1 前端这几层缓存具体是什么(追问时展开)

「内存里记一份网址」「浏览器本地存一份音频文件」其实是两类完全不一样的东西。

**第一类:内存缓存(关掉 tab 就没的那种)**

我用的就是普通的 JS `Map` 和 `Set`,挂在模块 scope 上 —— **不是 React state**,因为缓存要跨组件、跨页面共享,写到组件 state 里换个页面就全没了。一共四张表,各自分工:

| 内存表 | 存的是什么 | 解决什么问题 |
|---|---|---|
| `ttsUrlCache` | 内容哈希 → 音频 URL | 主缓存,45 分钟内同一段音频不再打接口 |
| `pendingTtsUrlRequests` | 内容哈希 → 正在跑的 Promise | 并发去重,场景页和 chunks 页同时要同一段音频,只发一个请求 |
| `preloadedAudioUrls` | 已经 preload 过的 URL 集合 | 避免重复 `<audio preload>` 把同一个文件下两遍 |
| `persistentAudioObjectUrls` | 内容哈希 → `blob:` URL | 从本地音频缓存取出的文件转成 blob URL 之后留着复用 |

为什么内存层这么快:这些 `Map` 就是 JS 堆内存,读写是纳秒级,不走 IO,也不走主线程之外的事件循环。代价就是 tab 一关全没,且每个 tab 都各有一份(不跨 tab 共享)。

**第二类:浏览器本地缓存(关 tab、刷新、断网都还在的那种)**

我用的是 **Cache Storage API**(`window.caches.open("tts-audio-v2")`),**不是 localStorage、不是 IndexedDB、不是 cookies**。这是浏览器自带的一套持久化 key-value 存储,跟 Service Worker / PWA 离线缓存底层是同一套东西。存到浏览器自己管理的磁盘空间里,容量很大(几十 GB 配额量级),关 tab、刷新、断网都还在。

我自己叠了一层 LRU 管理:单个 cache 上限 120 个文件 / 24 MB,超出就按写入顺序淘汰最旧的;淘汰的同时把对应的内存表也清掉、把 `blob:` URL `revokeObjectURL`,保持两层一致、避免内存泄漏。

### 7.2 几个会被追问的边界问题

**Q:为什么不用 localStorage?**

四个原因,任一个都足以否决:

1. 只能存字符串,二进制音频要 base64 编码,体积膨胀 33%
2. 同步 API,读写直接卡主线程
3. 容量上限大概 5MB,一个音频几 MB,存不了几条
4. 没有过期、没有 LRU,要自己写一套淘汰

**Q:为什么不用 IndexedDB?**

IndexedDB 也能存 Blob、容量也够,但:

- API 重得多(事务、版本号、游标),代码量上去
- 我这个场景就是 key-value 拿文件,不需要查询能力
- Cache Storage 天生为 Request/Response 设计,跟 fetch 无缝衔接,写起来很短

如果将来要做更复杂的本地学习数据(比如带索引的离线题库),那时候才考虑 IndexedDB。

**Q:为什么不用浏览器原生 HTTP 缓存(Cache-Control)?**

- HTTP 缓存的 key 是请求 URL,而 Supabase 临时链接每次签发都不一样,根本命不中
- 我的缓存 key 是「内容哈希」,URL 怎么变都不影响
- HTTP 缓存的过期策略由服务端响应头决定,我前端不可控

**Q:`blob:` URL 是怎么回事?要不要 revoke?**

`URL.createObjectURL(blob)` 把内存里的二进制 Blob 包装成一个临时 URL(形如 `blob:https://xxx/uuid`),让 `<audio src=...>` 不走网络就能直接播。这个 URL 跟 Blob 绑定,浏览器一直持有对应内存,**不主动 `URL.revokeObjectURL` 就会内存泄漏**。我在 LRU 淘汰那张 `persistentAudioObjectUrls` 表的时候顺手 revoke。

**Q:多个 tab 打开同一个场景会怎么样?**

- 内存层不共享(每个 tab 一个 JS 运行时,各自一份 `Map`),所以两个 tab 第一次都会各自打一次接口
- 但 Cache Storage 是同 origin 共享的,所以**第一个 tab 命中并写入之后,第二个 tab 下次播放就能直接从 Cache Storage 命中,不再打接口**
- 如果将来要做 tab 间也去重,可以接 `BroadcastChannel` 或者用 Service Worker 集中拦截,但目前不值当

**Q:Cache Storage 配额满了浏览器会怎么处理?**

浏览器有自己的全局存储配额(基于磁盘剩余空间动态算的)。我自己的 24MB 上限远小于这个,正常不会触发浏览器层面驱逐;万一真触发了,浏览器会按 LRU 把整个站的存储清掉(包括我的 Cache Storage),下次再播就是冷启动重新走链路,不会出错,只是慢一次。

### 7.3 服务端音频是怎么生成的(为什么这么选)

很多面试官会顺着前端 cache 追问"那音频本身是怎么生成的、为什么用这家",所以提前准备一下技术选型的理由。

**TTS 引擎用的是微软 Edge TTS**(`msedge-tts` npm 包)。它的本质是把浏览器 Edge 内置的「朗读网页」功能反向出来,通过 WebSocket 调用微软的免费 TTS 服务,返回的是高质量的 Neural Voice 音频。

为什么选它,而不是其他几个选项:

| 候选方案 | 不选的原因 |
|---|---|
| OpenAI TTS / ElevenLabs | 按字符收费,英语学习场景文本量大,跑起来成本扛不住 |
| Google Cloud TTS / Azure 官方接口 | 都要付费 + 配额管理 + 计费监控,工程化成本高 |
| 浏览器 `SpeechSynthesis` API | 音色取决于用户设备/系统,Mac/Windows/iOS/Android 出来声音都不一样,无法预生成、无法缓存、无法分发 |
| 自己跑开源 TTS(Bark / Coqui 等) | 要自己撑 GPU 推理,完全不划算 |

Edge TTS 在我这个场景里基本免费 + 音质够好(微软 Neural 系列)+ 支持 SSML 控制语速,是性价比最高的方案。代价是接口不稳定(详见 §14.15 那次进程级崩溃事故),所以服务端要做白名单兜底。

**缓存 key 用的是内容哈希,不是用户 ID 或者业务 ID**。这是音频缓存能跨用户复用的关键:

- 不同用户、不同设备,只要播的是同一段英文文本,都会算出同一个 key
- 所以一段热门场景的音频,全站只生成一次,所有人复用
- 如果用业务 ID(比如「场景 42 的第 3 句」)做 key,场景内容微调一下就要重新生成全部音频

实现见 [src/lib/shared/tts.ts](src/lib/shared/tts.ts) 的 `buildSentenceAudioKey` 和 `buildChunkAudioKey`:把文本归一化(去标点、小写、去多余空白)之后算一个稳定 hash,挂上一些可读前缀(便于存储里翻找)。

**音频文件存到 Supabase Storage**(不是放服务器本地、不是自建 CDN)。理由:

- 本地磁盘:多机器部署的时候音频文件不共享,要么做 NFS 要么每台都重生成
- 自建 CDN:运维成本太高,流量也不大
- Supabase Storage:免费额度够用,跟 Auth/数据库同源,生成签名链接就能直接给前端,**省掉一整套对象存储 + 鉴权 + CDN 的搭建**

**给前端的不是公开链接,是带签名的临时下载链接**(60 分钟有效)。理由:

- 防止音频被外站直接 hotlink 盗用
- 链接里带签名,改一个字节都拒绝
- 链接过期了不影响已经下到本地的音频(因为前端已经持久化到 Cache Storage,见 §7.1)

### 7.4 服务端这块也会被追问的边界问题

**Q:Edge TTS 是免费的?会不会被微软掐了?**

是的,有这个风险。我做了两层兜底:服务端调用失败时降级到浏览器原生 `SpeechSynthesis` 播放(声音差但能用),错误率聚合到 Sentry 监控趋势。如果将来真的不可用,可以切到付费 TTS,因为我把 TTS 调用封装成了独立 service([src/lib/server/tts/service.ts](src/lib/server/tts/service.ts)),换引擎只改这一处。

**Q:同一段文本同时被 100 个用户播,会生成 100 次吗?**

不会。服务端有一层「pending 请求合并」(类似前端 `pendingTtsUrlRequests` 那个 Map,见 [src/lib/server/tts/service.ts](src/lib/server/tts/service.ts) 的 `pendingSignedUrlRequests`):同一个 cache key 正在生成时,后来的请求都共享同一个 Promise,只生成一次。

**Q:为什么 SSML 这块没讲?**

SSML 是 TTS 行业标准,可以控制语速、停顿、强调等。我目前只用到语速控制(`<prosody rate="0.8">`)做慢速朗读,没用到更复杂的 SSML 标签。提一句让对方知道我懂这个概念就行,不展开。

**Q:音频文件大概多大?生成要多久?**

一句话 mp3 大概 20-80KB,生成耗时 1-3 秒(取决于文本长度和 Edge TTS 响应)。整段场景音频(几十句拼起来)大概 500KB - 2MB,生成 5-10 秒。所以**冷启动那次必然慢**,这也是为什么前端要做这么多层预热(见 §9)。

### 7.5 响应里的 cached/source 字段(接口能直接看到的证据)

讲完缓存层和生成路径之后,有一个**别人能在 Network 面板直接验证你说真话**的细节值得拎出来讲。`/api/tts` 的响应体除了 `url`,还带两个观测字段:

```typescript
// src/app/api/tts/route.ts 返回的形状
{ url: "...", cached: boolean, source: "storage-hit" | "fresh-upload" | "inline-fallback" }
```

| source | cached | 含义 | 触发场景 |
|---|---|---|---|
| `storage-hit` | `true` | Supabase Storage 已经有这段音频,直接返回 60min 签名 URL | **别的用户**之前生成过同样 hash 的内容 |
| `fresh-upload` | `false` | 这次是 msedge-tts 现场生成 + 上传 Storage 成功 | 内容是新的 |
| `inline-fallback` | `false` | 生成成功但 Storage 上传失败,把 mp3 直接 base64 内联到 data URL 返回 | Supabase Storage 临时故障 |

(代码在 [src/lib/server/tts/service.ts:392-427](src/lib/server/tts/service.ts#L392-L427))

**为什么这三个字段值得拎出来讲:**

1. **`cached: true` 是「跨用户共享」那一层的实锤**。§7.1 讲的几层缓存全是「本浏览器」(每个用户自己),`cached: true` 是说**别人已经生成过这段了,你直接用**。在「同一个 chunk 被 N 个用户学」的场景下,TTS 只花了 1 次成本。同 hash → 同 storage path → 同文件,这条链路 §7.3 已经讲过,`cached` 是它的可观测证据。

2. **`inline-fallback` 是一个有面子的降级故事**。Supabase Storage 上传失败的时候,**不抛错让用户看到「播放失败」**,而是把刚生成的 mp3 直接 base64 编码成 `data:audio/mp3;base64,...` 内联到 URL 里返回。用户感知不到 Storage 挂了,音频照常播。代价是这一次的音频跨用户不可复用(下次还得现场生成),但是「先让用户能听到」优先于「先存起来」。

3. **F12 → Network → `/api/tts` 响应体能直接看到 `cached: true`**。面试官想验证你说的真假很容易,这种「自带证据的字段」可信度比口头描述高很多。

### 7.6 cached/source 也会被追问的边界问题

**Q:客户端拿到 `cached: true` 之后做什么?**

什么都不做。客户端只用 `data.url`,不用 `data.cached` 做控制流(见 [src/lib/utils/tts-api.ts:860](src/lib/utils/tts-api.ts#L860) 附近)。原因是**服务端 cached 是「跨用户共享」的信号,客户端有自己独立的多层缓存判断**(见 §7.1)。两个维度分开:服务端关心「这段音频之前生成过没」,客户端关心「这个浏览器之前播过没」。混在一起反而难维护。`cached` 字段保留下来主要是埋点/调试用 —— 排查问题时直接看响应就能判断「是不是命中了 Storage」。

**Q:为什么 source 里没有 `"local-cache-hit"` 之类的客户端命中?**

不需要。客户端命中根本不会发出 `/api/tts` 请求(memory cache、persistent cache 命中都在 fetch 之前就 return 了)。所以 `source` 天然只描述「请求真的到了服务端之后」的三种结果,语义清晰。

**Q:`inline-fallback` 的 base64 URL 会不会爆内存?**

一句话 mp3 大概 20-80KB,base64 之后膨胀 1.33 倍,30-100KB 完全可接受。整段场景音频(500KB-2MB)走 inline-fallback 会比较大,但这种情况只在 Storage 真挂掉时才发生,优先保证「能播」就够了。

### 7.7 AI 文本结果也走同一套内容寻址(把缓存做成方法论)

「内容相同就共享」这套思路不只 TTS 用,**AI 文本生成也走同一套**。代码在 [src/lib/server/ai-cache/service.ts:26-73](src/lib/server/ai-cache/service.ts#L26-L73):

```typescript
export const buildStableCacheKey = (scope, payload) =>
  `${scope}:${sha256(stableStringify(payload))}`;
```

关键设计:

- **`stableStringify` 自己实现的排序序列化** —— 不用 `JSON.stringify`,因为 JSON.stringify 不保证对象 key 顺序。`{a:1,b:2}` 和 `{b:2,a:1}` 在标准 stringify 下可能产生不同字符串,导致同输入算出不同 hash → 缓存命中率掉零。`stableStringify` 强制对 key 排序,**同输入永远同 hash**。
- **scope 前缀做命名空间** —— `scene-parse:xxx`、`scene-variants:xxx`、`scene-generate:xxx`,三类业务不会撞 key。
- **`scene-parse` / `scene-variants` 的 key 里完全不带 userId** —— 全用户共享。一段英文文本 N 个用户用 AI 解析,只生成 1 次。`scene-generate` 带 userId 是因为它要做个性化(基于用户已经学过的内容来生成新场景),个性化输出不能跨用户复用。
- **观察工具有现成的** —— 后台 `/admin/cache` 能直接看 ai_cache 表的命中情况、最新写入时间、行数。

**为什么值得讲:** 一句话把它和 §7.3 的 TTS 跨用户共享挂钩,从「我做了一个缓存」升级成「我把这个内容寻址的思路抽象成方法论,在两条不同业务线上都生效」。面试里这种「同一个抽象 N 处复用」的故事很值钱。

**Q:为什么不直接用 stringify + key.sort()?**

stableStringify 是**递归**的:嵌套对象、嵌套数组每一层都排序。`JSON.stringify(obj, Object.keys(obj).sort())` 只能排第一层,嵌套层级仍然按插入顺序输出。我项目里 AI prompt 的参数对象有 3-4 层嵌套,必须递归排序才能稳。

**Q:hash 冲突怎么办?**

sha256 输出 256 bit,理论冲突概率小到可以忽略(生日攻击大概要 2^128 次)。业务侧不做额外防御,出问题再说。

### 7.8 Signed URL 缓存的 Redis + Memory 双后端(生产学到的)

§7.3 提了「Storage 签名 URL 60min 有效,我们做了缓存避免反复签」。但**怎么缓存**这件事在生产踩过坑,值得展开。

代码:[src/lib/server/tts/signed-url-cache.ts](src/lib/server/tts/signed-url-cache.ts)(文件头注释写得很完整,可以照念)

**最早只用进程内 Map,生产命中率掉了一截:**

线上跑 PM2 cluster 模式(多 worker 利用多核),每个 worker 自己的 Map 互不相通。同一个 storagePath 用户第一次落在 worker A 上,缓存只进了 A 的 Map;同一用户下一次请求路由到 worker B,Map miss,又得去 Supabase 签一次。**命中率被 worker 数量稀释**,默认 4 个 worker → 命中率最多 25%。

**改成 Upstash Redis 共享 + Memory 兜底:**

```typescript
// 写的时候 Redis + Memory 都写一份
backend.fallback.set(storagePath, { url, expiresAt });
try { await callUpstash(['SET', key, url, 'EX', ttl]); }
catch { /* 静默吞,Memory 已经写了 */ }
```

- **写双份**:Redis 抖动时本进程内 Map 还能继续命中,不会因为 Redis 临时不可用导致大量重复签名
- **读优先 Redis,失败退化到 Memory** —— 跨 worker 命中由 Redis 提供,worker 内命中由 Memory 提供
- **零配置开发**:本地没有 `UPSTASH_REDIS_REST_URL` 环境变量时自动只用 Memory,不强求开发者搭 Redis

这跟 §7.5 的 `inline-fallback` 是同一种设计哲学 —— **第三方存储抖动时不让用户感知**。两个点凑起来就是一个「存储链路降级」的完整故事:

| 抖动来源 | 降级策略 |
|---|---|
| Supabase Storage 上传失败 | inline-fallback:把 mp3 直接 base64 内联返回 |
| Upstash Redis 不可用 | 退化到进程内 Map,命中率打折但服务不中断 |

**Q:为什么不用 Supabase 自己的 KV?**

Supabase 没有原生 KV/Redis 服务。要么自建 Redis,要么走第三方。Upstash 免费额度够用 + REST API 不需要长连接(serverless 友好),最适合 PM2 worker 这种短生命周期场景。

**Q:Memory 兜底的内存会不会涨爆?**

每个 entry 是 `{ url: 200 字符左右, expiresAt: 8 字节 }`,1000 个 entry 大概 200KB。签名 URL 1h TTL,过期会被 `readMemoryEntry` 主动清理。即使被穿透,内存压力也极小。

## 8. 技术亮点四:场景循环播放(后台播放优化)

这是最近做的一块音频后台播放优化,面试时可以讲,但要讲准。

它**不是**"拿 ffmpeg 把多个音频文件拼起来",也**不是**"前端一段一段请求然后切歌"。真正的做法是**在音频分段级别打包**:

1. 从用户已经学过的场景里,挑出熟悉度够高的候选
2. 按今天稳定的顺序选一小批(保证预加载和点击播放命中同一份资源)
3. 读取每个场景的详情
4. 把每个场景拆成统一格式的音频片段
5. 把多个场景的片段合并成一个"打包数据"
6. 复用现有的"整场景音频生成"通道,生成一个完整的合并音频
7. 前端用一个 audio 标签开循环播放

为什么这么做:

- 手机端切到后台或者锁屏后,页面 JavaScript 不一定能稳定执行"切下一首"
- 如果一段一段播,后台播放强依赖网络和 JS 执行时机,经常断
- 生成一个整体音频后,浏览器原生播放器更容易持续播下去
- 同一天稳定的"打包"能提高缓存命中率,不用每次点击都重新生成

代价也要讲清楚:

- 优点:后台播放稳定、缓存命中稳定、请求次数少
- 缺点:播放过程中不实时切换"当前场景名称";候选变化后需要重新准备
- 它不是完整的离线音频系统,也不是"边播边在末尾热插新内容"

一句话总结:

> 我没有做文件级别的音频拼接,而是把多个场景先转成统一格式的音频片段,再复用现有 TTS 通道生成一个能循环播放的整体音频,用稳定性换掉后台 JavaScript 切歌的不确定性。

### 8.1 为什么不能让前端 JS 一段段切歌(展开后台播放的真正难点)

这是这一块最值得讲的部分,也是前端面试官最容易追的角度。

**痛点 1:后台 tab 的 setTimeout / setInterval 会被节流**

现代浏览器为了省电,**对后台 tab 的定时器最低降到 1 Hz 甚至更慢**。如果我用 `audio.onended` 触发"下一段开始",这个回调在前台没问题,切到后台后回调延迟会被拉到秒级 —— 中间就会有明显空档。

**痛点 2:移动端锁屏后 JS 完全暂停**

这个比节流更狠。iOS Safari 锁屏后,整个 JavaScript runtime 会被冻结。无论你怎么注册回调、怎么定时,都不会执行。**但 HTML5 `<audio>` 是浏览器主线程之外的原生组件,JS 冻死了它也能继续播下去**。所以唯一稳定的后台播放,必须是「交给浏览器原生播放器播一个连续音频」,不能让 JS 参与切歌。

**痛点 3:autoplay policy**

浏览器(尤其 iOS Safari)禁止页面在没有用户交互的情况下自动开始播放。这意味着我不能"用户切到后台之后悄悄起一个新 `<audio>`"。只能在用户点击「开始播放」那一刻就把所有要播的内容串成一个连续音频,后续靠浏览器自己循环。

**这就是为什么我选「整段打包 + `audio.loop = true`」而不是「N 段音频 + JS 控制切换」**:

- JS 后台靠不住 → 让浏览器原生组件接管
- 单 `<audio>` 元素 + `loop` 属性,浏览器自己负责无缝循环,不需要任何 JS 介入
- 哪怕用户锁屏 10 分钟回来,音频还在播放,因为浏览器播放器跟 JS 不在一个线程

**为什么不用 Web Audio API?**

Web Audio API 能力更强(可以做 EQ、淡入淡出、变速变调),但:

- 它依赖 `AudioContext`,后台同样会被挂起(`suspend` 状态)
- 拼接多段音频需要自己做时间轴调度,代码复杂度高
- 我这里只需要"循环播放一个文件",`<audio loop>` 已经够用
- Web Audio API 在 iOS Safari 锁屏后表现比 HTML5 audio 更不稳定

**为什么不用 Service Worker?**

Service Worker 可以拦截网络请求、做离线缓存,但**它不能直接控制音频播放**。Service Worker 适合做"音频文件持久化"(我用 Cache Storage 已经实现等效效果),不适合做"调度音频片段连续播放"。

### 8.2 后台播放也会被追问的边界问题

**Q:那用户锁屏的时候能看到「正在播放 XX」吗?能从锁屏控制吗?**

可以,用 **Media Session API**(`navigator.mediaSession`)。设置 `metadata`(标题/封面)和 `setActionHandler`(播放/暂停/上一首),浏览器就会把这些信息推到系统播放控制中心(iOS 锁屏、Android 通知栏、Mac 触控栏)。目前项目还没接,是已知的可以补的能力。

**Q:整段打包了,那用户想从第 5 段开始听怎么办?**

目前不支持。代价已经在文档里讲了 —— 用稳定性换掉了「实时切换当前场景」的能力。如果要做章节跳转,可以记录每段在合并音频里的时间偏移,用 `audio.currentTime = offset` 跳转,但代码复杂度上来,目前没做。

**Q:那如果合并音频生成失败/超时呢?**

整段 fallback 到逐段播放(浏览器在前台时仍能跑),并把失败原因聚合到 Sentry。同一段会进入冷却期(45 秒),避免反复打失败的请求。详见 [src/lib/utils/tts-api.ts](src/lib/utils/tts-api.ts) 的 `sceneFullFailureCooldowns`。

**Q:iOS Safari 跟 Android Chrome 表现一致吗?**

不完全一致,但 `<audio loop>` 这条路径在两边都最稳定。iOS Safari 更严格(autoplay policy、AudioContext 限制),所以我按 iOS 的约束来设计,Android 自然能跑。

**Q:能不能用 Web Worker 在后台调度?**

不行。Web Worker 跟主线程一样会被后台节流,而且 Worker 也不能访问 `<audio>` 元素(DOM 在主线程)。唯一能在锁屏后继续运行的是浏览器原生播放器本身。

### 8.3 失败冷却用了两套(场景级 vs 单元级,刻意拆开)

§8.2 提了「整段场景音频失败会进入 45s 冷却」,§14.6 里也提了一句「失败有冷却阶梯(5s → 60s → 5min → 30min)」。**这两个是不同的两套机制**,刻意拆开,值得展开讲:

| 维度 | 整段场景(sceneFull) | 单句/段(sentence/chunk) |
|---|---|---|
| 触发对象 | 一整场音频的预加载或播放 | 单个 sentence/chunk 的 TTS 请求 |
| 冷却长度 | 固定 45 秒 | 阶梯式:5s → 60s → 5min → 30min |
| 重置条件 | 45s 过完即结束 | **静默 30 分钟后失败计数清零**,下次失败重新从 5s 起算 |
| 设计意图 | 整场失败是大事,先快速止血,等浏览器/网络状态变化再说 | 单句失败可能只是瞬时抖动,头几次快速重试;**反复失败的疑难内容才进入长期回避** |

代码:[src/lib/utils/tts-api.ts:151-209](src/lib/utils/tts-api.ts#L151-L209) 的 `audioRetryCooldownLadderMs` + `markAudioRetryFailure` + `audioRetryRecordIdleResetMs`。

**为什么这样拆:**

- 整场失败是「这一次播 30 句拼起来的音频不行」,**重试代价高**(要重新拉一大包),所以冷却长一点(45s)防止反复重试拖死。
- 单句失败是「这一句话的 TTS 出错」,**重试代价低**(就一句话),所以前 2 次快速重试(5s、60s);但如果反复失败,说明这句内容本身有问题(比如包含 Edge TTS 处理不了的特殊字符),**继续重试是浪费**,所以阶梯到 5min、30min。
- 「静默 30min 后重置计数」是为了**给修复留余地** —— 比如开发改了 prompt、清了脏数据,30min 没碰过这句话,下次失败重新从 5s 起算,而不是被永久惩罚。

**可观测性:**

每次进入冷却都会发 `tts_request_cooling_down` 事件(见 [src/lib/utils/client-events.ts](src/lib/utils/client-events.ts)),带 `failureCount`、`cooldownMs`、`remainingMs` 三个维度。管理后台能看到「哪个 cacheKey 反复失败 N 次,当前还要冷却多少毫秒」,直接定位问题内容。

**Q:为什么不用统一的「指数退避」,非要分两套?**

统一的话要么对整场太敏感(45s 太短反复打)要么对单句太迟钝(单句一次失败就等 5min 体验很差)。两类失败的「重试代价」和「常态恢复时间」不一样,分两套调起来才能各自最优。

**Q:30 分钟这个数字怎么定的?**

经验值,不是算出来的。**够长**:确保大部分瞬时抖动(微软 TTS 偶发延迟、网络微断)都过去了;**不太长**:用户回来重新学的时候(典型间隔 1-2 天)早已重置。如果以后做用户分群,可以再细调。

## 9. 技术亮点五:缓存和预加载策略

这个项目里**缓存不是单点优化,而是围绕学习体验做了多层兜底**。

主要包括:

- 场景列表的缓存
- 场景详情的缓存
- 复习页的缓存
- 学习主页的缓存
- 表达列表的缓存
- TTS 音频网址的缓存
- 浏览器本地的音频文件缓存
- 空闲时间偷偷预加载

典型策略可以一句话概括:**先用缓存把页面打开,再去后台拉新数据;重资源放到空闲时预加载;弱网或者用户开了省流量模式时,自动减少预加载**。

面试可以强调:这不是为了炫技,而是学习产品的访问很碎片化 —— 用户会反复进入同一个场景、看同一个表达、播同一段音频。所以缓存收益很直接。

### 9.1 多层缓存的具体打法(前端面试展开)

「先用缓存把页面打开」这一行的背后,是一套**业内叫 stale-while-revalidate(简称 SWR)**的常用模式。展开讲就是:

1. 用户进页面时,**立刻**用本地缓存把 UI 渲染出来 —— 看到旧数据没关系,先有响应
2. **同时**在后台发请求拉新数据
3. 新数据回来后,如果跟缓存不一样,静默更新 UI;一样就不动
4. **零阻塞、零白屏、零 loading 转圈**(这是关键卖点)

我没有用社区的 SWR 库或者 React Query,而是自己写了一套统一的缓存层在 [src/lib/cache/](src/lib/cache/),理由是:

- 这些库默认是 in-memory,但我需要跨页面、跨刷新都稳定的多层缓存
- 我的不同数据类型有不同的失效策略(场景列表 30 分钟,表达列表实时跟随用户操作),用库需要按每个查询配,反而麻烦
- 项目里 cache key 跟 TTS / 学习状态这些复杂资产强耦合,自己维护更直观

**缓存层次按"读的成本"分**,典型路径是这样的:

```
用户进 /scenes 列表页
 ↓
1. 内存(JS 模块级 Map)──命中→ 立刻渲染
 ↓
2. sessionStorage(同 tab 内持久)──命中→ 立刻渲染 + 回填内存
 ↓
3. 服务端响应(实际网络请求)→ 渲染 + 回填两层缓存
```

**预加载的两类时机**:

- **「下一步可能要看」型**:用户在场景详情页时,我会提前 fetch 这个场景里前 2 个 chunk 的音频(`prefetchChunkAudio`),这样他真点击播放时基本是命中状态
- **「空闲时间偷偷准备」型**:用户在主页待着没操作时,后台用 `requestIdleCallback` 跑 idle warmup,把后面 2-3 段场景的音频也预热到 Cache Storage —— **关键是利用浏览器告诉我「现在没事干」的空闲帧,不抢用户交互的优先级**

**弱网/省流量模式自动跳过预热**:

我在 [src/lib/utils/resource-actions.ts](src/lib/utils/resource-actions.ts) 检测了 `navigator.connection.saveData` 和 `effectiveType`(2g / slow-2g),如果用户开了数据节省模式或者当前是 2G 网络,**所有预热都自动跳过,只保留按需加载**。理由很简单:在 2G 下预下载几 MB 音频是欺负用户。

### 9.2 缓存策略会被追问的边界问题

**Q:用户改了一个表达,缓存怎么失效?**

按数据生命周期分两类处理:

- **用户自己写的数据**(表达、学习进度):写操作成功后立即清掉本地对应 cache key,下次读必走网络
- **平台内容**(场景列表、场景详情):TTL 过期(几分钟到半小时不等),不主动失效。代价是用户偶尔会看到稍旧的数据,但场景内容更新频率很低,影响小

**Q:为什么不用 React Query / SWR?**

不是不能用,是几个具体原因让自建更合适:

- 我的缓存要跨 tab 持久(Cache Storage 那层),React Query 默认只管内存
- TTS 音频缓存跟通用数据缓存机制完全不同,塞到 React Query 里反而别扭
- 项目用 Next.js App Router + Server Components,服务端预渲染本身就消化了一部分「初次加载」的需求,客户端缓存层简单点更好

不过单纯的「列表 + 详情」型数据,如果将来要新接一块业务,会考虑直接上 React Query,没必要重复造。

**Q:saveData 怎么检测?哪些浏览器支持?**

`navigator.connection.saveData`(布尔)+ `navigator.connection.effectiveType`(`"2g" / "slow-2g" / "3g" / "4g"`),具体在 [src/lib/utils/resource-actions.ts](src/lib/utils/resource-actions.ts) 的 `isLowBandwidthEnvironment`。

支持情况:Chromium 系全支持(包括移动端 Chrome、Edge、国内大部分浏览器内核),Safari 和 Firefox 不支持 —— 那两家拿不到 `connection` 就当成「网络正常」处理,不影响主流程,只是省流量优化失效。

**Q:`requestIdleCallback` 在 Safari 上不支持怎么办?**

Safari 一直没实现 `requestIdleCallback`。我做了 polyfill 兜底(用 `setTimeout(fn, 1)` 模拟),功能上不损失,只是空闲调度的精度差一些。详见 [src/lib/utils/resource-actions.ts](src/lib/utils/resource-actions.ts) 的 `getRequestIdleCallback`。

**Q:页面缓存怎么避免越积越多?**

每一层都有自己的容量上限和 TTL:

- 内存层:`Map` 大小不主动限制,但每个独立 cache 都有 TTL,过期就清(45 分钟左右)
- sessionStorage:只在当前 tab 生命周期内存活,关 tab 自动清掉
- Cache Storage(音频):LRU,120 文件 / 24MB 上限(见 §7.1)

## 10. 技术亮点六:后端接口治理和权限边界

项目已经过了"先把功能跑通"的阶段,补了不少工程层面的能力。我按"做了什么 + 解决什么"来讲:

| 做的事 | 解决什么问题 |
|---|---|
| **每个请求带一个唯一编号**(请求编号) | 出问题时方便把前端 → 接口 → 业务层日志串起来 |
| **所有接口出错都返回同一种结构**(错误码 + 详情 + 请求编号) | 前端能稳定解析错误,不用每个接口都猜 |
| **高成本接口限流**(TTS / AI 生成 / 批量导入) | 防止单用户或单 IP 把贵接口打爆 |
| **每日额度 + 后台紧急开关** | AI 接口异常成本能在调用之前止损;出大事可一键关掉某个功能 |
| **注册入口安全防护**(注册模式开关 + 邀请码 + 邮箱验证码 + 邮箱链接验证 + 注册 IP 频率控制) | 防止爬虫批量注册薅 AI 接口 |
| **管理员后台** | 可以降级用户状态、开关高成本能力、管邀请码 |
| **重复请求去重** | 用户连点两次按钮只产生一条数据 |
| **请求来源校验** | 防止外站伪造请求改我们的数据 |
| **数据库行级权限** | 用户数据访问从"应用层判断"逐步收回数据库自带的最小权限模型 |
| **错误聚合到 Sentry**(错误监控平台) | 生产环境出 5xx 自动上报,带请求编号;事故响应从"等用户截图"降到分钟级 |
| **内容安全策略(CSP)上报模式** | 防止 XSS 等注入攻击;违规先收集观察,一键切到强制执行 |
| **启动期配置自检** | 服务启动就在日志里打印一行 10 个关键配置项的状态;部署后看一眼就能判断配置有没有漂移 |
| **自动检查工作流**(GitHub Actions) | 每次提交时跑一整套护栏检查(代码风格、文字乱码、单元测试、规范) |
| **接口错误响应一致性收口** | 全量接口的错误返回统一为"错误码 + 详情 + 请求编号";之前几处用错状态码(比如登录失效返回 500 让人以为是服务器挂了)都修正 |

可以这样讲:

> 早期项目最重要的是把闭环跑通。现在已经开始补限流、去重、追踪、权限边界、错误聚合、安全策略、启动自检和自动检查这些工程化能力,让高成本 AI 接口、用户数据、小范围开放和事故响应都更可控。

> **30 秒电梯版**:"我把接口治理按风险拆三类:公开注册入口要防爬虫,补了邀请码 + 邮箱验证 + IP 频率控制;高成本 AI 接口要防滥用,补了限流 + 每日额度 + 紧急开关;关键写操作要防重复,补了去重保护。所有请求带唯一编号,出错统一格式,生产 5xx 自动上报。"

## 12. 技术亮点八:匿名试用 + 配额隔离 + 防绕过(给"分享链接"开的灰度口)

### 12.0 一句话讲清要解决什么

> 注册门槛挡掉了大量"看一眼就想走"的人,但 AI 接口又是真金白银,直接开放又怕被脚本薅成本。我把匿名试用做成一个独立的灰度口,只在 `/share/scene/[slug]` 这类分享链接上打开;身份、配额、限流、表权限四层全部和已登录用户隔离,不污染主链路。

### 12.1 业务约束:为什么不直接"全站开放未登录"

- 全站开放 = 任何爬虫都能调 AI/TTS,成本不可控;同时既有"复习 / 进度 / 表达库"页面对未登录没有意义,看到只会困惑。
- 分享链接是真实增长场景:用户发场景给朋友,朋友点进来如果遇到登录墙,转化漏到几乎为零。
- 折中:**只在分享路径上灰度,主入口完全不动**。出问题一个开关切回 false 就能止血。

### 12.2 身份层:四道防线套娃

1. **localStorage 里的 `abridge:anon_id`**(UUID v4)+ **`X-Anonymous-Id` 请求头透传** — 不用 Cookie 是为了让分享链接跨域场景下 SPA 调用稳定;后端只要看到这个头就知道访客身份。脚本伪造头几乎免成本,所以**不靠这个做安全**,只用它做"同一访客"识别。
2. **请求侧 `ip_hash = SHA256(ip + daily_salt)`** — 每日 salt 轮换,不存明文 IP,做 IP 维度限流但保留隐私最小化。这才是真正的防绕过主力(改 anon_id 没用,IP 还在)。
3. **数据库 `anonymous_sessions`** — 同 IP 当日 session 数有上限(默认 5),触顶返 `ANON_IP_RATE_LIMITED(429)`。
4. **搜索引擎爬虫识别** — UA 命中 Googlebot/Bingbot 时不进配额扣减分支,只渲染最小可索引引导态,避免爬虫把全站匿名 AI 池薅光。

### 12.3 配额层:两类匿名 capability 走两套 quota,严格隔离

- Redis 命名空间严格隔离:已登录的 `quota:{userId}:{cap}:{date}` 完全不动,改一边不影响另一边。
- **AI 表达解释**(`explain_selection`,会调付费上游)走完整四层防御 + 全站日预算:
  - key:`anon:quota:global:explain_selection:{date}` + `anon:quota:session:{anonId}:explain_selection:{date}`
  - 默认:全站 200 次/天(对应 ≈ $0.14/天),单 anon 3 次/天,IP 滑窗 30/min
- **TTS 预生成播放**(`tts_play`,只读 Storage 零边际成本)走独立轻量 quota,**没有全站池**:
  - 只有 session 一层 + 共享同一个 IP 滑窗 scope
  - 默认:单 anon 30 次/天,IP 滑窗复用 explain 同一个限速桶
  - 设计取舍:预生成播放只是签发一次 signed URL,不调 MsEdgeTTS,全站池防失控的需求不存在;留 IP 滑窗就够防 DDoS 风格爆量
- 单价以函数内常量(`c_explain_unit_usd = 0.0009`,`c_tts_unit_usd = 0.0003`)推算每日估算成本,直接写进 `daily_anon_cost_report` 看板。

### 12.4 表权限层:Supabase RLS 显式 deny-by-default

- 公共内容表(`scenes` / `scene_variants` / `chunks` / `phrases`)各自加 anon SELECT 策略,过滤条件用 `is_public = true` 或 `is_builtin = true`。
- **所有用户态表(profiles / user_chunks / user_phrases / ...17 张)不加 anon 策略**,RLS 默认 deny 兜住,匿名用户即便伪造请求也读不到任何账户数据。
- `anonymous_sessions` / `anonymous_funnel_events` / `daily_anon_cost_report` 自己也 deny anon SELECT,只服务端 service role 能写读 → 隐私最小化 + 防止匿名用户互相观察。

### 12.5 前端引导:三层注册转化漏斗

- **L1 顶栏配额条**:常驻显示"AI 表达解释 剩 2/3 次",sessionRemaining ≤ 1 切红 + ⚠️ 前缀,把配额逼近做成可视化压力。
- **L2 内联卡片**:学完一个场景后弹"刚刚学的这个场景里有 N 个表达,注册保存",sessionStorage 持久化 dismiss,同一会话不再重复弹。
- **L3 阻断弹窗**:三种触发(功能禁用 / AI 解释配额耗尽 / TTS 配额耗尽),复用同一文案模板,统一"立即注册解锁 / 稍后"两按钮。

### 12.6 漏斗埋点 + 成本看板

- 8 个事件枚举(`anon_session_created` → `anon_first_scene_viewed` → ... → `anon_registered`),`recordAnonymousFunnelEvent` 强校验事件名,未知名直接抛错防止漏斗失真。
- `anon_registered` 事件 payload 携带 `from_anon_id`,匿名 → 注册的转化能精确归因。
- 日 cron 调 `aggregate_daily_anon_cost_report()`,把 8 个事件 + 单价常量聚合成 `total_sessions / ai_explain_calls / tts_play_count / estimated_cost_usd / conversion_rate / cost_per_conversion`,看板和告警都吃这张表。

### 12.7 紧急止血开关

- `ALLOW_ANONYMOUS_TRIAL` 总开关,异常情况下切回 false → 所有 `/share/...` 路径退回 `/login`,既有登录用户主链路零影响。
- 三条飞书告警:同 IP 当日 session > 10、全站匿名 AI 池 18:00 UTC 前 > 80%、`anon_quota_blocked / anon_session_created > 60%`,各对应一个明确的预案动作。

### 12.8 实施现实:开发完后自评审,3 个补丁 commit 闭环

灰度功能首版 (`145686a`) 跑完后我做了一次全局自评审,发现 7 个让"灰度核心目标失效"的问题。3 个 commit 闭环:

- `9b917c6` **share 真接场景**:design D4 原计划"复用 SceneDetailClientPage",但代码现实是这个组件强依赖 7+ 个受保护 API(学习态/练习/保存),搬过去会一连串 401。改成**新建 `ShareScenePreviewClient` 子组件**——主路由完全不动(保护不收项 §11.2)、走 supabase anon client 直读 scenes 表(RLS 已开 is_public=true)、触发 `explain-selection` 时带 X-Anonymous-Id 头。**面试讲点**:决策跟实施不一致时,选"不污染主链路"而不是"硬塞 anonymous mode 进主组件"。
- `b2aef79` **漏斗 + 回滚**:`anon_register_prompt_clicked` 4 个触发点接入(L1/L2/L3 各点击都上报);counter 加 `decrDailyCounter`,quota 命中阈值后 INCR 回滚,防止持续失败请求让 count 漂移到 limit+N(否则 Redis 计数会一直涨,虽然不影响判定但脏)。
- `7f6aff5` **TTS 接匿名**:design 说"TTS 在匿名分支仅允许播放预生成音频",但首版漏接了。补一个新路由 `/api/anonymous/tts/play` GET,**关键决策:`tts_play` 不进 HighCostCapability 数组**(它只读 Storage 零边际成本,加进去会污染 admin 紧急关闭面板 / 用户日 quota 表 / usage 统计的"高成本"语义)。新建独立 `tts-playback-quota.ts` 模块,复用 counter + IP 滑窗 + 漏斗埋点基础设施,但 quota 体系完全分家。

### 12.9 这次自评审的两个工程方法可以单独讲

1. **跟自己的代码做 code-review**:不只是 PR 流程那种"看了再说",而是把 proposal/design/spec 跟实际实现逐条比对,找"设计了但没接通"的事(这次发现的 7 个问题里有 4 个属于这类:漏斗 0 调用、share 渲染引导墙、cache 头无效注入、TTS 没接)。
2. **依赖注入 + audit test 双护栏**:server 模块全部把外部依赖(supabase admin / counter / requireProfile)作为 dependencies 形参,测试用 fake 注入;再加一个 SQL parser 解析 migration 文件的 audit test,守护"所有用户态表不能给 anon 加 SELECT 策略"这类**不变量**——文档说"以后新建表要注意"是没用的,改了测试会红才有用。

### 12.10 面试可以这样讲

> 这个能力做下来,核心收获不是"实现了匿名访问",而是"想清楚了一个新增功能怎么不去污染已有链路"。每一层都有显式的隔离边界:cookie / Redis 命名空间 / RLS 策略 / 路由前缀 / env 开关。出问题能精确归因(漏斗 + Sentry user_type tag),能精确止血(env gate 一切)。

> **30 秒电梯版**:"分享链接做了灰度匿名试用。身份用 HTTPOnly cookie + IP daily salt + 表权限四层套娃,配额走 Redis 命名空间和已登录用户严格隔离,前端三层注册引导,8 事件漏斗 + 成本看板把转化和成本都做成可观察的。出问题一个 env 开关一切。"

## 13. 最值得讲的工程取舍

### 13.1 没做大而全的微服务拆分

当前用 Next.js 接口 + Supabase 跑通闭环,开发速度快、联调成本低。等 TTS、推荐、AI 生成这些负载真的大到一台机器扛不动,再拆服务更合理。

### 13.2 没引入重型全局状态管理

真正需要长期保存的是学习状态和表达资产,不是前端 UI 状态。把长期状态放服务端,比堆一个全局仓库更有价值。

### 13.3 没做 ffmpeg 音频拼接

循环播放需要的是后台播放稳定性,不是音频编辑能力。分段级别打包能复用已有的 TTS 通道,复杂度更低,也和缓存链路一致。

### 13.4 没追求完整离线

当前做的是缓存兜底和二次播放加速,不是完整的离线应用。这样能先解决高频体验问题,避免维护成本过高。

## 14. 面试官追问树

按"面试官继续往下问"的方式组织。准备时不要死背整段,要记每个问题的回答结构:**先说问题约束,再说当前方案,再说边界**。

### 14.1 学习状态为什么放服务端

回答结构:

1. 业务约束:学习进度会影响今日推荐、复习队列、进度统计,不是当前页面的 UI 状态
2. 状态动作:开始 / 进度 / 暂停 / 完成 / 练习 / 变体,这些都是事实,要在服务端记录
3. 工程收益:刷新可恢复、跨页面可消费、重复提交可控、后续能审计和推荐
4. 边界:页面内的临时交互(展开了哪一段、当前选中哪个)还是留前端

可以这样答:

> 我把学习状态分两类:只影响当前页面体验的(比如展开了哪一段)留在前端;本轮学习是否开始、推进到哪、暂停、完成,这些会影响后续推荐和复习的事实放服务端。这样不管用户刷新、回到主页还是去做复习,大家消费的都是同一份事实,不会出现"前端说我学完了但后端不知道"。

容易被追问的点:

- **重复点击怎么办**:关键写操作有"去重钥匙",短时间内同样的请求会复用结果
- **网络失败怎么办**:前端不把"看起来成功"当成最终事实,以服务端响应为准
- **为什么不用 Redux**:Redux 只能解决前端共享,解决不了跨会话、跨设备、可审计

### 14.2 TTS 为什么要做这么多层缓存

回答结构:

1. 成本:TTS 生成慢、收费,而且用户会反复播放同一段
2. 缓存钥匙:不同粒度(整段、单句、整场景)有各自稳定的钥匙,不能混用
3. 服务端:先查文件存储有没有,有就返回,没有才生成
4. 客户端:浏览器本地缓存、内存缓存、避免重复请求、避免重复创建播放对象
5. 边界:不是完整离线,第一次没缓存还是要联网

可以这样答:

> 音频不是普通 JSON 数据。它生成慢、成本高,而且学习场景里会反复播放。所以我把它做成"稳定资源":服务端按内容算一个钥匙,优先看文件存储里有没有;客户端优先读本地音频,即使临时下载链接过期了,本地的音频文件还能继续播。这样第二次播放不再依赖远端接口。

容易被追问的点:

- **临时链接过期怎么办**:浏览器本地存的是音频文件本身,过期后转成本地播放地址继续播
- **缓存会无限大吗**:浏览器本地缓存有条目数和总大小上限,会自动清掉旧的
- **为什么还要内存缓存**:同一会话里反复点同一个表达,内存缓存能避免重复请求和重复创建播放对象

### 14.3 循环播放为什么是分段级打包

回答结构:

1. 目标:不是音频编辑,是后台 / 锁屏播放的连续性
2. 问题:一段一段播依赖 JS 在每段结束时切下一段,后台不稳定
3. 方案:同一天稳定的候选 → 转成统一格式的音频片段 → 合并 → 复用 TTS 通道生成整体音频
4. 失败处理:打包失败回退到一场景接一场景的队列
5. 代价:不实时切换"当前场景名",不做"边播边热插"

可以这样答:

> 我没有做 ffmpeg 拼接,因为需求不是音频剪辑,而是让后台播放更稳定。播放前把多个已学场景转成统一格式的音频片段,再复用现有 TTS 通道生成整体音频。播放开始后交给浏览器原生的循环播放,减少后台 JS 切歌的不确定性。

容易被追问的点:

- **候选变了怎么办**:同一天稳定打包优先,刷新或重新准备时生成新打包
- **为什么不随机**:完全随机会降低预加载命中率,稳定顺序能让预加载和点击播放指向同一份资源
- **为什么不是完整离线**:当前目标是缓存兜底和播放连续性,不承诺离线可用

### 14.4 接口治理具体保护了什么

回答结构:

1. 分三类风险:**公开注册入口**、**高成本的读 / 生成接口**、**关键写接口**
2. 注册:模式开关 + 邀请码 + 邮箱验证码 + 邮箱链接验证 + IP 频率控制
3. 高成本接口:用户 / IP 限流 + 每日额度 + 后台紧急开关
4. 关键写接口:去重保护(学习推进、复习提交、表达保存)
5. 排障:请求编号 + 统一错误返回 + 结构化日志
6. 权限:请求来源校验 + 账号状态 + 数据库行级权限逐步收紧

可以这样答:

> 我不是把接口治理当一个统一中间件口号,而是按风险来补。注册入口要有模式开关、邀请码、邮箱验证码、邮箱验证和 IP 频率控制;TTS 和 AI 生成这类高成本入口要限流、每日额度和紧急开关;学习推进、复习提交、表达保存这些写操作要去重;所有错误响应都带请求编号,方便从前端报错、接口日志和业务日志串起来。

容易被追问的点:

- **限流存哪里**:有共享存储(Upstash 这个 Redis 服务)就用它,没有就退回到本地内存
- **邮箱验证码是不是自建**:注册页那 6 位验证码是项目内自己做的最小闭环;账号体系、邮件链接确认这些还是用 Supabase 提供的
- **去重是不是持久化**:目前是轻量内存方案,适合当前阶段;并发再大可以迁到共享存储
- **数据库行级权限做到什么程度**:主要用户表逐步承接,不是完整的平台化权限系统
- **错误响应一致性怎么收口**:不是改一处补一处,而是先盘点(58 个接口分三档优先级),再起一个统一变更一次性收口高优问题,把"高成本接口必须按业务模块打日志"写进规范

### 14.5 AI 能力怎么避免变成 demo

回答结构:

1. 项目不是聊天框,是学习闭环
2. AI 输出必须进入业务生命周期:学习、保存、练习、复习、统计、推荐
3. 提示词只是入口,后面还有校验、结构化、落库和 UI 承接
4. 高成本 AI 接口要进限流、错误追踪和兜底体系

可以这样答:

> 我更关注 AI 输出进入业务后的生命周期。比如生成一个场景不算结束,它要能变成可学习的场景;生成一个相似表达也不算结束,它要能进入表达资产、表达簇和复习。AI 在这里是内容和训练能力的生产方式,不是孤立 demo。

### 14.6 这个项目最能体现工程能力的地方

不要只答"用了 Next.js 和 Supabase"。更有说服力的回答:

> 最能体现工程能力的是:我没有只做功能页面,而是把学习状态、表达资产、音频资源、AI 生成和接口治理放到同一个闭环里处理。一个场景不是展示完就结束,它会产生学习状态、表达资产、复习信号和音频资源,这些又会回到今日任务和进度页。技术设计服务的是这条闭环,而不是单点功能炫技。

可以补三点:

- 复杂状态放服务端,页面保持薄
- 高成本资源做缓存、预加载、失败兜底
- 有明确边界,不提前做大平台、不做完整离线、不做重型状态管理

### 14.7 你怎么证明这些不是简历上的空话

回答结构:

1. 先把能力落到用户链路,而不是先报代码文件
2. 再说对应的接口入口、服务层、缓存层或测试
3. 最后说验证方式:单元测试、交互测试、文档约束

可以这样答:

> 我会按链路证明。比如 TTS 不是一句"做了缓存",而是从音频接口进入服务端生成和文件存储命中,再到客户端先查本地缓存、内存缓存、避免重复请求,最后通过播放事件和管理后台看失败摘要。学习状态也不是一句"服务端管理",而是有开始 / 进度 / 暂停 / 完成入口,下游今日任务、复习、进度页消费同一份状态。

### 14.8 你遇到过最难的技术问题是什么

**优先讲 §14.15 那个第三方 TTS 库的线上事故** —— 那是真实生产环境的排查,有完整的"症状 → 根因 → 第一次修失败 → 重修 → 验证"闭环,比"后台音频播放稳定性"更有冲击力。

如果想讲"后台播放":

回答结构:

1. 问题:手机后台或锁屏后,页面 JS 不一定稳定执行下一段切换
2. 初始直觉:一段接一段播,但它依赖"播放结束 → 切下一段 → 开始播"这条 JS 链
3. 取舍:不用 ffmpeg 文件拼接,而是分段级打包,复用已有 TTS 通道
4. 结果:播放开始后尽量由单个整体音频承接,缓存命中也更稳
5. 边界:不是完整离线,不承诺所有浏览器都能后台启动新音频

### 14.9 项目里有哪些地方你刻意没做

这类问题要讲成熟度,不要只说"时间不够":

- **没做完整微服务拆分**:当前闭环优先,等独立能力负载变大再拆
- **没做完整离线应用**:当前先解决二次播放和缓存兜底,完整离线引入同步、版本和容量管理太复杂
- **没做重型状态管理**:长期状态应该在服务端,前端大仓库解决不了跨会话和可审计
- **没做 ffmpeg 拼接**:循环播放目标是连续性,不是音频编辑;分段打包更贴合现有链路

一句话收束:

> 我没把项目往"大而全平台"方向提前推,而是先把学习闭环里最影响稳定性和可维护性的点做实。

### 14.10 如果再给你两周会优化什么

回答要围绕现有边界,不要突然开一个新产品方向。优先级:

1. **强化观测**:当前本地事件 + 5xx 聚合够内测用了,下一步是升级到用户结果指标(7 / 14 / 30 天表达回忆率、主动使用率),用数据回答"产品目标是否在达成"
2. **细化 TTS 指标**:区分单段、整场景的"预热命中""冷启动""兜底失败"的比例,用数据判断预热是不是真的有用
3. **收紧接口契约**:把高频接口的输入输出和错误结构进一步稳定下来
4. **完善权限边界**:继续把用户数据访问收回数据库行级权限
5. **今日推荐引入"表达资产"维度**:现在推荐主要看入门路径和继续学习,引入"基于已保存表达推荐相关场景"能直接强化"迁移"维度,这是当前最大短板

- 立刻上完整 BI 平台
- 立刻做全量离线
- 立刻引入复杂推荐模型

这些可以作为长期方向,但不是当前最小高收益路径。

## 15. 技术细节深挖卡

这一节用来把"我做了某个技术点"继续往下讲到实现层。面试时不一定全讲,但被追问时可以从这里抽一条链路展开。

### 15.1 TTS 音频钥匙怎么保证不串味

可以这样讲:

> 音频缓存不是按网址粗暴缓存,因为临时下载链接会过期。我先把音频抽象成"稳定钥匙",再让服务端文件存储、浏览器本地缓存和播放层都围绕同一套钥匙工作。

关键细节:

- **单个表达的音频**:按清洗后的文本算钥匙,适合跨页面复用
- **句子 / 段落的音频**:把句子编号、说话人、语速和文本一起算指纹,避免同一句在不同语速下混用
- **整场景的音频**:先合并所有片段,再按整段内容和场景类型算哈希,任何内容变化都会换钥匙
- 浏览器本地缓存用稳定钥匙,不直接依赖会过期的临时链接

被追问时补一句:

> 所以"二次播放更快"不是因为链接没过期,而是因为真正复用的是稳定的音频资源和浏览器本地文件。

### 15.2 TTS 生成接口的链路怎么讲

回答结构:

1. 接口先做用户身份校验
2. 校验参数,解析类型(单段 / 句子 / 整场景)、语速、文本、分段
3. 先查服务端内存里的临时链接缓存
4. 再查 Supabase 文件存储里有没有这个对象
5. 命中就直接返回链接
6. 没命中才用 TTS 库生成音频文件,上传到存储
7. 上传失败时返回"直接内嵌音频数据"作为兜底,保证首次播放也有得听

可以这样答:

> TTS 链路是"先命中已有的,再生成"。这能避免用户每次点播放都触发合成。运行时真正复用的是 Supabase 的文件存储;本地代码目录里 `public/audio` 只是开发或兼容副产物,不算生产主缓存。

边界要讲清楚:

- 服务端的临时链接缓存是进程内存里的,不是跨实例强一致
- 当前判断"文件存不存在"用的是"尝试创建临时链接",规模变大后可以优化成查对象元信息
- 内嵌音频数据是兜底,不是主路径

### 15.3 客户端音频缓存为什么分四层

四层按"减少什么成本"来讲:

| 层 | 解决什么 |
|---|---|
| 内存网址缓存 | 减少同一会话重复请求音频接口 |
| 浏览器本地文件缓存 | 绕过临时链接过期问题,保存音频文件 |
| 预加载状态集合 | 避免同一个网址反复触发预加载 |
| 本地播放地址复用表 | 避免同一份文件反复创建本地播放地址 |

可以这样答:

> 客户端不是简单的"有缓存就用",而是按资源生命周期拆层。网址缓存解决短期重复请求,本地文件缓存解决链接过期后的复用,播放地址复用解决播放层重复创建对象。

### 15.4 场景详情为什么按"段落"播,不是按"句子"

可以这样讲:

> 场景详情的主播放单元从"单句"调整成"段落",是为了更贴近用户真实"听一段对话"或"听一个语义块"的方式。技术上仍然复用句子级的 TTS 通道,不新增接口协议,避免为了 UI 颗粒度变化扩接口。

关键细节:

- 段落音频复用句子级别的 TTS 类型
- 段落的"标识"用"block-段落ID"的格式,避免和真实句子混淆
- 段落文本优先来自段落自带的语音文本字段,没有时拼接段落内句子的文本
- 业务事件里用"音频单元 = block"区分"复用句子通道"和"真实句子音频"

取舍:

- **优点**:用户点击更贴近段落听熟;接口不新增协议
- **代价**:调试时必须看"音频单元"字段,否则容易误以为主链路仍在播放单句

### 15.5 整场景音频失败为什么不自动逐句串播

回答结构:

1. 整场景音频失败原因有多种(供应商错、超时、片段拼接失败、上传失败、链接失败、结果为空、冷却中)
2. 自动逐句串播会引入一套新的播放状态机,尤其涉及后台、暂停、切换、错误续播
3. 当前最小可维护方案是:识别失败原因 + 失败摘要 + 45 秒冷却 + 显式按钮回到逐段播放

可以这样答:

> 我没有把整场景失败自动降级成逐句串播,因为这会把一个"生成失败"问题扩大成新的"播放器状态机"问题。当前方案是识别失败原因,短时冷却避免重复打上游,再给用户明确的逐句跟读入口。这样能兜住体验,又不引入难维护的自动串播逻辑。

可展开的失败原因:

- 供应商出错
- 超时
- 片段拼接失败
- 上传失败
- 链接生成失败
- 结果为空
- 冷却中(短时间内已经失败过)

### 15.6 预加载收益怎么观测

可以这样讲:

> 我不是只做预加载,还做了轻量的"预加载登记表",把"预加载入队"和"后续播放事件"用稳定钥匙关联起来。这样能看"预加载命中"和"冷启动"的差异,而不是凭感觉判断有没有用。

关键细节:

- 登记表的钥匙和浏览器本地缓存的钥匙一致
- 来源区分"首次进入""空闲触发""播放触发"
- "用户点击" **不算**预加载来源,因为用户点击是即时需求,不是预加载收益
- 管理后台的统计是基于本地事件计算的,不代表全量用户数据
- 失败有冷却阶梯(5 秒 → 60 秒 → 5 分钟 → 30 分钟),疑难内容失败后不会立即重试;30 分钟静默后失败计数才重置
- **跨场景取消**:用户从场景 A 切到场景 B 时,A 的预加载任务会被取消,避免后台空跑占带宽
- **弱网统一守卫**:`scheduleLessonAudioWarmup` / `scheduleSceneIdleAudioWarmup` 内置 `shouldAvoidHeavyAudioWarmup()`(检测 `saveData` / `2g` / `slow-2g`);[chunks 页](src/app/(app)/chunks/page.tsx)用的是更底层的 `scheduleChunkAudioWarmup`,守卫要在调用处加 —— 不然首屏 6 条 × 2 套预热在 2G 下会把带宽打满

补边界:

> 样本少时"预加载收益"可能为负,这不是失败,只是说明当前调度需要继续观察。

### 15.7 场景练习题生成链路怎么讲

回答结构:

1. 场景详情先把课程数据转成可出题的结构
2. 让 AI 基于段落、句子、表达出题
3. AI 返回结构不合法时,回退到本地出题逻辑
4. 把填空题的题面和答案对齐
5. 前端最终组装成四层训练模块:**填空 → 引导回忆 → 句子复述 → 完整听写**
6. 前一模块全部答对后,下一模块才解锁

可以这样答:

> 练习生成不是模型返回什么就直接展示。AI 负责提供题目候选,本地还会做结构校验、兜底、填空标准化和四层训练模块编排。这样能兼顾生成灵活性和前端训练流程稳定性。

可展开的取舍:

- 首轮填空当前更偏保守,最多围绕 5 到 8 题
- 兜底会优先挑高价值表达,但不会把同一句拆成很多碎空
- 引导回忆会过滤太短的句子,避免半句复现没有训练意义

### 15.8 练习题为什么要在服务端持久化

可以这样讲:

> 练习题不能只存在本地缓存里,否则用户刷新、换设备或提交答题时,服务端无法确认这套题是不是属于当前用户和当前场景。现在练习题本身会保存到服务端,本地缓存只负责"秒开"和失败兜底。

关键细节:

- 生成接口只负责生成题目,不负责落库
- 场景详情生成或重新生成后,会调用专门的接口把练习题保存到用户名下
- 答题写入前会校验"这套题真的属于当前用户和当前场景"
- 手动重新生成会创建新的一套,旧的标记为废弃

边界:

- 当前不做公共题库
- 不做跨用户共享题集
- 不把题目拆成标准化的公共表

### 15.9 表达保存链路为什么复杂

可以这样讲:

> 表达保存的复杂点不在 UI,而在"保存动作会同时影响多张表"。所以项目把数据拼装、写入语义和表达簇操作拆开,避免每个入口各自拼一套规则。

关键实体:

- **表达本身**:标准化的表达实体
- **用户维度学习条目**:区分整段表达和单句
- **用户维度关系表**:相似、对照
- **用户维度表达簇**
- **用户每日学习统计**

关键边界:

- 相似表达可以进入同一个表达簇
- 对照表达只写"对照关系",不应并入相似表达簇

被问"为什么复杂"时:

> 因为表达资产不是一条收藏记录。它后续要支持相似表达、对照表达、表达簇、复习和统计,所以保存时必须把关系和资产结构一起维护。

### 15.10 接口分层怎么讲得更具体

可以按这条链路讲:

- **接口入口层**:身份校验、参数解析、调用业务层、返回统一响应
- **可测试动作层**:部分模块把接口里的动作抽出来,可以脱离 HTTP 单独测
- **业务层**:业务语义、状态推进、跨表副作用
- **数据访问层**:Supabase 查询
- **共享逻辑层**:纯函数、选择器、规范化和可单测规则

可以这样答:

> 我没有做非常重的领域驱动设计,但也避免接口入口堆所有逻辑。接口入口主要处理 HTTP 边界,业务层承担业务语义,数据访问层处理查询,共享逻辑放纯函数。这样测试时能绕开 HTTP 直接测,也能在需要时补真实 HTTP 验证。

### 15.11 测试怎么证明链路不是手测

回答结构:

1. 纯规则用单元测试(提示词构造、业务逻辑、缓存钥匙、去重)
2. 页面交互用组件测试
3. 高风险入口用接口测试,验证错误结构和请求编号
4. 维护类脚本检查文字乱码、规范一致性、护栏
5. 不是默认跑全量,而是按改动风险跑最小相关集

可以这样答:

> 我会按风险选测试,不是无脑全量。改 TTS 就跑音频接口、预加载、服务端 TTS、重新生成接口;改场景练习就跑练习生成、场景练习视图、练习题;改表达保存就跑保存契约、手动组合、焦点辅助和页面交互。

面试时可以强调:

> 测试策略和架构拆分是配套的。逻辑拆到业务层 / 共享层后,才能用轻量测试覆盖核心语义;交互测试只覆盖真正需要 DOM 的路径。

### 15.12 复习调度怎么消费 5 类正式信号

这是项目里比较深的一条链路。面试官追问"复习不只是对错怎么实现"时可以展开:

回答结构:

1. **UI 收集**:复习页面收集 5 类信号 —— 识别状态、输出信心、完整输出状态、变体改写状态、完整输出覆盖度
2. **数据库落库**:专门的表保留每次复习完整事件 + 主反馈(再次 / 困难 / 良好)
3. **排序消费**:把 5 类信号映射成 5 级"焦点"(比如"输出信心低""目标表达未覆盖"),决定到期列表的优先级
4. **节奏消费**:在保留主方向的前提下,根据信号细调下次复习时间(12 小时到 10 天)
5. **解释回流**:"焦点"字段透传到 UI,映射成稳定文案("仍需输出训练""尚未覆盖目标表达"),用户能看到"为什么这条优先出现"
6. **历史兼容**:所有空值走保守中性解释,新字段按新增记录逐步生效,不要求回填

可以这样答:

> 复习不是简单的间隔重复算法。我们已经把"用户识别 / 主动回忆 / 变体迁移 / 完整输出" 4 类阶段信号正式入库,并且让排序和节奏都按这些信号细调。比如"完整输出未覆盖目标表达"的条目,即使用户提交"良好",下次也不会被排到很远;提交"再次"但同时未完成迁移训练的,会进一步收紧到 12 小时。这让复习不止"判断对错",而是基于"在哪个训练深度卡住"的细粒度调度。

被追问的点:

- **为什么不直接覆盖再次 / 困难 / 良好**:这 3 个是用户最直接反馈,新信号是补充维度;保留主方向兼容前端 UI 和历史数据
- **为什么新信号有空值兼容**:历史复习没有这些字段,强制回填会伪造结果;保守中性解释比假数据更安全
- **当前最大缺口**:有信号 + 有消费 + 有 UI 解释,但没有量化指标(7 / 14 / 30 天表达回忆率)来验证"这套调度真的让用户记住更多"

### 15.13 可观测性的多个入口

被问"出事故了怎么定位"时按入口讲,不要只答错误监控:

| 入口 | 用途 | 时效 |
|---|---|---|
| 错误监控平台 | 生产 5xx 自动聚合 + 请求编号标签 + 按业务模块自动分桶 | 分钟级告警 |

### 15.14 超重页面多轮拆分:工程方法论的真实案例

被问"项目里有哪些能体现工程方法论的东西"时,这是一条很有说服力的链路 —— 有量化数据,有"失败 → 反思 → 改策略 → 验证"的完整闭环。

#### 背景

表达管理页一开始是 2368 行单文件,是仓库里最大的前端文件。直接重写有两个问题:一是一次性大改容易出回归 bug;二是不知道哪种拆法收益最高。所以走"按规范化变更多轮迭代"的路线,每一轮都记录拆法和减幅,给下一轮策略提供数据。

#### 时间线和量化

| 轮次 | 起始行数 | 收口行数 | 减幅 | 关键策略 |
|---|---|---|---|---|
| 第一轮 | 2368 | 2125 | -243(-10.3%) | 抽样式常量 + 3 个 hook + 顶部组件 |
| 第二轮 | 2125 | 2102 | -23(-1.1%) | 抽 3 个 hook + 一个区域组件 |
| 第三轮 | 2108 | 2041 | -67(-3.2%) | 抽"属性很多的子树"(一次抽 41 个属性的列表组件) |

场景详情页同模式应用一次:**1326 → 849 行(-36%)**,抽出 4 块逻辑。

#### 关键发现(写进第二轮规范,直接指导第三轮)

第二轮拆完发现减幅远低于预期。逐对象算账才发现:

| 抽离对象 | 移走代码 | 页面新增"装配代码" | 净减 |
|---|---|---|---|
| 表达图相关 hook | ~10 行 | ~20 行解构 | **+10**(反而增长) |
| 句子表达保存 hook | ~26 行 | ~8 行解构 | -18 |
| 焦点关系标签 hook | ~10 行 | ~13 行解构 | +3 |
| 焦点模式区域组件 | ~55 行 JSX | ~42 行属性透传 | -13 |

**关键发现**:抽小 hook 时,"页面里写的装配代码"和"抽走的代码"差不多抵消,页面几乎不瘦身。

**对第三轮的指导**:第三轮应该优先抽"属性很多的子树",而不是继续抽小 hook。

#### 第三轮验证发现成立

第三轮按第二轮策略只抽一个"属性多的列表子树",把和这个子树强耦合的辅助函数一并迁入,减幅 **-67 行,大约是第二轮的 3 倍**。策略验证成立。

#### 怎么讲

> 这条线最值得讲的不是"我把大页面拆小了",而是"我把'怎么拆'变成了一个可量化、可迭代的工程问题"。第一轮收口后我没立刻跳进下一轮,而是先看减幅;第二轮减幅低于预期,我把每个抽离对象的"移走代码 vs 新增装配开销"拆开算账,发现"装配代码"和"抽走代码"近似抵消;这个发现直接写进规范,指导第三轮改抽"属性多的子树",实际减幅立刻翻 3 倍。核心是把策略验证留在规范里,下一个维护者(或 AI)能直接复用结论,不用从头试错。

#### 容易被追问

- **为什么不一次重写**:重写成本高、风险大,且不知道目标值;多轮迭代每次都有规范变更把策略和量化留档
- **怎么保证不破回归**:每轮拆分都要求页面渲染结果字节级兼容(相同的 className、属性);交互测试不弱化断言;每一轮跑完整测试
- **为什么把"关键发现"写进规范而不是开发日志**:规范是给"未来推第四、第五轮的人"看的硬约束,开发日志是给当下的人看的实施记录

### 15.15 线上事故复盘:第三方 TTS 库的隐藏 bug 导致进程反复崩

这是非常新鲜的真实生产事故,完整的"症状 → 根因 → 第一次修复失败 → 重修 → 验证"闭环。面试官问"最难的技术问题""线上事故复盘""怎么处理第三方库的 bug"都能直接讲。

#### 症状

用户反馈"网站时好时坏,刷新几次能进去"。Chrome 报"连接已重置",看网络面板请求大约 300 毫秒就被打断,0 字节响应 —— 是网络层被强制关闭,**不是后端返回了 5xx 错误**。

服务器上看进程状态,主应用的"重启计数"已经 102 次,但运行时间只有 5 分钟。说明进程在反复崩 + 自动拉起,拉起的窗口里 nginx 拿不到后端响应 → 用户看到的就是"连接被打断"。

#### 根因

进程日志反复出现:

```
未捕获异常: TypeError: Cannot read properties of undefined (reading 'audio')
```

源头是 `msedge-tts` 这个 npm 包 —— 我们用来生成 TTS 的库。

它在 WebSocket(一种长连接)的回调里,**裸读了一个对象的 `audio` 属性**。当我们应用层主动关掉连接后,微软 Edge 服务器还会继续推几帧残余数据,这些回调拿到的对象已经被清掉了,访问 `.audio` 就是访问"未定义.audio" → 报错。

这个错误**发生在长连接回调里,完全在我们 `await` 边界之外**,调用方的 `try / catch` 拦不到 → 直接冒到 Node 的"未捕获异常" → 进程崩。

#### 第一次修复失败,真正原因

直觉做法:加一个"进程级守卫",**只**白名单这一类已知错误(其它异常仍然让进程退出,保留出错及时崩的原则)。

第一版用"错误栈包含 msedge-tts"这个条件匹配。本地开发环境的栈长这样:

```
at WebSocket.onmessage (.../node_modules/msedge-tts/dist/MsEdgeTTS.js:142:42)
```

部署生产后,栈变成:

```
at ignore-listed frames
```

—— **Next.js 生产模式会把 `node_modules` 里的栈折叠成这一行**。我的"栈包含 msedge-tts"永远拿不到匹配,守卫没生效,进程继续崩。

这是一个**只在生产复现、本地永远查不到**的坑。教训:排查时要把"开发环境符号丢了没"和"生产环境符号丢了没"分开看。

#### 最终方案

改成只用**错误消息特征匹配**:

```typescript
const isKnownMsEdgeTtsWebSocketRace = (error: unknown): boolean => {
  if (!(error instanceof TypeError)) return false;
  const message = error.message ?? "";
  return (
    message.includes("Cannot read properties of undefined") &&
    (message.includes("'audio'") || message.includes('"audio"'))
  );
};
```

三层约束保证白名单不会误伤:

- 必须是 TypeError(这条消息只 TypeError 才会发)
- 消息同时命中 "undefined" 和 "audio"(单双引号变体都覆盖)
- **前置条件**:应用层所有读 `.audio` 的地方都加了"可选链"(`?.`)保护,**这才是主防线**,进程级守卫只做兜底

#### 测试覆盖

6 条单元测试,正反两侧都覆盖:

- 命中:开发模式完整栈
- 命中:**生产折叠栈**(防止后人改时退化回栈匹配又踩同样的坑)
- 命中:双引号变体
- **不**命中:同形态但属性名不是 audio 的 TypeError
- **不**命中:同消息但不是 TypeError(普通 Error 仍然让进程退出)
- **不**命中:`null` / `undefined` / 字符串等非 Error 拒绝原因

#### 验证生效

- 进程"重启计数"从 102 降到 0,运行时间持续增长
- 日志出现"已吞下 msedge-tts 长连接竞态"这条记录,**证明守卫确实在拦,不是 race 自己消失了**
- 用户反馈"刷新几次才能进"的现象消失

#### 怎么讲

> 我们用一个第三方 TTS 库,它的长连接回调里有时序竞态:我们应用层主动关连接后,远端服务器还会继续推几帧,触发对已清理对象的裸读 → TypeError → 未捕获异常 → 进程崩,生产累计了 102 次重启。这种异常出现在 await 边界外,应用层拦不到。
>
> 我加了一个进程级守卫,**只**白名单这一类特定错误,其它异常仍然让进程退出保留出错及时崩的原则。但第一版踩了个生产专有的坑 —— 我用错误栈包含库名来匹配,本地测都通过,生产没生效。后来才发现 Next.js 生产模式会把 `node_modules` 的栈折叠成 "at ignore-listed frames",我的匹配条件永远不命中。改成只用错误消息特征匹配,同时把"生产折叠栈"的情况写进单元测试防止后人退化。
>
> 这件事让我对"白名单异常"的纪律更严格:必须有具体到错误类型 + 关键消息特征的匹配;必须有覆盖生产符号丢失情况的测试;**只在应用层已经主动防护之后才用进程级白名单兜底**,而不是反过来。

#### 容易被追问

- **为什么不 fork 这个库修源码**:技术上可行,但维护一个 fork 的长期成本高于"应用层防护 + 进程级窄白名单"两层防护,后者在生产已经稳定
- **为什么不直接 catch 所有异常不让进程崩**:未捕获异常之后 Node 进程状态可能不一致(资源没关、事件循环残留),catch 所有会把"短崩快重启"变成"半死长崩",反而更难排查。窄白名单 + 其它一律退出才安全
- **怎么定位到是这个库的问题**:我们应用层所有读 `.audio` 的地方都有可选链保护,但日志反复出现同一个消息 → 只有这个库的源码里有裸读 `.audio` → 唯一嫌疑源
- **怎么验证不是误判**:不是看"崩溃消失了",而是看"已吞下"这条日志出现了 → 证明守卫确实在干活,race 还在发生只是被吞了。**如果消失了反而要怀疑是不是没生效**

### 15.16 预热效果怎么量化(warmupGain 这个指标怎么来的)

§14.6 提了「我做了预加载登记表能区分预热和冷启动」,但**怎么从这两个数算出「预热到底有没有用」**值得展开。

代码:[src/lib/utils/client-events.ts](src/lib/utils/client-events.ts) 的 `buildTtsWarmupEffectivenessSummary`,管理后台展示在 [src/components/admin/client-events-panel.tsx:81-120](src/components/admin/client-events-panel.tsx#L81-L120)。

**埋点维度的设计:**

每个音频播放事件带 6 个维度:`audioUnit`(sentence/chunk/scene_full)、`readiness`(cache/network/wait_fetch)、`wasWarmed`(布尔)、`warmupSource`(initial/idle/playback)、`failureReason`、`cooldownMs`。

**关键计算:**

```
warmHitRate = 预热过且命中缓存 / 预热过的总数
coldHitRate = 没预热过且命中缓存 / 没预热过的总数
warmupGain = warmHitRate − coldHitRate
```

`warmupGain` 这个差值就是**「预热到底带来了多少额外命中」**的直接量化。比如冷启动命中率 30%,预热后命中率 85%,gain = 55 个百分点。如果 gain ≈ 0,说明预热完全没用,该改逻辑;如果 gain 是负数(罕见),说明预热反而干扰了 —— 比如预热把不该缓存的内容塞进去挤掉热数据。

**为什么不直接看「总命中率」?**

总命中率会被「这一阵子用户都在看老内容(本来就命中)」拉高,看不出预热的边际贡献。`warmupGain` 是**控制变量**(同样是这批内容,有预热 vs 没预热的差),才能归因到预热本身。

**还按 source 分了三档:**

`sources.initial`、`sources.idle`、`sources.playback` 各算一份命中率。能看出「进页面就预热」「空闲时预热」「播放中预热」哪一档收益最大。如果发现 `idle` 的命中率比 `initial` 还高,说明用户行为决定的「真用到」的内容比「我猜用户要用的」更准 —— 那 initial 阶段可以收缩。

**为什么值得讲:**

这个故事闭合了「做优化 → 量化收益」这个环。没埋点平台不代表没数据驱动,我用 120 条本地事件 + 一个 React 算的 summary 表就能回答「这个优化是不是真的有用」。面试里如果问「你怎么证明你做的优化有效」,这是最直接的答案。

**还有第二条观测维度:用户点击的瞬间到底踩没踩到预热好的数据**

`warmupGain` 测的是「整体看预热值不值」,但还有个更直接的问题:**用户点「打开练习/变体」的那一下,数据是不是已经备好了**。这个事件叫 `practice_open_prewarm_hit/miss` 和 `variant_open_prewarm_hit/miss`(在 [src/lib/utils/client-events.ts](src/lib/utils/client-events.ts) 的 `ClientEventName`,记录在 [src/app/(app)/scene/[slug]/use-scene-detail-actions.ts](src/app/(app)/scene/[slug]/use-scene-detail-actions.ts) 的 `ensurePracticeSetReady` / `ensureVariantSetReady`)。

- `hit`:用户点开时数据已经在缓存里 → 预热确实在用户感知前面跑完了
- `miss`:用户点开时还在请求 → 预热没追上点击,或者根本没预热

跟 `warmupGain` 配着看:`warmupGain` 高说明缓存命中率提升明显,但如果 `variant_open_prewarm_miss` 很多,说明预热触发得太晚或者太窄(只预热了一部分变体)。这两个指标一起才能说清「预热到底有没有救到具体的开屏体验」。

顺带一提,**变体打开这条路径之前漏了音频预热** —— `handleOpenVariant` 只在切换 ID 时刷 token,没顺手把变体课的音频暖起来。补上 `scheduleLessonAudioWarmup(variantLesson, { sentenceLimit: 2, chunkLimit: 2 })` 之后,变体页打开时音频已经能命中浏览器缓存。这个改动让 `variant_open_prewarm_hit` 的统计变得有意义 —— 之前根本没在预热,统计永远是 miss。

### 15.17 高消耗接口的额度是「两阶段」写的(reserve → mark)

§10 提了「每日额度 + 后台紧急开关」,但**怎么实现「成功才扣、失败回滚、并发安全」**这个细节值得展开。

代码:[src/lib/server/high-cost-usage.ts:105-171](src/lib/server/high-cost-usage.ts#L105-L171),调用样式 [src/app/api/tts/route.ts:28-39](src/app/api/tts/route.ts#L28-L39)。

**两阶段:**

```typescript
// 阶段 1:reserve(占名额,数据库 RPC 原子操作)
const reservation = await reserveHighCostUsage({ userId, capability: 'tts_generate' });

// 阶段 2:try 业务,然后 markSuccess / markFailed
try {
  result = await generateTtsAudio(payload);
  await markHighCostUsage(reservation, 'success');
} catch (error) {
  await markHighCostUsage(reservation, 'failed');
  throw error;
}
```

**数据库里维护 3 个计数,不是 1 个:**

- `reserved_count`:占了几次额度(还没出结果)
- `success_count`:成功几次
- `failed_count`:失败几次

`reserved_count` 是限流上限的依据(预防并发把额度打穿),`success_count` + `failed_count` 用来对账:**reserved - success - failed 应该 = 0**,如果不为 0 说明有请求挂了没回写,可以排查问题。

**为什么不只用「成功才扣」?**

如果只在成功后扣,**并发场景下额度会被打穿** —— 用户限 10 次,同时发起 20 个请求,先到的还没完成,后到的看「success_count=0」全部放行。reserve 阶段先把名额占住,后来者就被挡了。

**为什么不只用「先扣后看是否成功」?**

那样失败的请求也占额度,用户体验差(我点了 10 次接口都挂了,告诉我额度满了)。所以失败要回滚 —— `markFailed` 这一步会把 `failed_count` 加一同时把 `reserved_count` 减一(或者用其他对账方式),让额度可以重试。

**紧急开关:**

`app_runtime_settings.high_cost_disabled_capabilities` 是个 JSON 数组,后台改一行 → 所有 `reserveHighCostUsage` 调用都抛 `HighCostCapabilityDisabledError`,立刻阻断。**不用重启服务**。这是 §10 提的「紧急开关」的具体实现。

**Q:reserve 用的是什么?DB 事务?**

是 Postgres 的 RPC(`reserve_daily_high_cost_usage`),SQL 函数里做「读当前 count + 比对 limit + 写新行」三步,数据库层面保证原子性,Node 这边只要 await。比应用层 SELECT-then-UPDATE 安全得多(后者要自己加分布式锁)。

**Q:跟限流 (rate-limit) 是什么关系?**

两个不同维度。**限流**是「单位时间不能超 N 次」(防爆击),**额度**是「单天累计不能超 M 次」(控成本)。一个用户能一天均匀地把额度跑满,但不能 1 秒内全打过来。代码里两者也分离:`enforceHighCostRateLimit` 和 `reserveHighCostUsage` 是独立两层,顺序是「限流先过 → 额度再占 → 业务跑」。

### 15.18 关键写接口的去重保护是 Promise/Value 双态切换

§10 提了「重复请求去重」,但**怎么做到「连点 2 次的并发请求也合并、不只是顺序请求合并」**这个细节是关键加分点。

代码:[src/lib/server/idempotency.ts](src/lib/server/idempotency.ts) 的 `runIdempotentMutation`,调用样式 [src/app/api/phrases/save/route.ts:25-33](src/app/api/phrases/save/route.ts#L25-L33)。

**核心数据结构:**

```typescript
interface CachedEntry {
  expiresAt: number;
  value?: unknown;    // 已经出结果
  promise?: Promise<unknown>;  // 还在进行
}
```

同一个 key 在 Map 里有**两种状态**:进行中(挂 Promise) / 已完成(挂 value)。来一个新请求:

- 如果 entry 不存在 → 执行业务,立刻把 promise 塞进 Map(**这一步是关键**),执行完切成 value
- 如果 entry 存在且挂的是 promise → **直接返回这个 promise**(并发请求复用)
- 如果 entry 存在且挂的是 value → 直接返回 value(15s 内的重复请求复用)

**这跟「先查 value、查不到再写 value」的朴素版本区别:**

朴素版本在并发场景下两个请求会同时看到「value 不存在」,各自跑一次业务,**完全没去重**。这套 promise/value 双态切换在第一个请求**进入 Map 的瞬间**就让第二个请求看到 promise 拿来 await,**真正合并了并发**。

**deterministic key 兜底(客户端忘传 header 也接得住):**

```typescript
export const buildDeterministicIdempotencyKey = (...parts) =>
  sha256(stableStringify(parts));
```

调用方:`getRequestIdempotencyKey(request, buildDeterministicIdempotencyKey(scope, userId, payload))`。客户端**没传** `x-idempotency-key` header 也会被服务端**用 scope+userId+payload 的 sha256 作为兜底 key**。所以连点两次同 payload 的请求,即使前端没生成 UUID,后端也能合并。

**失败立刻清掉:**

业务抛错时 `idempotencyStore.delete(normalizedKey)`,**不缓存失败结果**。下次同 key 请求会重新跑,不会被一个偶发失败永久卡住。

**为什么值得讲:**

去重听起来是个简单需求,但**「并发去重 + 自动兜底 key + 失败立刻重试」三件事一起做对**的实现不多。这套写法把「客户端忘传 header」「网络重试」「连点」「真并发」四种触发都覆盖了。

**Q:为什么 TTL 是 15 秒?**

经验值。够长:覆盖大部分「连点 / 重试 / 网络抖」的间隔;不太长:不会让用户「真心改了内容再提交」被旧结果拦截(15s 后人为修改重提交是正常的写)。

**Q:进程内 Map 在 PM2 cluster 下会不会失效?**

会。这是已知 gap —— 跨 worker 的去重目前没做。理由是:这套 idempotency 主要防「同一用户连点」,同一用户的连续请求大概率粘在同一个 worker(sticky session 或 nginx ip_hash),跨 worker 的真并发概率低。如果将来要做严格跨进程去重,把 Map 换成 Upstash Redis 就行,接口签名不变。

**Q:客户端这边还有没有去重?**

有,而且必要 —— 服务端 idempotency 是「跨进程的兜底层」,客户端 inflight Promise 是「同一会话里的第一道闸」。代码在 [src/app/(app)/scene/[slug]/use-scene-detail-actions.ts](src/app/(app)/scene/[slug]/use-scene-detail-actions.ts) 的 `ensurePracticeSetReady` / `ensureVariantSetReady`,用 `useRef<Promise<T | null> | null>` 存「正在跑的生成请求」:

```typescript
const inflightPracticeRef = useRef<Promise<PracticeSet | null> | null>(null);
// ...
let generationPromise = inflightPracticeRef.current;
if (!generationPromise) {
  generationPromise = (async () => { /* 真正发请求 */ })();
  inflightPracticeRef.current = generationPromise;
  const clearInflight = () => {
    if (inflightPracticeRef.current === generationPromise) {
      inflightPracticeRef.current = null;
    }
  };
  generationPromise.then(clearInflight, clearInflight);
}
await generationPromise;
```

**为什么不是只靠 `scheduleIdleAction` 的 key 去重?**

`scheduleIdleAction` 的 key 去重发生在「调度入队」这一层 —— 同一个 idle key 不会被排队两次。但**走快路径绕过 scheduler 的调用**(比如用户点击立刻触发 `ensurePracticeSetReady`)就跟 scheduler 排队的那次成了**真并发**,各自跑一次生成 API。Promise 引用 dedup 是在更下游的「发请求」这一层合并,这两条路径无论谁先到都会复用同一个 Promise。

**`.then(cleanup, cleanup)` 而不是 `.finally(cleanup)`,为什么?**

`.finally()` 链出来的是一个新 Promise,如果原 Promise reject 而新 Promise 没被消费,Node 会报 unhandled rejection。`.then(onFulfilled, onRejected)` 两个回调都返回 `undefined`,无论 resolve / reject 这条链都是「已处理」的,不会污染外部错误监控。

**两层一起讲是「双保险」叙事:**

- 客户端 inflight:同一会话里把「用户连点」「点击 vs 空闲预热」这种**真并发**合并成一个 API 调用 —— 网络都不发第二次
- 服务端 idempotency:即使客户端漏了(比如不同标签页同时点),Promise/Value 双态 Map 在服务端这一层再合并一次 —— 业务函数只跑一次

这是 §10「重复请求去重」在前后端都落地的具体形态,面试时可以连着 14.18 一起讲完。

### 15.19 客户端有一层错误 normalize,把后端契约翻译成用户中文

§10 讲了「所有接口出错都返回同一种结构(错误码 + 详情 + 请求编号)」,但**前端怎么消费这套契约**是闭环的另一半,值得讲。

代码:[src/lib/client/api-error.ts:100-225](src/lib/client/api-error.ts#L100-L225) 的 `resolveReadableMessage` + `createClientApiError`。

**翻译规则按优先级排:**

1. **网络层失败**(`Failed to fetch` / `NetworkError`) → "网络异常,请检查连接"
2. **限流**(status=429 / code=RATE_LIMITED) → "操作太频繁,请稍后再试"
3. **邀请码无效** → "邀请码无效或已过期"
4. **登录页 401** → "邮箱或密码不正确"(语义化,不暴露后端原话)
5. **其他 401 / AUTH_UNAUTHORIZED** → "登录已失效,请重新登录"
6. **403** → "没有权限执行此操作"
7. **5xx** → fallback message 或 "服务暂时不可用"
8. **兜底**:rawMessage 直接显示,但**排除包含 request id 字样的内部消息**(那是给开发者看的,不该让用户看到)

**requestId 一律打进 console:**

```typescript
console.error(`[client-api:${context}] requestId=${requestId}`, { status, code, error });
```

用户截图给客服时,console 里有 requestId,**直接拿去后端日志搜就能定位到具体一次调用**。不用让用户复述时间、操作步骤。

**ClientApiError 是个真实 Error 实例往上抛:**

带 status / code / requestId / rawMessage 四个字段,组件层 catch 之后既能 `toast.error(error.message)` 给用户看翻译版,也能拿 code/requestId 做更精细的处理(比如登录页拿到 invalid_credentials 就高亮密码框)。

**为什么值得讲:**

后端契约只是「材料」,把材料翻译成用户能看懂的中文是前端的本分。这套 normalize 把「错误响应契约 → 客户端消费 → 用户文案 + 开发者可排查」整条链路闭合了。面试如果问「你们怎么处理后端错误」,展开讲这一套比泛泛说「弹个 toast」深得多。

**Q:为什么不让后端直接返回中文?**

两个原因:① 后端 message 可能给多个端用(Web/API/未来 mobile),保持 English 更通用;② 同一个后端错误在不同上下文需要不同文案(401 在登录页是「密码错」,在其他页是「登录失效」),前端按 `context` 路由更灵活。

**Q:遇到 rawMessage 是合法英文怎么办?**

兜底显示。但加了一道过滤:`REQUEST_ID_PATTERN.test(rawMessage)` 命中就走 fallback,不显示。这是因为有些内部错误的 message 里带「Request ID: xxx」字样,直接给用户看体验差。

## 16. 分模块讲解卡

面试前快速过一遍每个模块讲什么、做到哪里、边界是什么。

### 16.1 今日任务

核心讲法:

> 今日任务不是入口列表,而是把"继续学习、复习、表达沉淀和进度"重新编排成今天的优先任务。

可展开:

- 为什么推荐这一步
- 如何承接关闭浏览器后再打开的恢复
- 如何把复习和进度的结果重新组织成下一步入口

边界:

- 当前不是复杂推荐系统
- 个性化还能继续增强

### 16.2 场景学习

核心讲法:

> 场景是主学习工作台,承载真实语境、音频、表达提取、练习和变体学习。

可展开:

- 页面只负责编排,复杂判断下沉到下层

> 表达管理不是收藏夹,而是表达资产工作台。

可展开:

- 保存后可以补全翻译、例句、用法说明、语义焦点
- 相似 / 对照表达和表达簇让表达形成关系
- 复习和今日任务后续能消费这些资产

边界:

- 当前仍偏重型工作台,后续可以简化操作路径
- AI 补全不是最终事实,仍需要业务结构承接和用户维护

### 16.4 复习

核心讲法:

> 复习不只是提交对错,而是把表达重新拉回回忆和输出训练,并产生后续调度信号。

可展开:

- 到期 / 提交 / 总结 分入口
- 复习结果写回服务端,影响后续回流
- 队列清空后有收束反馈和回今日任务的下一步

边界:

- 当前输出质量分析还不算很深
- 调度模型仍能继续细化

### 16.5 管理后台 / 可观测性

核心讲法:

> 管理后台不只是后台管理入口,也承接音频重新生成、缓存排障和最小业务事件回看。

可展开:

- TTS 浏览器缓存面板能辅助排查音频缓存
- 业务事件回看能看到最近的关键动作和失败摘要
- 管理按钮、列表、筛选逐步统一,降低误操作

边界:

- 当前不是正式 BI 或全量埋点平台
- 业务事件主要服务本地验收和排障

## 17. 产品问题准备卡

技术面试也可能追问产品判断。回答时不要把产品讲成"功能列表",要回到产品北极星:**让每一次场景学习沉淀为未来真实场景中能回忆、能使用、能迁移的表达资产**。

### 17.1 这个产品解决的真实问题

回答结构:

1. 用户问题:学过很多内容,但离开学习页后很难主动用出来
2. 产品判断:问题不只是"内容不够",而是学习入口、表达沉淀、复习回流和长期状态没有串起来
3. 方案:今日任务编排入口,场景提供真实语境,表达管理沉淀资产,复习做回忆训练,进度给长期反馈

可以这样答:

> 我想解决的不是"再做一个课程页",而是"学过的表达不能变成自己资产"的问题。很多学习产品停在"看懂内容",这个项目更关注后续能不能回忆、使用和迁移。所以它把场景学习、表达保存、复习训练和每日编排串成闭环。

### 17.2 为什么从场景开始,而不是从单词或题库

回答结构:

1. 单词和题库更适合局部训练,但容易脱离真实使用语境
2. 场景能提供人物、语气、目的和上下文,用户更容易理解表达为什么这样说
3. 表达从场景中被提取后,再进入表达管理和复习,既保留语境,又能沉淀成资产

可以这样答:

> 因为这个产品的目标不是让用户记住孤立词条,而是让用户未来在相似场景里能说出来。场景提供语境,表达管理把有价值的表达抽出来,复习再把它拉回主动回忆。这样输入、沉淀和输出之间是连续的。

容易被追问的点:

- **单词是不是不重要**:不是不重要,而是不作为产品主入口;词汇可以服务表达,但不替代表达
- **题库是不是不重要**:题目用于训练和反馈,但不能替代真实语境输入

### 17.3 今日任务的产品价值

回答结构:

1. 用户每天打开学习产品时,最大成本之一是"不知道先做什么"
2. 今日任务不是功能入口集合,而是任务编排层
3. 它把"继续学习、复习、表达沉淀和进度"重新组织成今天的优先任务
4. 推荐解释能降低用户不信任感

可以这样答:

> 今日任务的价值是降低启动成本。用户不需要每天重新判断要学新场景、继续上次中断、还是先复习。系统用服务端聚合状态把这些任务编排出来,并解释推荐原因。它不是内容生产层,而是学习闭环的调度入口。

### 17.4 表达管理为什么是资产层,不是收藏夹

回答结构:

1. 收藏夹只是保存,资产层要能维护、关联、复习和回流
2. 表达管理支持翻译、例句、用法说明、语义焦点、相似 / 对照表达、表达簇
3. 这些后续能被复习和今日任务消费
4. 资产化的目标是迁移,不是保存数量

可以这样答:

> 如果只是收藏夹,用户保存完就结束了。表达管理的定位是表达资产工作台,保存后的表达还能补全、关联、聚类和复习。它让一次场景学习产生的表达,变成后续可以继续维护和调用的个人资产。

### 17.5 复习和普通背卡片有什么区别

回答结构:

1. 普通卡片常围绕"记得 / 不记得"
2. 这个项目的复习更强调回忆、熟悉度、输出信心和完整输出
3. 复习不是孤立训练,它消费场景和表达管理的沉淀结果,也把信号写回后续调度
4. 队列清空后给收束反馈和下一步入口,避免训练结束后断掉

可以这样答:

> 复习不只是判断对错,而是把表达重新拉回主动回忆和输出训练。它的价值在于产生正式训练信号,再回流到今日任务和进度。这样复习不是一个孤立页面,而是学习闭环的一环。

### 17.6 怎么判断一个新需求该不该做

回答结构:

1. 先过产品北极星:是否强化回忆、使用、迁移
2. 再看是否补主链路短板:进入学习、表达沉淀、复习回流、状态承接
3. 再看成本和复杂度:是否需要改变数据模型、状态流、权限或缓存
4. 最后决定是立即做、降级做,还是暂缓

可以这样答:

> 我不会只看一个功能酷不酷,而是先问它是否让用户更容易在未来真实场景中回忆、使用或迁移表达。如果只是增加一个展示入口,但不进入沉淀、复习或回流链路,我会降低优先级。当前阶段更优先补今日编排、复习反馈、音频稳定性和表达资产维护。

### 17.7 核心指标会怎么设计

不要一上来讲日活。可以从学习闭环指标讲起:

- **进入**:今日任务首要点击率、继续学习点击率
- **学习**:场景完成率、场景中断后恢复率
- **沉淀**:每个场景保存表达数、表达补全率、表达簇组织率
- **回忆**:复习完成率、熟悉度提升、完整输出比例
- **回流**:复习后回今日任务的下一步点击率、次日继续学习率
- **稳定性**:整场景音频兜底率、预加载命中率、关键接口错误率

可以这样答:

> 我会把指标围绕闭环设计,而不是只看访问量。比如用户是否从今日任务进入学习,是否完成场景,是否保存并维护表达,是否回来做复习,复习信号是否让下一次推荐更准。技术指标也要配合产品指标,比如音频兜底率会直接影响听熟体验。

### 17.8 竞品已经有背单词、AI 对话,这个项目差异在哪

回答结构:

1. 背单词强在词汇覆盖,但弱在真实表达迁移
2. AI 对话强在即时互动,但生成内容如果不沉淀容易变成一次性体验
3. 这个项目的差异是把真实语境、表达资产、复习回流、每日编排和工程治理串起来

可以这样答:

> 我不把它定位成背单词工具,也不定位成单纯 AI 对话。它更像一个表达资产闭环:先在场景里遇到表达,再把表达沉淀下来,后续复习、回流和继续学习。AI 是生产和辅助能力,但核心价值是生成内容进入学习生命周期。

### 17.9 为什么现在不做完整 AI 口语对话

回答结构:

1. AI 口语对话很有价值,但如果没有表达资产和复习回流,容易变成一次性聊天
2. 当前阶段先建立"输入 → 沉淀 → 回忆 → 再进入"的基础闭环
3. 后续可以把口语对话作为输出训练层,消费已经沉淀的表达

可以这样答:

> 我不是认为 AI 口语不重要,而是它更适合建立在表达资产层之后。否则用户聊完一次,内容很难沉淀和复用。当前先把场景学习、表达资产和复习跑稳,后续口语对话可以作为更强的输出训练入口。

### 17.10 这个产品现在最大的产品短板

可以答得具体一点:

- 今日推荐还可以更个性化,目前更偏基于当前学习状态的任务编排
- 复习输出质量分析还不够深,还能进一步判断用户是否真的会迁移使用
- 表达管理是重型工作台,长期需要简化成更自然的表达维护路径
- 本地事件还只是本地回看,不是正式跨设备埋点和分析平台

一句话收束:

> 当前产品已经有闭环骨架,短板主要不在"有没有功能",而在推荐、反馈、资产维护和观测体系还可以继续细化。

## 18. 技术怎么服务产品目标

用于回答"你做的技术优化到底服务了什么产品目标"。

### 18.1 音频缓存服务什么

不要只答性能优化。更好的讲法:

> 音频缓存服务的是"听熟"和"重复学习"。英语场景学习里,用户会反复听同一句、同一个表达、同一个完整场景。如果每次都重新生成或等待远端,学习节奏会被打断。所以我做多层缓存、预加载和循环播放,本质是降低重复听的摩擦。

产品收益:

- 降低重复播放等待
- 支持碎片化回到同一场景
- 让循环复习更接近"离开页面也能继续听熟"的体验

### 18.2 服务端状态闭环服务什么

> 它服务的是"系统能接住用户"。用户今天中断了,明天打开还能继续;用户完成复习,后续推荐能消费这个结果。如果状态只在前端,产品很难做到长期回流。

产品收益:

- 今日任务能解释为什么推荐当前任务
- 复习能从真实学习行为里拿信号
- 进度能聚合长期表现
- 重复点击和网络重试不容易破坏学习事实

### 18.3 接口治理服务什么

> 接口治理不是纯后端洁癖,它服务的是"学习体验稳定"。TTS、AI 生成、批量导入都是高成本能力,如果没有限流、去重和错误追踪,用户会遇到重复提交、长时间等待、失败不可解释的问题。治理能力让产品从"能跑"变成"可持续维护"。

产品收益:

- 高成本接口不容易被重复请求打爆
- 用户数据访问边界更清楚
- 出问题时能通过请求编号和事件摘要定位

### 18.4 管理后台 / 可观测性服务什么

> 管理后台和可观测性的目标不是做后台花活,而是让这个学习产品可维护。音频失败、复习中断、继续学习点击这些事情,维护者看不到就很难判断用户在哪里掉出闭环。

## 19. 常见追问回答

### 19.1 这个项目最大的技术亮点

> 最大亮点是它不是单点页面项目,而是一个学习闭环型全栈应用。学习状态由服务端维护,表达能沉淀成资产,音频有生成、存储、缓存、预加载和循环播放链路,接口层也补了限流、去重、追踪和权限边界。

### 19.2 你最有把握讲深的是哪块

优先三块:

- 学习状态闭环:开始 / 进度 / 暂停 / 完成 如何支撑今日任务、复习、进度
- TTS 音频链路:文件存储、临时链接、浏览器本地缓存、预加载、循环播放
- 接口治理:限流、去重、请求编号、来源校验、数据库行级权限

### 19.3 循环播放为什么不是逐个音频播放

> 因为目标是后台或锁屏后尽量继续播放。逐个音频播放需要 JS 在上一段结束时切下一段,手机后台不稳定,而且还可能继续发网络请求。循环播放先把多个场景组成一个整体音频,再交给浏览器原生循环,连续性更好。

### 19.4 如果音频候选场景变化怎么办

> 当前策略是同一天稳定打包。当天候选变化不会频繁打乱已经准备好的资源;用户刷新或重新准备时可以生成新的打包。这样牺牲一点实时性,换缓存命中和播放稳定性。

### 19.5 这个项目还有哪些没做完

> 还没有做完整离线应用、平台级熔断、全面强制执行的内容安全策略和更细粒度的推荐模型。当前阶段重点是先把学习闭环、表达资产、音频链路和接口治理跑稳。

### 19.6 工程侧已知的几个 gap(被追问时主动讲反而加分)

下面三个是「做了一半 / 没做」但**有清晰下一步计划**的工程点。面试官问「还有什么没做完」时主动讲,比被追问出来好得多。

**1. `Retry-After` 标准 HTTP header 没设(只在 JSON body 里给)**

- 现状:429 响应只在 `body.details.retryAfterSeconds` 里给重试时间(代码 [src/lib/server/errors.ts](src/lib/server/errors.ts)),HTTP header 那一层缺。
- 影响:浏览器和中间层(nginx / fetch 内部重试)看不到标准 `Retry-After`,不能自动 backoff,得前端自己读 body 处理。
- 下一步:在 `toApiErrorResponse` 里把 `retryAfterSeconds` 同时写进 response header,客户端老逻辑不受影响,中间层能受益。
- 一句话讲:「我们 429 给了重试时间,但只在 body 里,标准 header 那一层还没补,这是个 5 分钟的活,先去补别的了。」

**2. `withClientActionTimeout` 只包了认证流,其他 fetch 没普及**

- 现状:登录 / 注册 / 发邮箱码 / 自动登录这几条客户端 fetch 都包了 15-20 秒超时(代码 [src/lib/client/action-timeout.ts](src/lib/client/action-timeout.ts) 是通用的),但 TTS / AI explain / scene parse / phrase save 这些还没套上。
- 影响:遇到上游卡住(比如 Edge TTS 长连接挂死、AI 接口不返回)时,前端可能转圈一直等。
- 下一步:把所有「会真的卡」的客户端 fetch 都套上 timeoutMs,然后 toast 一个友好提示。属于是「现成轮子,只是没批量铺开」。
- 一句话讲:「客户端超时这套机制写好了,只在认证流先验证了一遍,接下来要铺到 TTS、AI 这些更可能卡的接口。」

**3. AI 接口不是 SSE 流式响应**

- 现状:`/api/explain-selection` 是一次性 POST,等 AI 全部生成完再一起返回。没做 chunk 流式输出。
- 影响:首字时延等同于全文时延,UI 上「思考中…」要持续几秒。
- 为什么不做:当前业务里 AI 释义都是短文案(< 200 字),全文时延也就 1-3 秒,**首字时延的优化收益相对小**。更划算的是先把缓存命中率(§7.7 的 ai_cache)打上去,让大部分请求根本不走 AI,而不是优化单次 AI 调用的体验。
- 下一步:如果将来做长输出场景(比如「AI 生成一篇 500 字的解析」),才有意义切到 SSE。
- 一句话讲:「AI 接口我现在用一次性返回,因为业务里都是短文案,流式收益小。缓存命中率优先,真要做长输出场景再切 SSE。」

**讲这种 gap 的姿势:** 不要说「这个我没做」,要说「这个我现在是这样做的、为什么、下一步打算怎么演进」。**让面试官看到的是「我清楚自己的取舍和路线」,而不是「我没做完」。**

## 20. 简历版表达

### 20.1 项目描述

- 基于 Next.js 16 + React 19 + TypeScript + Supabase 搭建单仓全栈英语学习系统,覆盖今日任务、场景学习、表达资产、复习训练、进度聚合等核心模块
- 设计并落地服务端主导的学习状态闭环,支持学习进度回写、关闭浏览器后恢复、复习信号沉淀和进度聚合
- 实现 TTS 音频生成、文件存储持久化、浏览器本地缓存、空闲预加载、循环播放打包和播放兜底,降低重复请求、二次播放延迟和后台切歌失败概率
- 建设接口治理基线:请求编号追踪、统一错误响应、服务端注册准入、邮箱验证码与邮箱验证闭环、高成本接口限流和每日额度、关键写接口去重、来源校验、用户数据权限边界收紧
- 建设可观测性与上线治理:接入 Sentry 自动捕获 5xx 并注入请求编号标签,启动期检查 10 个关键配置,落地内容安全策略上报模式,搭建 GitHub Actions 自动检查工作流,产出灾备 / 事故响应剧本和可直接套用的 Nginx 反向代理 + 限流模板
- 主导接口错误响应一致性收口:先对 58 个接口做盘点分级,再一次性收口高优问题,全量错误响应统一为"错误码 + 详情 + 请求编号",7 处高成本接口补上按业务模块打日志的能力
- 主导超重页面多轮拆分:表达管理页经三轮拆分(2368 → 2041 行)、场景详情页经一轮(1326 → 849 行,-36%);第二轮收口后建立"装配开销 vs 抽走开销"量化算账方法,把"高属性子树优先"策略写进规范指导第三轮,第三轮减幅是第二轮的 3 倍

### 20.2 优化点表达

- 通过场景预加载、列表 / 详情缓存回填和空闲资源预热,优化场景进入速度与重复访问体验
- 通过音频网址缓存、浏览器本地缓存、播放地址复用和同日稳定循环打包,减少 TTS 重复生成、远端请求和后台播放切歌依赖
- 通过共享限流、注册 IP 频率控制、每日额度、管理员紧急开关、去重和统一日志追踪,提升公开注册与高成本 AI 接口的稳定性与可排障性
- 通过用户数据查询和写入逐步切换到数据库行级权限模型,提升后端最小权限边界与安全纵深
- 通过 Sentry 5xx 聚合 + 启动期自检 + 内容安全策略上报 + GitHub Actions 自动检查,把事故响应从"等用户截图"降到分钟级,把配置漂移检测前置到部署后第一时间
- 通过复习调度消费 5 类正式信号(识别状态、输出信心、完整输出状态、变体改写状态、完整输出覆盖度),让复习节奏按"在哪个训练深度卡住"细调,不只靠"再次 / 困难 / 良好"三档
- 通过跨场景取消 + 失败冷却阶梯 + 共享存储签名链接缓存,避免后台空跑、反复打上游、多进程命中率打折三类高成本资源失败放大问题
- 通过接口错误响应契约一次性收口,全量错误响应带"错误码 + 详情 + 请求编号",前端能稳定解析,后端高成本路径自动按模块进入错误监控分桶
- 通过进程级窄白名单守卫修复第三方 TTS 库 WebSocket 回调时序竞态导致的未捕获异常,把生产 102 次进程重启降到 0,守卫包含 6 条单元测试覆盖开发栈 + 生产折叠栈两种符号场景
