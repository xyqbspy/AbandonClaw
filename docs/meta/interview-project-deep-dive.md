# AbandonClaw 面试项目深挖稿

## 1. 文档目标

这份文档是给面试和项目讲解用的，不替代正式技术规则，也不替代详细实现文档。

它要解决三个问题：

- 面试前快速想起项目里真正值得讲的技术点。
- 面试时能把“为什么这样设计”讲清楚，而不是只报技术栈。
- 被追问时能说出实现链路、工程取舍和后续优化方向。

如果只想看项目技术现状，读 `docs/meta/technical-overview.md`。如果要背、要讲、要准备追问，读这份文档。

## 2. 30 秒项目介绍

这是一个基于 `Next.js 16 + React 19 + TypeScript + Supabase` 的英语学习全栈应用。

它不是普通内容展示站，而是围绕 `Today -> Scene -> Chunks -> Review -> Progress` 做学习闭环：`Scene` 提供真实语境，`Chunks` 沉淀表达资产，`Review` 做回忆训练，`Today` 把进度、复习和继续学习重新编排成每日入口。

我做得比较重的部分包括：学习状态闭环、表达资产模型、TTS 音频链路、前端缓存与预取、高成本接口治理、公开注册准入与管理员处置、权限边界收紧，以及 AI 能力接入真实学习流程。

## 3. 2 分钟架构介绍

项目采用单仓全栈结构。页面、API route、服务端聚合逻辑、缓存逻辑和数据库迁移都在同一个仓库里，核心目录大致是：

- `src/app`：App Router 页面和 API route。
- `src/features`：按业务域组织的页面级组件和交互组件。
- `src/components`：公共 UI 和通用组件。
- `src/lib/server`：服务端业务逻辑、聚合逻辑、权限与数据访问。
- `src/lib/cache`：前端缓存、预取和回填。
- `supabase/sql`：数据库结构、RLS、policy 和阶段性迁移。

我会这样概括架构思路：

> 页面层尽量薄，业务判断下沉到 feature logic / selectors / controller，跨页面状态和聚合逻辑尽量由服务端承接，前端缓存负责体验兜底，Supabase 承担认证、数据库和对象存储。

## 4. 为什么这样设计

### 4.1 为什么用 Next.js + Supabase 单仓全栈

这个项目处在产品闭环快速成型阶段，学习主链路、复习、表达资产、音频和 AI 能力都需要频繁联动。单仓全栈的收益是：

- 页面和 API 协议能一起演进，减少前后端分仓联调成本。
- 服务端聚合逻辑可以靠近页面入口，例如 Today、Progress、Review。
- Supabase 直接提供 Auth、Database、Storage，适合先把闭环跑通。
- 数据迁移、RLS 和业务代码能在同一个仓库里追踪。

它不是最终的大平台架构。后续如果 TTS、推荐、AI 生成变成独立高负载能力，可以再拆成独立服务。

### 4.2 为什么不用重型前端状态管理

当前项目的核心状态不是一个纯前端大 Store 能解决的。学习进度、复习信号、短语资产、音频资源都需要跨页面、跨会话、跨设备持续存在。

所以更合理的方式是：

- 页面内交互用 React state 和局部 hook。
- 派生展示逻辑放在 selector / logic。
- 可恢复、可聚合、可审计的学习状态放服务端。
- 列表和详情体验用本地 cache 回填。

这样比把所有东西塞进 Redux 更贴合业务。

### 4.3 为什么页面要薄

项目有一个很重要的工程约定：页面负责编排，不吞业务。

例如 `scene/[slug]` 不是一个普通详情页，它会组合数据加载、路由视图、学习同步、音频播放、表达保存、练习生成和变体学习。把这些都写进一个 page 会很快失控。

所以项目把复杂度拆到：

- `selectors`：派生展示数据。
- `logic`：纯判断和状态转换。
- `controller`：动作编排。
- `actions`：数据构造和副作用入口。
- `components`：展示和交互。

面试时可以说：这个项目不是按技术名词堆文件，而是按页面职责拆复杂度。

## 5. 技术亮点一：学习状态闭环

学习不是前端点一下按钮就算完成，而是服务端维护完整状态机。

核心动作包括：

- `start`：进入场景时开始学习。
- `progress`：停留、阅读、切换时增量回写。
- `pause`：离开页面时暂停。
- `complete`：完成练习或变体学习后推进完成态。

这套设计带来的价值是：

- 支持 session 恢复。
- 支持 Today 继续学习。
- 支持 Progress 聚合统计。
- 支持 Review 从真实学习行为里拿信号。
- 学习进度不是临时 UI 状态，而是可长期沉淀的数据。

如果面试官问“这和普通学习页面有什么区别”，可以答：

> 普通页面只记录用户当前看到哪里；这个项目把学习行为做成服务端状态闭环，后续 Today 推荐、Review 训练、Progress 统计都可以消费同一份状态。

## 6. 技术亮点二：表达资产模型

这个项目不是只让用户读完一个场景，而是把场景里的表达沉淀成资产。

表达资产大致经历：

1. 用户在 `Scene` 中遇到真实表达。
2. 保存为 phrase / chunk。
3. 在 `Chunks` 中补全翻译、例句、usage note、semantic focus。
4. 生成 similar / contrast 表达。
5. 组织进 expression cluster。
6. 进入 Review 和 Today 的长期循环。

这里值得讲的是：`Chunks` 不是收藏夹，而是表达资产工作台。它让“学过一个场景”变成“积累一组可复用表达”。

面试表达：

> 我把表达从一次性内容消费里抽出来，做成可以补全、聚类、复习和迁移的资产层。这样 Scene 是输入入口，Chunks 是资产沉淀层，Review 是长期回忆训练。

## 7. 技术亮点三：TTS 音频链路

音频链路是项目里很值得讲的一块，因为它不只是“请求一个音频 URL”。

完整链路大致是：

1. 服务端根据文本或 segments 生成 TTS。
2. 音频上传到 Supabase Storage。
3. 服务端返回签名 URL。
4. 客户端做 URL 缓存。
5. 浏览器 Cache Storage 异步持久化音频响应。
6. 后续播放优先命中本地缓存。
7. 弱网或签名 URL 过期时尽量复用本地资源。

这个设计兼顾了：

- 首次播放可用。
- 二次播放更快。
- 重复 TTS 生成减少。
- 弱网下有兜底。
- 高成本音频接口不被频繁打爆。

## 8. 技术亮点四：场景循环播放 review pack

这是最近做的音频后台播放优化，面试时可以讲，但要讲准。

它不是 ffmpeg 音频文件拼接，也不是后台逐段请求、逐段切歌。真正做法是 segments 级组包：

1. 从已学场景里筛出达到熟悉度门槛的候选。
2. 用同日稳定顺序取一个小窗口，保证预热和点击播放命中同一份资源。
3. 读取每个场景的 scene detail cache 或接口详情。
4. 把每个 lesson 通过 `buildSceneFullSegmentsFromLesson()` 转成 scene full segments。
5. 合并多个场景的 segments，形成一个 review pack payload。
6. 复用现有 scene full TTS 通道生成一个整体音频。
7. 客户端用单个 `<audio loop>` 播放这个整体音频。

为什么这么做：

- 移动端后台或锁屏后，页面 JS 不一定能稳定切下一首。
- 如果逐段请求，后台播放会强依赖网络和 JS 调度。
- 生成一个整体音频后，浏览器原生音频播放链路更容易持续播放。
- 同日稳定 pack 能提高缓存命中率，避免每次点击都生成新资源。

取舍也要讲清楚：

- 优点：后台播放更稳定，缓存命中更稳定，请求更少。
- 代价：播放过程中不实时切换“当前场景名称”，候选变化后需要准备新 pack。
- 当前不是完整离线音频系统，也不是增量热插入正在播放音频末尾。

一句话讲法：

> 我没有做文件级拼接，而是把多个场景先转换成同一种 scene full segments，再复用现有 TTS 通道生成一个可循环播放的整体音频，用稳定性换掉后台 JS 切歌的不确定性。

## 9. 技术亮点五：缓存与预取策略

项目里缓存不是单点优化，而是围绕学习体验做了多层兜底。

主要包括：

- Scene 列表缓存。
- Scene 详情缓存。
- Review 页面缓存。
- Learning dashboard 缓存。
- Phrase list 缓存。
- TTS URL 缓存。
- 浏览器 Cache Storage 音频缓存。
- 空闲时预取和预热。

典型策略可以概括为：

> cache first + network refresh + idle prefetch + weak-network guard

也就是先让用户尽快看到可用内容，再异步刷新网络数据；重资源放到空闲时预热；弱网、省流量场景下抑制重预热。

面试可以强调：这不是为了炫技做缓存，而是学习产品的访问很碎片化，用户会反复进入同一个场景、同一个表达、同一段音频，所以缓存收益很直接。

## 10. 技术亮点六：接口治理与权限边界

项目已经不是裸 API 阶段，已经补了基础治理：

- `requestId`：方便串联一次请求的日志。
- 统一错误响应：前端和排障更稳定。
- 高成本接口限流：保护 TTS、AI 生成、导入等接口。
- daily quota、调用前预占和后台紧急开关：异常成本能在上游调用前止损。
- 服务端注册准入：注册模式、邀请码、邮箱验证码、邮箱验证回调、重发验证邮件和注册 IP 频控都在服务端收口。
- 管理员处置入口：用户状态降级、高成本能力开关、邀请码和注册模式管理都能在后台操作。
- 幂等去重：保护关键写接口和重复点击。
- Origin 校验：保护受保护写接口。
- RLS 承接：用户态数据逐步回到数据库最小权限模型。

这里可以讲成“从功能可用走向可治理”：

> 早期项目最重要的是闭环跑通；到现在已经开始补限流、幂等、追踪、权限边界和公开注册准入这些工程化能力，让高成本 AI 接口、用户态数据访问和小范围开放都更可控。

## 11. 技术亮点七：AI 能力不是孤立 demo

项目里 AI 不是单独做一个聊天框，而是嵌入学习闭环：

- 场景生成。
- 表达解释。
- 表达补全。
- 相似 / 对照表达生成。
- 场景练习生成。
- 场景变体生成。
- TTS 音频生成。

面试时可以这样讲：

> 我更关注 AI 输出进入业务后的生命周期。生成内容只是第一步，更重要的是它能不能进入学习、保存、复习、统计和再次推荐。

## 12. 最值得讲的工程取舍

### 12.1 不做大而全后端拆分

当前先用 Next.js API route + Supabase 跑通闭环。这样开发速度快，数据和页面联动成本低。等到 TTS、推荐、AI 生成负载明显变大，再拆服务更合理。

### 12.2 不引入重型状态管理

因为真正需要长期保存的是学习状态和表达资产，不是纯前端 UI 状态。把长期状态放服务端，比堆一个全局 Store 更有价值。

### 12.3 不做 ffmpeg 音频拼接

review pack 需要的是后台播放稳定性，不是音频编辑能力。segments 级组包能复用已有 TTS 通道，复杂度更低，和现有缓存链路更一致。

### 12.4 不追求全量离线

当前做的是缓存兜底和二次播放加速，不是 PWA 级离线系统。这样能先解决高频体验问题，避免把维护复杂度拉太高。

## 13. 面试官深挖追问树

这一节按“面试官继续往下问”的方式组织。准备时建议不要死背整段，而是记住每个问题的回答结构：先讲问题约束，再讲当前方案，再讲取舍边界。

### 13.1 如果问：学习状态为什么要放服务端

回答结构：

1. 先说明业务约束：学习进度会影响 Today 推荐、Review 队列、Progress 聚合，不是当前页面 UI 状态。
2. 再说明状态动作：start / progress / pause / complete / practice / variant 都是服务端事实。
3. 再说明工程收益：刷新可恢复、跨页面可消费、重复提交更可控、后续能审计和推荐。
4. 最后讲边界：页面内播放、展开、临时选择仍留在前端，不做重型全局 store。

可以这样答：

> 我把学习状态分成两类：只影响当前页面体验的状态留在 React，本轮学习是否开始、推进、暂停、完成这类会影响后续 Today、Review 和 Progress 的事实放到服务端。这样刷新页面、回到 Today 或做 Review 时，消费的是同一份服务端状态，而不是某个页面临时记住的状态。

可展开实现：

- API 入口：`/api/learning/scenes/[slug]/start|progress|pause|complete`
- 服务端聚合：`src/lib/server/learning/service.ts`
- 下游消费：`learning dashboard`、`today continue`、`review signals`

容易被继续追问的点：

- 重复点击怎么办：关键写操作有幂等 key 和短时间结果复用。
- 网络失败怎么办：前端不把临时成功当最终事实，核心状态以后端响应为准。
- 为什么不用 Redux：Redux 只能解决前端共享状态，不能解决跨会话、跨设备和可审计。

### 13.2 如果问：TTS 为什么要做这么多层缓存

回答结构：

1. 先说明成本：TTS 是高成本、相对慢、用户会反复播放的资源。
2. 再说明 key：chunk / sentence / scene full 各自有稳定 key，避免混用。
3. 再说明服务端：先查 Storage 和签名 URL 缓存，未命中才生成。
4. 再说明客户端：Cache Storage、内存 URL、in-flight、`blob:` URL 复用。
5. 最后讲边界：不是完整离线，首次未缓存仍要联网。

可以这样答：

> 音频不是普通 JSON 数据。它生成慢、成本高，而且学习场景里会重复播放。所以我把它做成稳定资源：服务端按内容生成 key，优先命中 Storage；客户端优先读 Cache Storage，签名 URL 过期后也可以继续用本地 Blob。这样二次播放不再强依赖远端接口。

可展开实现：

- key 规则：`src/lib/shared/tts.ts`
- 生成与 Storage：`src/lib/server/tts/service.ts`
- 客户端缓存与播放：`src/lib/utils/tts-api.ts`
- 预热调度：`src/lib/utils/audio-warmup.ts`、`src/lib/utils/resource-actions.ts`

容易被继续追问的点：

- 签名 URL 过期怎么办：浏览器 Cache Storage 存的是音频响应，命中后转 `blob:` URL 播放。
- 缓存会不会无限长：Cache Storage 有条目数和总大小上限，会裁剪旧条目。
- 为什么还要内存缓存：减少同一会话内重复请求和重复创建播放 URL。

### 13.3 如果问：review pack 为什么是 segments 组包

回答结构：

1. 先说明目标：不是做音频编辑，而是提高后台/锁屏播放连续性。
2. 再说明问题：逐段播放依赖 JS 在上一段结束时切下一段，后台不稳定。
3. 再说明方案：同日稳定候选 -> scene full segments -> 合并 payload -> 复用 TTS 通道生成整体音频。
4. 再说明失败处理：pack 失败回退逐场景队列。
5. 最后讲代价：不实时切换当前场景名，不做增量热插入。

可以这样答：

> 我没有做 ffmpeg 拼接，因为需求不是音频剪辑，而是让后台播放更稳定。review pack 在播放前把多个已学场景转成同一种 scene full segments，再复用现有 TTS 通道生成整体音频。播放开始后交给浏览器原生 audio loop，减少后台 JS 切歌的不确定性。

可展开实现：

- 组包与回退：`src/app/(app)/scenes/use-scene-random-review-playback.ts`
- segments 构建：`buildSceneFullSegmentsFromLesson()`
- 播放能力：`playSceneLoopAudio()` / `playSceneFullAudioOnce()`

容易被继续追问的点：

- 候选变了怎么办：同日稳定 pack 优先，刷新或重新准备时生成新 pack。
- 为什么不是随机：完全随机会降低预准备命中率，同日稳定顺序能让预热和点击播放指向同一资源。
- 为什么不是完整离线：当前目标是缓存兜底和播放连续性，不做 PWA 级离线承诺。

### 13.4 如果问：接口治理具体保护了什么

回答结构：

1. 先分三类风险：公开注册入口、高成本读/生成接口、关键写接口。
2. 注册入口用模式开关、邀请码、邮箱验证码、邮箱验证和 IP 频控保护。
3. 高成本接口用 user/IP 限流、daily quota、预占和后台紧急开关保护。
4. 关键写接口用幂等保护：学习推进、review submit、phrase save。
5. 排障用 requestId、统一错误响应、结构化日志串联。
6. 权限边界用 Origin 校验、账号状态和 RLS 逐步收紧。

可以这样答：

> 我不是把接口治理当成一个统一中间件口号，而是按风险补。注册入口要有模式、邀请码、邮箱验证码、邮箱验证和 IP 频控；TTS 和 AI 生成这类高成本入口要限流、quota 和紧急开关；学习推进、复习提交、短语保存这类写操作要幂等；所有错误响应带 requestId，方便从前端错误、route 日志和 service 日志串起来。

可展开实现：

- `src/lib/server/rate-limit.ts`
- `src/lib/server/idempotency.ts`
- `src/lib/server/api-error.ts`
- `src/lib/server/request-context.ts`
- `src/lib/server/request-guard.ts`

容易被继续追问的点：

- 限流存哪里：共享限流可用时走 Upstash Redis，不可用 fallback 到本地内存。
- 邮箱验证码是不是自建：注册页 6 位验证码是项目内最小闭环；账号体系、邮件链接确认、callback、重发入口和未验证拦截仍复用 Supabase Auth。
- 幂等是不是持久化：当前是轻量内存方案，适合阶段性保护；更高并发可迁到共享存储。
- RLS 做到什么程度：主要用户态表逐步承接，仍不是完整平台化权限系统。

### 13.5 如果问：AI 能力怎么避免变成 demo

回答结构：

1. 先说明项目不是聊天框，而是学习闭环。
2. AI 输出必须进入业务生命周期：学习、保存、练习、复习、统计、推荐。
3. Prompt 只是入口，后面还有校验、结构化、落库和 UI 承接。
4. 高成本 AI 接口要进入限流、错误追踪和 fallback 体系。

可以这样答：

> 我更关注 AI 输出进入业务后的生命周期。比如生成一个场景不算结束，它要能变成可学习的 scene；生成一个 similar expression 也不算结束，它要能进入表达资产、cluster 和 review。AI 在这里是内容和训练能力的生产方式，不是孤立 demo。

可展开实现：

- prompts：`src/lib/server/prompts/*`
- 场景生成：`src/app/api/scenes/generate/route.ts`
- 表达补全：`src/app/api/phrases/manual-assist/route.ts`
- 相似表达：`src/app/api/phrases/similar/generate/route.ts`
- 练习生成：`src/app/api/practice/generate/route.ts`

### 13.6 如果问：这个项目最能体现工程能力的地方

不要只答“用了 Next.js 和 Supabase”。更有说服力的回答是：

> 我觉得最能体现工程能力的是：我没有只做功能页面，而是把学习状态、表达资产、音频资源、AI 生成和接口治理放到同一个闭环里处理。比如一个场景不是展示完就结束，它会产生学习状态、表达资产、复习信号和音频资源；这些又会回到 Today 和 Progress。技术设计服务的是这条闭环，而不是单点功能炫技。

可以补三点：

- 复杂状态放服务端，页面保持薄。
- 高成本资源做缓存、预热、失败兜底。
- 有明确边界，不提前做大平台、不做完整离线、不做重型状态管理。

### 13.7 如果问：你怎么证明这些不是写在简历上的空话

回答结构：

1. 先把能力落到用户链路，而不是先报代码文件。
2. 再说对应的工程入口、服务层、缓存层或测试。
3. 最后说明验证方式：单元测试、交互测试、baseline 或文档约束。

可以这样答：

> 我会按链路证明。比如 TTS 不是一句“做了缓存”，而是从 `/api/tts` 进入服务端生成和 Storage 命中，再到客户端 `requestTtsUrl()` 先查 Cache Storage、内存缓存和 in-flight，最后通过播放事件和 admin observability 看失败摘要。学习状态也不是一句“服务端管理”，而是有 start/progress/pause/complete 入口，下游 Today、Review、Progress 消费同一份状态。

可展开证据：

- 技术链路证据：route -> service -> cache / repo -> UI hook。
- 测试证据：`*.test.ts` 验证服务逻辑，`*.test.tsx` 验证交互链路。
- 文档证据：system-design 记录音频、缓存、失败兜底边界。

### 13.8 如果问：你遇到过最难的技术问题是什么

优先讲“后台音频播放稳定性”，因为它能体现约束分析和取舍。

回答结构：

1. 问题：移动端后台或锁屏后，页面 JS 不一定稳定执行下一段切换。
2. 初始直觉：逐个 scene full 播放，但它依赖 `onended -> 切下一段 -> play()`。
3. 方案取舍：不用 ffmpeg 文件拼接，而是 segments 级组包，复用已有 TTS 通道。
4. 结果：播放开始后尽量由单个整体音频承接，缓存命中更稳定。
5. 边界：不是完整离线，不承诺所有浏览器后台启动新音频。

可以这样答：

> 最难的不是把音频播放出来，而是让它在移动端后台或锁屏场景下尽量稳定。逐段播放看起来简单，但实际依赖 JS 在每段结束后继续调度。我的方案是把多个已学场景提前转成统一的 scene full segments，生成一个整体 review pack，再用浏览器原生 audio loop 播放。这个方案牺牲了一点实时性，但换来了更稳定的连续播放和缓存命中。

### 13.9 如果问：项目里有哪些地方你刻意没做

这类问题要讲成熟度，不要只说“时间不够”。

可以答四个点：

- 没做完整微服务拆分：当前闭环成型优先，Next.js API route + Supabase 联动成本更低；等 TTS / 推荐 / AI 生成负载独立变大，再拆更合理。
- 没做完整 PWA 离线：当前先解决二次播放、缓存兜底和弱网体验；完整离线会引入同步、版本和容量治理复杂度。
- 没做重型状态管理：长期状态应该在服务端，前端大 store 不能解决跨会话和可审计问题。
- 没做 ffmpeg 拼接服务：review pack 的目标是播放连续性，不是音频编辑；segments 组包更贴合现有 TTS 与缓存链路。

一句话收束：

> 我没有把项目往“大而全平台”方向提前推，而是优先把学习闭环里最影响稳定性和可维护性的点做实。

### 13.10 如果问：如果再给你两周会优化什么

回答要围绕现有边界，不要突然开一个全新产品方向。

优先级可以这样排：

1. 强化观测链路：把当前本地 client-events 中最关键的失败摘要升级为更正式的服务端事件或日志采样。
2. 细化 TTS 指标：区分 block / scene full 的 warm hit、cold hit、fallback rate，用数据判断预热是否值得。
3. 收紧 API 契约：把高频 route 的输入输出 schema 和错误结构继续稳定下来，为 OpenAPI 平台化做准备。
4. 完善权限边界：继续把用户态访问收回 `createSupabaseServerClient + RLS`。
5. 优化 review 调度：在已有信号上增强熟悉度、输出信心和完整输出的权重。

不要优先说：

- 直接重写成 Java / 微服务。
- 立刻上完整 BI 平台。
- 立刻做全量离线。
- 立刻引入复杂推荐模型。

这些可以作为长期方向，但不是当前最小高收益路径。

## 14. 技术细节深挖卡

这一节用于把“我做了某个技术点”继续往下讲到实现层。面试时不一定全讲，但被追问时可以从这里抽一条链路展开。

### 14.1 TTS 音频 key 怎么保证复用边界

可以这样讲：

> 音频缓存不是按 URL 粗暴缓存，因为签名 URL 会过期。项目先把音频抽象成稳定资源 key，再让服务端 Storage、浏览器 Cache Storage 和播放层都围绕同一套 key 工作。

关键细节：

- chunk 音频：按清洗后的表达文本生成 key，适合跨页面复用。
- sentence / block 音频：把 `sentenceId`、speaker、mode 和文本算入指纹，避免同一句在不同语速或说话人下混用。
- scene full 音频：先做 segments 合并，再按整段内容和 scene type 算 hash，任何内容变化都会换 key。
- 浏览器 Cache Storage 使用稳定 cache key，而不是直接依赖短期签名 URL。

被追问时补一句：

> 所以“二次播放更快”不是因为 URL 没过期，而是因为真正复用的是稳定音频资源和浏览器本地 Blob。

### 14.2 `/api/tts` 的生成链路怎么讲

回答结构：

1. route 先做用户态校验。
2. service 校验 payload，解析 kind / mode / text / segments。
3. 先查服务端进程内签名 URL 缓存。
4. 再查 Supabase Storage 是否已有对象。
5. Storage 命中直接返回 signed URL。
6. 未命中才用 `msedge-tts` 生成 Buffer，上传 Storage。
7. Storage 上传失败时退回 inline fallback，保证首次播放仍有兜底。

可以这样答：

> TTS 链路是“先命中已有资源，再生成”。这能避免用户每次点播放都触发合成。运行时真正复用的是 Supabase Storage，本地 `public/audio` 只是开发或兼容副产物，不作为生产主缓存判断依据。

边界要讲清楚：

- 服务端签名 URL 缓存是进程内缓存，不是跨实例强一致缓存。
- Storage 是否存在当前主要通过创建 signed URL 判断，规模变大后可以优化为对象元信息查询。
- inline fallback 是兜底，不是主路径。

### 14.3 客户端 TTS 缓存为什么分四层

四层可以按“减少什么成本”来讲：

- 内存 URL 缓存：减少同一会话重复请求 `/api/tts`。
- 浏览器 Cache Storage：绕过 signed URL 时效，保存音频 Blob。
- 预加载状态集合：避免同一个 URL 反复 `Audio().load()`。
- Blob URL 复用表：避免同一条缓存反复 `createObjectURL`。

可以这样答：

> 客户端不是简单 cache-first，而是按资源生命周期拆层。URL 缓存解决短期重复请求，Cache Storage 解决签名 URL 过期后的本地复用，Blob URL 复用解决播放层的对象 URL 重复创建。

可继续展开：

- Cache Storage 有条目数和总大小上限，不是无限增长。
- 新音频写入后会按旧条目裁剪，裁剪失败不阻塞播放。
- 重生成 chunk 音频前会先停止播放、清浏览器缓存，再重新预取。

### 14.4 scene detail 为什么做 block-first 音频

可以这样讲：

> scene detail 的主消费单元从单句调整成 block，是为了更贴近用户真实听一段对话或一个语义块的方式。技术上它仍复用 sentence TTS 通道，不新增 `/api/tts` 协议，避免为了 UI 颗粒度变化扩接口。

关键细节：

- block 音频复用 `sentence` TTS kind。
- block 的 `sentenceId` 使用 `block-${block.id}`。
- block 文本优先来自 `block.tts`，没有时拼接 block 内句子的 `tts / audioText / text`。
- observability 事件用 `audioUnit = block` 区分“复用 sentence 通道”和“真实 sentence 音频”。

取舍：

- 优点：用户点击更贴近段落听熟；接口不新增协议分支。
- 代价：调试时必须看 `audioUnit`，否则容易误以为主链路仍在播放单句。

### 14.5 scene full 失败为什么不自动逐句串播

回答结构：

1. scene full 是完整场景音频，失败时可能来自 provider、timeout、segments、Storage、signed URL 等多种原因。
2. 自动逐句串播会引入新的播放状态机，尤其涉及后台播放、暂停、切换和错误续播。
3. 当前最小可维护方案是受控错误 + 失败摘要 + 45 秒冷却 + 显式 CTA 回到 block / sentence。

可以这样答：

> 我没有把 scene full 失败自动降级成逐句串播，因为这会把一个生成失败问题扩大成新的播放器状态机问题。当前方案是识别失败原因，短时冷却避免重复打上游，再给用户明确的逐句跟读入口。这样能兜住体验，又不引入难维护的自动串播逻辑。

可展开失败原因：

- `provider_error`
- `timeout`
- `segment_assembly_failed`
- `storage_upload_failed`
- `signed_url_failed`
- `empty_audio_result`
- `cooling_down`

### 14.6 预热收益怎么观测

可以这样讲：

> 我不是只做预热，还做了轻量的 warmup registry，把“预热入队”和“后续播放事件”用稳定 cache key 关联起来。这样能看 warm hit 和 cold hit 的差异，而不是凭感觉判断预热有没有用。

关键细节：

- registry key 与 Cache Storage key 一致。
- 来源区分 `initial` / `idle` / `playback`。
- `user-click` 不算 warmup source，因为用户点击是 demand，不是预热收益。
- Admin summary 基于本地 client-events 计算，不代表全量用户数据。

可以补一句边界：

> 样本少时 warmup gain 可能为负，这不是自动失败，只是说明当前调度需要继续观察。

### 14.7 scene 练习题生成链路怎么讲

回答结构：

1. scene detail 先把 `Lesson` 转成可供出题的 `ParsedScene`。
2. `/api/practice/generate` 让 AI 基于 sections / blocks / sentences / chunks 出题。
3. AI 返回结构不合法时，回退本地 `buildExerciseSpecsFromScene`。
4. `normalizePracticeExercisesForScene` 把 cloze 的题面和答案对齐。
5. 前端最终组装成四层模块：`cloze -> guided_recall -> sentence_recall -> full_dictation`。
6. 前一模块全部 typing 题答对后，下一模块才解锁。

可以这样答：

> 练习生成不是模型返回什么就直接展示。AI 负责提供题目候选，本地还会做结构校验、fallback、cloze 标准化和四层训练模块编排。这样能兼顾生成灵活性和前端训练流程稳定性。

可展开取舍：

- 首轮 cloze 当前更偏保守，最多围绕 5 到 8 题。
- fallback 会优先挑高价值 chunk，但不会把同一句拆成很多碎空。
- guided recall 会过滤太短的句子，避免半句复现没有训练意义。

### 14.8 practice set 为什么要服务端持久化

可以这样讲：

> 练习题不能只存在本地缓存，否则用户刷新、换设备或提交 attempt 时，服务端无法确认这套题是否属于当前用户和当前 scene。现在 practice set 本体会保存到服务端，本地缓存只负责秒开和失败降级。

关键细节：

- `/api/practice/generate` 只负责生成题目，不负责用户态落库。
- scene detail 生成或重新生成后，会调用 `/api/learning/scenes/{slug}/practice/set` 保存。
- `user_scene_practice_sets` 保存用户级 practice set。
- run / attempt 写入前会校验 `practiceSetId` 属于当前 user + scene。
- 手动重新生成会创建新的 practice set，并把旧 generated set 标记为 abandoned。

边界：

- 当前不做公共题库。
- 不做跨用户共享题集。
- 不把 exercises 拆成标准化公共表。

### 14.9 Chunks 保存链路怎么讲清楚

可以这样讲：

> Chunks 的复杂点不在 UI，而在保存动作会同时影响 phrase、user phrase、relation、cluster、daily stats 和 review 入口。所以项目把 payload 拼装、后端写入语义和 cluster 操作拆开，避免每个入口各自拼一套规则。

关键实体：

- `phrases`：标准化表达实体。
- `user_phrases`：用户维度学习条目，区分 expression / sentence。
- `user_phrase_relations`：维护 similar / contrast。
- `user_expression_clusters`：表达簇。
- `user_daily_learning_stats`：当日保存统计。

关键边界：

- `phrases/service.ts` 负责 phrase upsert、user phrase、relation、cluster 同步判断、stats。
- `expression-clusters/service.ts` 负责 ensure / set main / merge / move / detach。
- contrast 只写对照关系，不应并入 similar cluster。
- similar 可以进入同类 cluster。

被问“为什么复杂”时可以答：

> 因为表达资产不是一条收藏记录。它后续要支持相似表达、对照表达、cluster、review 和统计，所以保存时必须把关系和资产结构一起维护。

### 14.10 接口分层怎么回答得更具体

可以按这条链路讲：

- route：鉴权、参数解析、调用 service、返回统一响应。
- handler：部分模块把 route 里的动作抽成可测试 handler。
- service：业务编排、状态推进、跨表副作用。
- repo / repository：数据访问和 Supabase 查询。
- shared / logic：纯函数、选择器、规范化和可单测规则。

可以这样答：

> 我没有做非常重的 DDD，但也避免 route 里堆所有逻辑。route 主要处理 HTTP 边界，service 承担业务语义，repo 处理数据访问，shared logic 放纯函数。这样测试时能绕开 HTTP 直接测 handler / service / logic，也能在需要时补真实 HTTP baseline。

### 14.11 测试怎么证明链路不是手测

回答结构：

1. 纯规则用 `*.test.ts`，例如 prompt builder、service logic、cache key、idempotency。
2. 页面交互用 `*.test.tsx`，例如 scene detail、chunks、review、pull-to-refresh。
3. 高风险入口用 route handler test，验证错误结构和 requestId。
4. 维护类脚本检查 mojibake、OpenSpec、guardrails。
5. 不是默认跑全量，而是按改动风险跑最小相关集。

可以这样答：

> 我会按风险选测试，不是每次无脑全量。改 TTS 就跑 tts-api、audio-warmup、server tts、regenerate route；改 scene 练习就跑 practice generate、scene practice view、practice set；改 chunks 保存就跑 save contract、manual composer、focus assist 和 page interaction。

面试时可以强调：

> 测试策略和架构拆分是配套的。逻辑拆到 service / shared 后，才能用轻量 Node test 覆盖核心语义；交互测试只覆盖真正需要 DOM 的路径。

## 15. 分模块深挖准备卡

这一节用于面试前快速过一遍：每个模块讲什么、落到哪里、边界是什么。

### 15.1 Today

核心讲法：

> Today 不是入口列表，而是把继续学习、review、表达沉淀和进度聚合重新编排成每天的优先任务。

可展开：

- 为什么推荐这一步。
- 如何承接 session 恢复。
- 如何把 review / progress 的结果重新组织成下一步入口。

边界：

- 当前不是复杂推荐系统。
- 个性化仍有继续增强空间。

### 15.2 Scene

核心讲法：

> Scene 是主学习工作台，承载真实语境、音频、表达提取、练习和变体学习。

可展开：

- 页面层负责编排，复杂判断下沉到 selectors / logic / controller / hooks。
- 学习状态通过服务端 start / progress / pause / complete 推进。
- 音频从 block-first、scene full、sentence fallback 分层承接。

边界：

- scene full 失败后不自动逐句串播，只给受控提示和替代入口。
- 不把所有交互状态都写服务端。

### 15.3 Chunks / Phrases

核心讲法：

> Chunks 不是收藏夹，而是表达资产工作台。

可展开：

- 保存表达后可以补全翻译、例句、usage note、semantic focus。
- similar / contrast 和 expression cluster 让表达形成关系。
- Review 和 Today 后续能消费这些资产。

边界：

- 当前仍偏重型工作台，后续可以继续简化操作路径。
- AI 补全不是最终事实，仍需要业务结构承接和用户维护。

### 15.4 Review

核心讲法：

> Review 不只是提交对错，而是把表达重新拉回回忆和输出训练，并产生后续调度信号。

可展开：

- due / submit / summary 分入口。
- 复习结果写回服务端，影响后续回流。
- 队列清空后有收束反馈和回 Today 的下一步。

边界：

- 当前输出质量分析还不算很深。
- 调度模型仍有继续细化空间。

### 15.5 Admin / Observability

核心讲法：

> Admin 不只是后台管理入口，也承接音频重生成、缓存排障和最小业务事件回看。

可展开：

- TTS 浏览器缓存面板能辅助排查音频缓存。
- `/admin/observability` 能回看本地 client-events 和失败摘要。
- 管理端按钮、列表、筛选逐步统一，降低维护误操作。

边界：

- 当前不是正式 BI 或全量埋点平台。
- client-events 主要服务本地验收和排障。

## 16. 产品问题准备卡

技术面试也可能追问产品判断。回答时不要把产品讲成“功能列表”，要回到产品北极星：让每一次场景学习沉淀为未来真实场景中能回忆、能使用、能迁移的表达资产。

### 16.1 如果问：这个产品解决的真实问题是什么

回答结构：

1. 先说用户问题：学过很多内容，但离开学习页后很难主动用出来。
2. 再说产品判断：问题不只是“内容不够”，而是学习入口、表达沉淀、复习回流和长期状态没有串起来。
3. 最后说方案：用 Today 编排入口，Scene 提供真实语境，Chunks 沉淀表达资产，Review 做回忆训练，Progress 给长期反馈。

可以这样答：

> 我想解决的不是“再做一个课程页”，而是学过的表达不能变成自己资产的问题。很多学习产品停在看懂内容，AbandonClaw 更关注后续能不能回忆、使用和迁移。所以它把场景学习、表达保存、复习训练和每日编排串成闭环。

### 16.2 如果问：为什么从场景开始，而不是从单词或题库开始

回答结构：

1. 单词和题库更适合局部训练，但容易脱离真实使用语境。
2. 场景能提供人物、语气、目的和上下文，用户更容易理解表达为什么这样说。
3. 表达从场景中被提取后，再进入 Chunks 和 Review，既保留语境，又能沉淀成资产。

可以这样答：

> 因为这个产品的目标不是让用户记住孤立词条，而是让用户未来在相似场景里能说出来。场景提供语境，Chunks 把有价值的表达抽出来，Review 再把它拉回主动回忆。这样输入、沉淀和输出之间是连续的。

容易被继续追问的点：

- 单词是不是不重要：不是不重要，而是不作为产品主入口；词汇可以服务表达，但不替代表达。
- 题库是不是不重要：题目用于训练和反馈，但不能替代真实语境输入。

### 16.3 如果问：Today 的产品价值是什么

回答结构：

1. 用户每天打开学习产品时，最大成本之一是“不知道先做什么”。
2. Today 不是功能入口集合，而是任务编排层。
3. 它把继续学习、review、表达沉淀和进度状态重新组织成今天的优先任务。
4. 推荐解释能降低用户不信任感，让用户知道为什么现在该做这一步。

可以这样答：

> Today 的价值是降低启动成本。用户不需要每天重新判断要学新场景、继续上次中断、还是先复习。系统用服务端聚合状态把这些任务编排出来，并解释推荐原因。它不是内容生产层，而是学习闭环的调度入口。

### 16.4 如果问：Chunks 为什么是资产层，不是收藏夹

回答结构：

1. 收藏夹只是保存，资产层要能维护、关联、复习和回流。
2. Chunks 支持翻译、例句、usage note、semantic focus、相似/对照表达、cluster。
3. 这些内容后续能被 Review 和 Today 消费。
4. 资产化的目标是迁移，而不是保存数量。

可以这样答：

> 如果只是收藏夹，用户保存完就结束了。Chunks 的定位是表达资产工作台，保存后的表达还能补全、关联、聚类和复习。它让一次场景学习产生的表达，变成后续可以继续维护和调用的个人资产。

### 16.5 如果问：Review 和普通背卡片有什么区别

回答结构：

1. 普通卡片常围绕“记得/不记得”。
2. 当前 Review 更强调回忆、熟悉度、输出信心和完整输出。
3. Review 不是孤立训练，它消费 Scene 和 Chunks 的沉淀结果，也把信号写回后续调度。
4. 队列清空后给收束反馈和下一步入口，避免训练结束后断掉。

可以这样答：

> Review 不只是判断对错，而是把表达重新拉回主动回忆和输出训练。它的价值在于产生正式训练信号，再回流到 Today 和 Progress。这样复习不是一个孤立页面，而是学习闭环的一环。

### 16.6 如果问：你怎么判断一个新需求该不该做

回答结构：

1. 先过产品北极星：是否强化回忆、使用、迁移。
2. 再看是否补主链路短板：进入学习、表达沉淀、复习回流、状态承接。
3. 再看成本和复杂度：是否需要改变数据模型、状态流、权限或缓存。
4. 最后决定是立即做、降级做，还是暂缓。

可以这样答：

> 我不会只看一个功能酷不酷，而是先问它是否让用户更容易在未来真实场景中回忆、使用或迁移表达。如果只是增加一个展示入口，但不进入沉淀、复习或回流链路，我会降低优先级。当前阶段更优先补 Today 编排、Review 反馈、音频稳定性和表达资产维护。

### 16.7 如果问：这个产品的核心指标会怎么设计

不要一上来讲 DAU。可以从学习闭环指标讲起：

- 进入：Today 首要任务点击率、continue learning 点击率。
- 学习：scene 完成率、scene 中断后恢复率。
- 沉淀：每个 scene 保存表达数、表达补全率、cluster 组织率。
- 回忆：review 完成率、熟悉度提升、完整输出比例。
- 回流：review 后回 Today 的下一步点击率、次日继续学习率。
- 稳定性：scene full 音频 fallback 率、TTS warm hit rate、关键 API 错误率。

可以这样答：

> 我会把指标围绕闭环设计，而不是只看访问量。比如用户是否从 Today 进入学习，是否完成 scene，是否保存并维护表达，是否回来做 Review，Review 信号是否让下一次 Today 更准确。技术指标也要配合产品指标，比如音频 fallback 率会直接影响听熟体验。

### 16.8 如果问：竞品已经有背单词、AI 对话，这个项目差异在哪里

回答结构：

1. 背单词强在词汇覆盖，但弱在真实表达迁移。
2. AI 对话强在即时互动，但生成内容如果不沉淀，容易变成一次性体验。
3. AbandonClaw 的差异是把真实语境、表达资产、复习回流、每日编排和工程治理串起来。

可以这样答：

> 我不把它定位成背单词工具，也不定位成单纯 AI 对话。它更像一个表达资产闭环：先在场景里遇到表达，再把表达沉淀下来，后续复习、回流和继续学习。AI 是生产和辅助能力，但核心价值是生成内容进入学习生命周期。

### 16.9 如果问：为什么现在不做完整 AI 口语对话

回答结构：

1. AI 口语对话很有价值，但如果没有表达资产和复习回流，容易变成一次性聊天。
2. 当前阶段先建立“输入 -> 沉淀 -> 回忆 -> 再进入”的基础闭环。
3. 后续可以把口语对话作为输出训练层，消费 Chunks 和 Review 中已经沉淀的表达。

可以这样答：

> 我不是认为 AI 口语不重要，而是它更适合建立在表达资产层之后。否则用户聊完一次，内容很难沉淀和复用。当前先把场景学习、表达资产和 Review 跑稳，后续口语对话可以作为更强的输出训练入口。

### 16.10 如果问：这个产品现在最大的产品短板是什么

可以答得具体一点：

- Today 推荐还可以更个性化，目前更偏基于当前学习状态的任务编排。
- Review 输出质量分析还不够深，还可以进一步判断用户是否真的会迁移使用。
- Chunks / Phrases 是重型工作台，长期需要简化成更自然的表达维护路径。
- client-events 还只是本地回看，不是正式跨设备埋点和分析平台。

一句话收束：

> 当前产品已经有闭环骨架，短板主要不在“有没有功能”，而在推荐、反馈、资产维护和观测体系还可以继续细化。

## 17. 技术与产品联动回答

这一节用于回答“你做的技术优化到底服务了什么产品目标”。

### 17.1 音频缓存服务什么产品目标

不要只答性能优化。更好的讲法是：

> 音频缓存服务的是听熟和重复学习。英语场景学习里，用户会反复听同一句、同一个 chunk、同一个完整场景。如果每次都重新生成或等待远端 URL，学习节奏会被打断。所以我做多层缓存、预热和 scene review pack，本质是降低重复听的摩擦。

产品收益：

- 降低重复播放等待。
- 支持碎片化回到同一场景。
- 让 scenes 循环复习更接近“离开页面也能继续听熟”的体验。

### 17.2 服务端状态闭环服务什么产品目标

可以这样答：

> 服务端状态闭环服务的是“系统能接住用户”。用户今天中断了，明天打开 Today 还能继续；用户完成 Review，Progress 和后续推荐能消费这个结果。如果状态只在前端，产品很难做到长期回流。

产品收益：

- Today 能解释为什么推荐当前任务。
- Review 能从真实学习行为里拿信号。
- Progress 能聚合长期表现。
- 重复点击和网络重试不容易破坏学习事实。

### 17.3 接口治理服务什么产品目标

可以这样答：

> 接口治理不是纯后端洁癖，它服务的是学习体验稳定。TTS、AI 生成、导入都是高成本能力，如果没有限流、幂等和错误追踪，用户会遇到重复提交、长时间等待、失败不可解释的问题。治理能力让产品从“能跑”变成“可持续维护”。

产品收益：

- 高成本接口不容易被重复请求打爆。
- 用户态数据访问边界更清楚。
- 出问题时能通过 requestId 和事件摘要定位。

### 17.4 Admin / Observability 服务什么产品目标

可以这样答：

> Admin 和 observability 的目标不是做后台花活，而是让这个学习产品可维护。音频失败、review 中断、today continue 点击这些事情，如果维护者看不到，就很难判断用户在哪里掉出闭环。

产品收益：

- 发布前能看关键链路事件。
- 音频缓存和重生成有排障入口。
- 维护操作统一，降低误操作。

## 18. 常见追问回答

### 18.1 这个项目最大的技术亮点是什么

可以答：

> 最大亮点是它不是单点页面项目，而是一个学习闭环型全栈应用。学习状态由服务端维护，表达能沉淀成资产，音频有生成、存储、缓存、预热和循环播放链路，接口层也补了限流、幂等、追踪和权限边界。

### 18.2 你最有把握讲深的是哪块

优先讲三块：

- 学习状态闭环：start / progress / pause / complete 如何支撑 Today、Review、Progress。
- TTS 音频链路：Storage、签名 URL、Cache Storage、预热、review pack。
- 接口治理：限流、幂等、requestId、Origin、RLS。

### 18.3 review pack 为什么不是逐个音频播放

可以答：

> 因为目标是后台或锁屏后尽量继续播放。逐个音频播放需要 JS 在上一段结束时切下一段，移动端后台不稳定，而且还可能继续发网络请求。review pack 先把多个场景组成一个整体音频，再交给浏览器原生 audio loop，连续性更好。

### 18.4 如果音频候选场景变化怎么办

可以答：

> 当前策略是同日稳定 pack。当天候选变化不会频繁打乱已经准备好的资源；用户刷新或重新准备时可以生成新的 pack。这样牺牲一点实时性，换缓存命中和播放稳定性。

### 18.5 这个项目还有哪些没做完

可以答：

> 还没有做完整 PWA 离线体系、完整 OpenAPI 平台化、平台级熔断、全面 CSP 报告机制和更细粒度的推荐模型。当前阶段重点是先把学习闭环、表达资产、音频链路和接口治理跑稳。

## 19. 简历版表达

### 19.1 项目描述

- 基于 `Next.js 16`、`React 19`、`TypeScript`、`Supabase` 搭建单仓全栈英语学习系统，覆盖 Today、场景学习、表达资产、复习训练与进度聚合等核心模块。
- 设计并落地服务端主导的学习状态闭环，支持学习进度回写、session 恢复、Review 信号沉淀和 Progress 聚合。
- 实现 TTS 音频生成、Storage 持久化、浏览器 Cache Storage 缓存、空闲预热、review pack 循环播放与播放兜底，降低重复请求、二次播放延迟和后台切歌失败概率。
- 建设接口治理基线，包括 `requestId` 追踪、统一错误响应、服务端注册准入、邮箱验证码与邮箱验证闭环、高成本接口限流和 quota、关键写接口幂等、Origin 校验与用户态数据权限边界收紧。

### 19.2 优化点表达

- 通过场景预取、列表/详情缓存回填和空闲时资源预热，优化场景进入速度与重复访问体验。
- 通过音频 URL 缓存、浏览器持久缓存、`blob:` URL 复用和同日稳定 review pack，减少 TTS 重复生成、远端请求和后台播放切歌依赖。
- 通过共享限流、注册 IP 频控、daily quota、管理员紧急开关、幂等去重和统一日志追踪，提升公开注册与高成本 AI 接口的稳定性与可排障性。
- 通过用户态查询与写入逐步切换到 `createSupabaseServerClient + RLS`，提升后端最小权限边界与安全纵深。

### 19.3 招聘平台短版

负责基于 `Next.js 16 + React 19 + Supabase` 搭建英语学习全栈应用，围绕 `Today -> Scene -> Chunks -> Review -> Progress` 建立完整学习闭环。主导学习状态回写、表达资产沉淀、TTS 音频链路和后端接口治理，落地了场景预取、音频多层缓存、空闲预热、review pack 循环播放、服务端注册准入、邮箱验证码与邮箱验证闭环、接口限流、幂等去重、统一错误追踪和基于 `RLS` 的用户态数据权限边界收紧。

### 19.4 口头自我介绍版

我这个项目比较偏全栈产品工程，不只是做页面，而是把学习主链路、表达资产、AI 能力和后端治理一起做了。前端用的是 `Next.js 16 + React 19`，后端主要依赖 `Supabase`。我自己做得比较重的部分，一块是 TTS 音频链路和缓存预热，一块是学习状态闭环和接口治理，比如限流、幂等、requestId、RLS 权限边界这些。

1. 30 秒项目介绍。

4. 表达资产模型。
