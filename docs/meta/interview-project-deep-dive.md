# AbandonClaw 面试项目深挖稿

## 1. 这份文档是给谁、为什么写

这份文档是给我自己准备面试和给别人讲项目用的,不是正式的技术规范。

它要解决三件事:

- 面试前能快速想起项目里值得讲的技术点
- 面试时能把"为什么这样设计"说清楚,而不是只报技术栈名字
- 被追问细节时能讲到实现链路、工程取舍和边界

如果只想看项目用了什么、做到什么程度,看 `docs/meta/technical-overview.md`。要面试、要讲给别人听,看这份。

**讲解原则**:技术细节可以讲,但要先讲清楚业务问题和取舍,再讲实现;能用大白话说的尽量不堆英文术语。

## 2. 30 秒讲完这个项目是什么

这是一个用 Next.js + Supabase 做的英语学习全栈应用。

它不是普通的内容展示网站,而是围绕一条学习主线做闭环:**今日任务 → 场景学习 → 把表达存下来 → 复习 → 看进度**。场景给真实语境,把里面有价值的表达存下来变成自己的资产,复习把这些表达拉回主动回忆,今日任务把所有学习状态汇总起来告诉用户今天该做什么。

我做得比较深的部分包括:学习状态的服务端管理、表达资产的关系维护、文字转语音(后面简称 TTS)的音频生成和缓存链路、前端多层缓存和预加载、高成本接口的限流和治理、注册入口的安全防护、权限边界的逐步收紧,以及把 AI 能力真正融进学习流程而不是做成孤立的演示。

> **30 秒电梯版**:"我做了一个英语学习应用,核心想解决的不是'看懂内容',而是'学过的表达能不能在未来真实场景里用出来'。所以围绕场景学习、表达沉淀、复习训练做了一条闭环,前端用 Next.js 后端用 Supabase,做了比较深的音频缓存链路、AI 能力嵌入和后端接口治理。"

## 3. 2 分钟讲架构

项目是单仓库全栈结构 —— 前端页面、后端接口、数据库迁移都在同一个仓库里。

核心目录大致是:

- `src/app`:页面和后端接口的入口
- `src/features`:按业务分类组织的页面级组件
- `src/components`:公共 UI 组件
- `src/lib/server`:服务端业务逻辑、权限、数据访问
- `src/lib/cache`:前端缓存和预加载
- `supabase/sql`:数据库的表结构、权限规则和迁移历史

架构思路一句话总结:

> 页面层尽量薄,只负责"该展示什么、该派发什么事件";业务判断都下沉到下面的逻辑层;跨页面共享的状态尽量放服务端管;前端缓存负责让体验流畅;Supabase 提供账号体系、数据库和文件存储。

## 4. 为什么这样选

### 4.1 为什么用 Next.js + Supabase 单仓全栈

这个项目还在快速成型阶段,学习主线、复习、表达管理、音频、AI 这些功能需要频繁互动。单仓全栈的好处:

- **页面和接口一起改**,不用前后端分两个仓库联调
- **服务端的聚合逻辑可以贴着页面入口写**,今日任务、进度页这种"汇总展示"的页面写起来直接
- **Supabase 直接提供账号、数据库、文件存储**,不用自己搭一套
- **数据库结构、权限规则和业务代码都能在同一个仓库里追**

它不是最终架构。等以后 TTS、推荐、AI 生成这些独立能力负载变大,可以再拆出来成独立服务。

### 4.2 为什么没用重型的全局状态管理(比如 Redux)

这个项目真正需要长期保存的不是前端 UI 状态,而是**学习进度、复习信号、表达资产、音频资源** —— 这些都要跨页面、跨会话、跨设备保存。把这些都塞进一个前端的大仓库,关掉浏览器就没了,解决不了问题。

所以做法是:

- 页面内的临时交互,用 React 自己的 state 处理
- 派生出来给页面展示的数据,放专门的"展示逻辑"文件里
- 能恢复、能聚合、能追踪的核心状态,放服务端管
- 列表和详情的体验优化,用本地缓存兜底

这比硬塞一个 Redux 更贴合业务实际。

### 4.3 为什么页面要薄

项目有一条很重要的约定:**页面只负责"编排",不吞业务**。

举个例子:场景详情页不是普通的展示页,它要同时管:数据加载、视图切换、学习状态同步、音频播放、表达保存、练习生成、变体学习。这些全塞进一个页面文件,很快就会变成几千行没人敢动。

所以项目把复杂度拆成几层:

- **展示逻辑层**:负责"页面要展示什么数据"的派生计算
- **判断逻辑层**:纯函数,只做判断和状态转换(比如"这个表达算不算熟了")
- **动作调度层**:负责"用户点了按钮之后,该按什么顺序执行哪些操作"
- **数据写入层**:负责真正去调接口、改数据
- **组件层**:只管展示和交互

面试可以说:这个项目不是按技术名词堆文件,而是按"页面职责"拆复杂度。

## 5. 技术亮点一:学习状态的服务端闭环

学习不是在前端点一下"完成"按钮就算完成,而是**服务端维护一套完整的状态机**。

关键动作有四个:

- **开始**:用户进入一个场景,标记开始学习
- **进度**:用户在场景里停留、阅读、切换内容时,增量回写进度
- **暂停**:用户离开页面时记录暂停
- **完成**:做完练习或者变体学习,推进到完成态

这套设计的价值:

- 用户关掉浏览器再回来,能从上次的位置继续
- 今日任务页能告诉用户"上次没学完的还有这个"
- 进度页能统计长期学习量
- 复习页能从真实学习行为里拿信号,而不是凭空猜
- 学习进度不是临时 UI 状态,而是可以长期沉淀的数据

如果面试官问"这和普通学习页面有什么区别":

> 普通页面只记录用户当前看到哪里。这个项目把学习行为做成服务端状态机,之后的"今日推荐""复习训练""进度统计"都消费同一份状态。这样不管用户在哪个设备、什么时间回来,状态都是连续的。

## 6. 技术亮点二:表达资产模型

这个项目不是只让用户读完一个场景就完事,而是**把场景里的表达沉淀成"个人资产"**。

一个表达大致会经历这样的生命周期:

1. 用户在场景里遇到一个真实表达(比如 "I'm just trying to figure things out")
2. 把它存下来
3. 在表达管理页里补全翻译、例句、用法说明、语义焦点
4. 生成相似表达和对比表达
5. 把相关表达组织成"表达簇"
6. 进入复习和今日任务的长期循环

值得讲的点是:**"表达管理页"不是收藏夹,而是表达的工作台**。它让"学过一个场景"变成"积累了一组能复用的表达"。

面试可以这样说:

> 我把表达从"一次性内容消费"里抽出来,做成可以补全、关联、复习和迁移的资产层。场景是输入入口,表达管理是资产沉淀层,复习是长期回忆训练。

## 7. 技术亮点三:文字转语音(TTS)的音频链路

音频链路是项目里很值得讲的一块,因为它**不是简单的"请求一个音频网址来播"**。

完整链路是这样:

1. 服务端拿到文本(或分好段的文本),调用微软的 Edge TTS 生成音频
2. 把音频上传到 Supabase 的文件存储里
3. 服务端给前端返回一个**临时下载链接**(有效期是有限的)
4. 前端拿到链接后做几层缓存:内存里记一份网址、浏览器本地存一份音频文件
5. 之后再播放同一段音频,优先用本地的,不再请求服务端
6. 即使临时链接过期了,本地存的音频还能继续播

这个设计同时兼顾了:

- 第一次播放可用
- 第二次播放更快
- 减少重复生成音频(TTS 本身慢且收费)
- 弱网环境也有兜底
- 高成本的 TTS 接口不会被反复打爆

> **30 秒电梯版**:"音频在我们这是高成本资源 —— 生成慢、收费、用户会反复听。所以我把它做成稳定资源:服务端按内容算一个钥匙,文件存到 Supabase 上;客户端做了内存缓存、浏览器本地音频缓存、避免重复请求几层,即使下载链接过期了,本地音频还能继续播。"

## 8. 技术亮点四:场景循环播放(后台播放优化)

这是最近做的一块音频后台播放优化,面试时可以讲,但要讲准。

它**不是**"拿 ffmpeg 把多个音频文件拼起来",也**不是**"前端一段一段请求然后切歌"。真正的做法是**在音频分段级别打包**:

1. 从用户已经学过的场景里,挑出熟悉度够高的候选
2. 按今天稳定的顺序选一小批(保证预加载和点击播放命中同一份资源)
3. 读取每个场景的详情
4. 把每个场景拆成统一格式的音频片段
5. 把多个场景的片段合并成一个"打包数据"
6. 复用现有的"整场景音频生成"通道,生成一个完整的合并音频
7. 前端用一个 audio 标签开循环播放

为什么这么做:

- 手机端切到后台或者锁屏后,页面 JavaScript 不一定能稳定执行"切下一首"
- 如果一段一段播,后台播放强依赖网络和 JS 执行时机,经常断
- 生成一个整体音频后,浏览器原生播放器更容易持续播下去
- 同一天稳定的"打包"能提高缓存命中率,不用每次点击都重新生成

代价也要讲清楚:

- 优点:后台播放稳定、缓存命中稳定、请求次数少
- 缺点:播放过程中不实时切换"当前场景名称";候选变化后需要重新准备
- 它不是完整的离线音频系统,也不是"边播边在末尾热插新内容"

一句话总结:

> 我没有做文件级别的音频拼接,而是把多个场景先转成统一格式的音频片段,再复用现有 TTS 通道生成一个能循环播放的整体音频,用稳定性换掉后台 JavaScript 切歌的不确定性。

## 9. 技术亮点五:缓存和预加载策略

这个项目里**缓存不是单点优化,而是围绕学习体验做了多层兜底**。

主要包括:

- 场景列表的缓存
- 场景详情的缓存
- 复习页的缓存
- 学习主页的缓存
- 表达列表的缓存
- TTS 音频网址的缓存
- 浏览器本地的音频文件缓存
- 空闲时间偷偷预加载

典型策略可以一句话概括:**先用缓存把页面打开,再去后台拉新数据;重资源放到空闲时预加载;弱网或者用户开了省流量模式时,自动减少预加载**。

面试可以强调:这不是为了炫技,而是学习产品的访问很碎片化 —— 用户会反复进入同一个场景、看同一个表达、播同一段音频。所以缓存收益很直接。

## 10. 技术亮点六:后端接口治理和权限边界

项目已经过了"先把功能跑通"的阶段,补了不少工程层面的能力。我按"做了什么 + 解决什么"来讲:

| 做的事 | 解决什么问题 |
|---|---|
| **每个请求带一个唯一编号**(请求编号) | 出问题时方便把前端 → 接口 → 业务层日志串起来 |
| **所有接口出错都返回同一种结构**(错误码 + 详情 + 请求编号) | 前端能稳定解析错误,不用每个接口都猜 |
| **高成本接口限流**(TTS / AI 生成 / 批量导入) | 防止单用户或单 IP 把贵接口打爆 |
| **每日额度 + 后台紧急开关** | AI 接口异常成本能在调用之前止损;出大事可一键关掉某个功能 |
| **注册入口安全防护**(注册模式开关 + 邀请码 + 邮箱验证码 + 邮箱链接验证 + 注册 IP 频率控制) | 防止爬虫批量注册薅 AI 接口 |
| **管理员后台** | 可以降级用户状态、开关高成本能力、管邀请码 |
| **重复请求去重** | 用户连点两次按钮只产生一条数据 |
| **请求来源校验** | 防止外站伪造请求改我们的数据 |
| **数据库行级权限** | 用户数据访问从"应用层判断"逐步收回数据库自带的最小权限模型 |
| **错误聚合到 Sentry**(错误监控平台) | 生产环境出 5xx 自动上报,带请求编号;事故响应从"等用户截图"降到分钟级 |
| **内容安全策略(CSP)上报模式** | 防止 XSS 等注入攻击;违规先收集观察,一键切到强制执行 |
| **启动期配置自检** | 服务启动就在日志里打印一行 10 个关键配置项的状态;部署后看一眼就能判断配置有没有漂移 |
| **自动检查工作流**(GitHub Actions) | 每次提交时跑一整套护栏检查(代码风格、文字乱码、单元测试、规范) |
| **接口错误响应一致性收口** | 全量接口的错误返回统一为"错误码 + 详情 + 请求编号";之前几处用错状态码(比如登录失效返回 500 让人以为是服务器挂了)都修正 |

可以这样讲:

> 早期项目最重要的是把闭环跑通。现在已经开始补限流、去重、追踪、权限边界、错误聚合、安全策略、启动自检和自动检查这些工程化能力,让高成本 AI 接口、用户数据、小范围开放和事故响应都更可控。

> **30 秒电梯版**:"我把接口治理按风险拆三类:公开注册入口要防爬虫,补了邀请码 + 邮箱验证 + IP 频率控制;高成本 AI 接口要防滥用,补了限流 + 每日额度 + 紧急开关;关键写操作要防重复,补了去重保护。所有请求带唯一编号,出错统一格式,生产 5xx 自动上报。"

## 11. 技术亮点七:AI 能力不是孤立的演示

项目里 AI 不是单独做一个聊天框,而是**真正嵌入学习闭环**:

- 场景生成(用 AI 生成完整学习场景)
- 表达解释(用户选中一段不懂的,AI 解释)
- 表达补全(给一个表达,AI 补上翻译、例句、用法)
- 相似 / 对照表达生成
- 场景练习题生成
- 场景变体生成
- TTS 音频生成

面试可以这样讲:

> 我更关注 AI 输出进入业务后的生命周期。生成内容只是第一步,更重要的是它能不能进入学习、保存、复习、统计和再次推荐。如果生成完就完了,那就是个 demo;能进入闭环,才算真正的能力。

## 12. 最值得讲的工程取舍

### 12.1 没做大而全的微服务拆分

当前用 Next.js 接口 + Supabase 跑通闭环,开发速度快、联调成本低。等 TTS、推荐、AI 生成这些负载真的大到一台机器扛不动,再拆服务更合理。

### 12.2 没引入重型全局状态管理

真正需要长期保存的是学习状态和表达资产,不是前端 UI 状态。把长期状态放服务端,比堆一个全局仓库更有价值。

### 12.3 没做 ffmpeg 音频拼接

循环播放需要的是后台播放稳定性,不是音频编辑能力。分段级别打包能复用已有的 TTS 通道,复杂度更低,也和缓存链路一致。

### 12.4 没追求完整离线

当前做的是缓存兜底和二次播放加速,不是完整的离线应用。这样能先解决高频体验问题,避免维护成本过高。

## 13. 面试官追问树

按"面试官继续往下问"的方式组织。准备时不要死背整段,要记每个问题的回答结构:**先说问题约束,再说当前方案,再说边界**。

### 13.1 学习状态为什么放服务端

回答结构:

1. 业务约束:学习进度会影响今日推荐、复习队列、进度统计,不是当前页面的 UI 状态
2. 状态动作:开始 / 进度 / 暂停 / 完成 / 练习 / 变体,这些都是事实,要在服务端记录
3. 工程收益:刷新可恢复、跨页面可消费、重复提交可控、后续能审计和推荐
4. 边界:页面内的临时交互(展开了哪一段、当前选中哪个)还是留前端

可以这样答:

> 我把学习状态分两类:只影响当前页面体验的(比如展开了哪一段)留在前端;本轮学习是否开始、推进到哪、暂停、完成,这些会影响后续推荐和复习的事实放服务端。这样不管用户刷新、回到主页还是去做复习,大家消费的都是同一份事实,不会出现"前端说我学完了但后端不知道"。

容易被追问的点:

- **重复点击怎么办**:关键写操作有"去重钥匙",短时间内同样的请求会复用结果
- **网络失败怎么办**:前端不把"看起来成功"当成最终事实,以服务端响应为准
- **为什么不用 Redux**:Redux 只能解决前端共享,解决不了跨会话、跨设备、可审计

### 13.2 TTS 为什么要做这么多层缓存

回答结构:

1. 成本:TTS 生成慢、收费,而且用户会反复播放同一段
2. 缓存钥匙:不同粒度(整段、单句、整场景)有各自稳定的钥匙,不能混用
3. 服务端:先查文件存储有没有,有就返回,没有才生成
4. 客户端:浏览器本地缓存、内存缓存、避免重复请求、避免重复创建播放对象
5. 边界:不是完整离线,第一次没缓存还是要联网

可以这样答:

> 音频不是普通 JSON 数据。它生成慢、成本高,而且学习场景里会反复播放。所以我把它做成"稳定资源":服务端按内容算一个钥匙,优先看文件存储里有没有;客户端优先读本地音频,即使临时下载链接过期了,本地的音频文件还能继续播。这样第二次播放不再依赖远端接口。

容易被追问的点:

- **临时链接过期怎么办**:浏览器本地存的是音频文件本身,过期后转成本地播放地址继续播
- **缓存会无限大吗**:浏览器本地缓存有条目数和总大小上限,会自动清掉旧的
- **为什么还要内存缓存**:同一会话里反复点同一个表达,内存缓存能避免重复请求和重复创建播放对象

### 13.3 循环播放为什么是分段级打包

回答结构:

1. 目标:不是音频编辑,是后台 / 锁屏播放的连续性
2. 问题:一段一段播依赖 JS 在每段结束时切下一段,后台不稳定
3. 方案:同一天稳定的候选 → 转成统一格式的音频片段 → 合并 → 复用 TTS 通道生成整体音频
4. 失败处理:打包失败回退到一场景接一场景的队列
5. 代价:不实时切换"当前场景名",不做"边播边热插"

可以这样答:

> 我没有做 ffmpeg 拼接,因为需求不是音频剪辑,而是让后台播放更稳定。播放前把多个已学场景转成统一格式的音频片段,再复用现有 TTS 通道生成整体音频。播放开始后交给浏览器原生的循环播放,减少后台 JS 切歌的不确定性。

容易被追问的点:

- **候选变了怎么办**:同一天稳定打包优先,刷新或重新准备时生成新打包
- **为什么不随机**:完全随机会降低预加载命中率,稳定顺序能让预加载和点击播放指向同一份资源
- **为什么不是完整离线**:当前目标是缓存兜底和播放连续性,不承诺离线可用

### 13.4 接口治理具体保护了什么

回答结构:

1. 分三类风险:**公开注册入口**、**高成本的读 / 生成接口**、**关键写接口**
2. 注册:模式开关 + 邀请码 + 邮箱验证码 + 邮箱链接验证 + IP 频率控制
3. 高成本接口:用户 / IP 限流 + 每日额度 + 后台紧急开关
4. 关键写接口:去重保护(学习推进、复习提交、表达保存)
5. 排障:请求编号 + 统一错误返回 + 结构化日志
6. 权限:请求来源校验 + 账号状态 + 数据库行级权限逐步收紧

可以这样答:

> 我不是把接口治理当一个统一中间件口号,而是按风险来补。注册入口要有模式开关、邀请码、邮箱验证码、邮箱验证和 IP 频率控制;TTS 和 AI 生成这类高成本入口要限流、每日额度和紧急开关;学习推进、复习提交、表达保存这些写操作要去重;所有错误响应都带请求编号,方便从前端报错、接口日志和业务日志串起来。

容易被追问的点:

- **限流存哪里**:有共享存储(Upstash 这个 Redis 服务)就用它,没有就退回到本地内存
- **邮箱验证码是不是自建**:注册页那 6 位验证码是项目内自己做的最小闭环;账号体系、邮件链接确认这些还是用 Supabase 提供的
- **去重是不是持久化**:目前是轻量内存方案,适合当前阶段;并发再大可以迁到共享存储
- **数据库行级权限做到什么程度**:主要用户表逐步承接,不是完整的平台化权限系统
- **错误响应一致性怎么收口**:不是改一处补一处,而是先盘点(58 个接口分三档优先级),再起一个统一变更一次性收口高优问题,把"高成本接口必须按业务模块打日志"写进规范

### 13.5 AI 能力怎么避免变成 demo

回答结构:

1. 项目不是聊天框,是学习闭环
2. AI 输出必须进入业务生命周期:学习、保存、练习、复习、统计、推荐
3. 提示词只是入口,后面还有校验、结构化、落库和 UI 承接
4. 高成本 AI 接口要进限流、错误追踪和兜底体系

可以这样答:

> 我更关注 AI 输出进入业务后的生命周期。比如生成一个场景不算结束,它要能变成可学习的场景;生成一个相似表达也不算结束,它要能进入表达资产、表达簇和复习。AI 在这里是内容和训练能力的生产方式,不是孤立 demo。

### 13.6 这个项目最能体现工程能力的地方

不要只答"用了 Next.js 和 Supabase"。更有说服力的回答:

> 最能体现工程能力的是:我没有只做功能页面,而是把学习状态、表达资产、音频资源、AI 生成和接口治理放到同一个闭环里处理。一个场景不是展示完就结束,它会产生学习状态、表达资产、复习信号和音频资源,这些又会回到今日任务和进度页。技术设计服务的是这条闭环,而不是单点功能炫技。

可以补三点:

- 复杂状态放服务端,页面保持薄
- 高成本资源做缓存、预加载、失败兜底
- 有明确边界,不提前做大平台、不做完整离线、不做重型状态管理

### 13.7 你怎么证明这些不是简历上的空话

回答结构:

1. 先把能力落到用户链路,而不是先报代码文件
2. 再说对应的接口入口、服务层、缓存层或测试
3. 最后说验证方式:单元测试、交互测试、文档约束

可以这样答:

> 我会按链路证明。比如 TTS 不是一句"做了缓存",而是从音频接口进入服务端生成和文件存储命中,再到客户端先查本地缓存、内存缓存、避免重复请求,最后通过播放事件和管理后台看失败摘要。学习状态也不是一句"服务端管理",而是有开始 / 进度 / 暂停 / 完成入口,下游今日任务、复习、进度页消费同一份状态。

### 13.8 你遇到过最难的技术问题是什么

**优先讲 §14.15 那个第三方 TTS 库的线上事故** —— 那是真实生产环境的排查,有完整的"症状 → 根因 → 第一次修失败 → 重修 → 验证"闭环,比"后台音频播放稳定性"更有冲击力。

如果想讲"后台播放":

回答结构:

1. 问题:手机后台或锁屏后,页面 JS 不一定稳定执行下一段切换
2. 初始直觉:一段接一段播,但它依赖"播放结束 → 切下一段 → 开始播"这条 JS 链
3. 取舍:不用 ffmpeg 文件拼接,而是分段级打包,复用已有 TTS 通道
4. 结果:播放开始后尽量由单个整体音频承接,缓存命中也更稳
5. 边界:不是完整离线,不承诺所有浏览器都能后台启动新音频

### 13.9 项目里有哪些地方你刻意没做

这类问题要讲成熟度,不要只说"时间不够":

- **没做完整微服务拆分**:当前闭环优先,等独立能力负载变大再拆
- **没做完整离线应用**:当前先解决二次播放和缓存兜底,完整离线引入同步、版本和容量管理太复杂
- **没做重型状态管理**:长期状态应该在服务端,前端大仓库解决不了跨会话和可审计
- **没做 ffmpeg 拼接**:循环播放目标是连续性,不是音频编辑;分段打包更贴合现有链路

一句话收束:

> 我没把项目往"大而全平台"方向提前推,而是先把学习闭环里最影响稳定性和可维护性的点做实。

### 13.10 如果再给你两周会优化什么

回答要围绕现有边界,不要突然开一个新产品方向。优先级:

1. **强化观测**:当前本地事件 + 5xx 聚合够内测用了,下一步是升级到用户结果指标(7 / 14 / 30 天表达回忆率、主动使用率),用数据回答"产品目标是否在达成"
2. **细化 TTS 指标**:区分单段、整场景的"预热命中""冷启动""兜底失败"的比例,用数据判断预热是不是真的有用
3. **收紧接口契约**:把高频接口的输入输出和错误结构进一步稳定下来
4. **完善权限边界**:继续把用户数据访问收回数据库行级权限
5. **今日推荐引入"表达资产"维度**:现在推荐主要看入门路径和继续学习,引入"基于已保存表达推荐相关场景"能直接强化"迁移"维度,这是当前最大短板

不要优先说的:

- 直接重写成 Java 或微服务
- 立刻上完整 BI 平台
- 立刻做全量离线
- 立刻引入复杂推荐模型

这些可以作为长期方向,但不是当前最小高收益路径。

## 14. 技术细节深挖卡

这一节用来把"我做了某个技术点"继续往下讲到实现层。面试时不一定全讲,但被追问时可以从这里抽一条链路展开。

### 14.1 TTS 音频钥匙怎么保证不串味

可以这样讲:

> 音频缓存不是按网址粗暴缓存,因为临时下载链接会过期。我先把音频抽象成"稳定钥匙",再让服务端文件存储、浏览器本地缓存和播放层都围绕同一套钥匙工作。

关键细节:

- **单个表达的音频**:按清洗后的文本算钥匙,适合跨页面复用
- **句子 / 段落的音频**:把句子编号、说话人、语速和文本一起算指纹,避免同一句在不同语速下混用
- **整场景的音频**:先合并所有片段,再按整段内容和场景类型算哈希,任何内容变化都会换钥匙
- 浏览器本地缓存用稳定钥匙,不直接依赖会过期的临时链接

被追问时补一句:

> 所以"二次播放更快"不是因为链接没过期,而是因为真正复用的是稳定的音频资源和浏览器本地文件。

### 14.2 TTS 生成接口的链路怎么讲

回答结构:

1. 接口先做用户身份校验
2. 校验参数,解析类型(单段 / 句子 / 整场景)、语速、文本、分段
3. 先查服务端内存里的临时链接缓存
4. 再查 Supabase 文件存储里有没有这个对象
5. 命中就直接返回链接
6. 没命中才用 TTS 库生成音频文件,上传到存储
7. 上传失败时返回"直接内嵌音频数据"作为兜底,保证首次播放也有得听

可以这样答:

> TTS 链路是"先命中已有的,再生成"。这能避免用户每次点播放都触发合成。运行时真正复用的是 Supabase 的文件存储;本地代码目录里 `public/audio` 只是开发或兼容副产物,不算生产主缓存。

边界要讲清楚:

- 服务端的临时链接缓存是进程内存里的,不是跨实例强一致
- 当前判断"文件存不存在"用的是"尝试创建临时链接",规模变大后可以优化成查对象元信息
- 内嵌音频数据是兜底,不是主路径

### 14.3 客户端音频缓存为什么分四层

四层按"减少什么成本"来讲:

| 层 | 解决什么 |
|---|---|
| 内存网址缓存 | 减少同一会话重复请求音频接口 |
| 浏览器本地文件缓存 | 绕过临时链接过期问题,保存音频文件 |
| 预加载状态集合 | 避免同一个网址反复触发预加载 |
| 本地播放地址复用表 | 避免同一份文件反复创建本地播放地址 |

可以这样答:

> 客户端不是简单的"有缓存就用",而是按资源生命周期拆层。网址缓存解决短期重复请求,本地文件缓存解决链接过期后的复用,播放地址复用解决播放层重复创建对象。

### 14.4 场景详情为什么按"段落"播,不是按"句子"

可以这样讲:

> 场景详情的主播放单元从"单句"调整成"段落",是为了更贴近用户真实"听一段对话"或"听一个语义块"的方式。技术上仍然复用句子级的 TTS 通道,不新增接口协议,避免为了 UI 颗粒度变化扩接口。

关键细节:

- 段落音频复用句子级别的 TTS 类型
- 段落的"标识"用"block-段落ID"的格式,避免和真实句子混淆
- 段落文本优先来自段落自带的语音文本字段,没有时拼接段落内句子的文本
- 业务事件里用"音频单元 = block"区分"复用句子通道"和"真实句子音频"

取舍:

- **优点**:用户点击更贴近段落听熟;接口不新增协议
- **代价**:调试时必须看"音频单元"字段,否则容易误以为主链路仍在播放单句

### 14.5 整场景音频失败为什么不自动逐句串播

回答结构:

1. 整场景音频失败原因有多种(供应商错、超时、片段拼接失败、上传失败、链接失败、结果为空、冷却中)
2. 自动逐句串播会引入一套新的播放状态机,尤其涉及后台、暂停、切换、错误续播
3. 当前最小可维护方案是:识别失败原因 + 失败摘要 + 45 秒冷却 + 显式按钮回到逐段播放

可以这样答:

> 我没有把整场景失败自动降级成逐句串播,因为这会把一个"生成失败"问题扩大成新的"播放器状态机"问题。当前方案是识别失败原因,短时冷却避免重复打上游,再给用户明确的逐句跟读入口。这样能兜住体验,又不引入难维护的自动串播逻辑。

可展开的失败原因:

- 供应商出错
- 超时
- 片段拼接失败
- 上传失败
- 链接生成失败
- 结果为空
- 冷却中(短时间内已经失败过)

### 14.6 预加载收益怎么观测

可以这样讲:

> 我不是只做预加载,还做了轻量的"预加载登记表",把"预加载入队"和"后续播放事件"用稳定钥匙关联起来。这样能看"预加载命中"和"冷启动"的差异,而不是凭感觉判断有没有用。

关键细节:

- 登记表的钥匙和浏览器本地缓存的钥匙一致
- 来源区分"首次进入""空闲触发""播放触发"
- "用户点击" **不算**预加载来源,因为用户点击是即时需求,不是预加载收益
- 管理后台的统计是基于本地事件计算的,不代表全量用户数据
- 失败有冷却阶梯(5 秒 → 60 秒 → 5 分钟 → 30 分钟),疑难内容失败后不会立即重试;30 分钟静默后失败计数才重置
- **跨场景取消**:用户从场景 A 切到场景 B 时,A 的预加载任务会被取消,避免后台空跑占带宽

补边界:

> 样本少时"预加载收益"可能为负,这不是失败,只是说明当前调度需要继续观察。

### 14.7 场景练习题生成链路怎么讲

回答结构:

1. 场景详情先把课程数据转成可出题的结构
2. 让 AI 基于段落、句子、表达出题
3. AI 返回结构不合法时,回退到本地出题逻辑
4. 把填空题的题面和答案对齐
5. 前端最终组装成四层训练模块:**填空 → 引导回忆 → 句子复述 → 完整听写**
6. 前一模块全部答对后,下一模块才解锁

可以这样答:

> 练习生成不是模型返回什么就直接展示。AI 负责提供题目候选,本地还会做结构校验、兜底、填空标准化和四层训练模块编排。这样能兼顾生成灵活性和前端训练流程稳定性。

可展开的取舍:

- 首轮填空当前更偏保守,最多围绕 5 到 8 题
- 兜底会优先挑高价值表达,但不会把同一句拆成很多碎空
- 引导回忆会过滤太短的句子,避免半句复现没有训练意义

### 14.8 练习题为什么要在服务端持久化

可以这样讲:

> 练习题不能只存在本地缓存里,否则用户刷新、换设备或提交答题时,服务端无法确认这套题是不是属于当前用户和当前场景。现在练习题本身会保存到服务端,本地缓存只负责"秒开"和失败兜底。

关键细节:

- 生成接口只负责生成题目,不负责落库
- 场景详情生成或重新生成后,会调用专门的接口把练习题保存到用户名下
- 答题写入前会校验"这套题真的属于当前用户和当前场景"
- 手动重新生成会创建新的一套,旧的标记为废弃

边界:

- 当前不做公共题库
- 不做跨用户共享题集
- 不把题目拆成标准化的公共表

### 14.9 表达保存链路为什么复杂

可以这样讲:

> 表达保存的复杂点不在 UI,而在"保存动作会同时影响多张表"。所以项目把数据拼装、写入语义和表达簇操作拆开,避免每个入口各自拼一套规则。

关键实体:

- **表达本身**:标准化的表达实体
- **用户维度学习条目**:区分整段表达和单句
- **用户维度关系表**:相似、对照
- **用户维度表达簇**
- **用户每日学习统计**

关键边界:

- 相似表达可以进入同一个表达簇
- 对照表达只写"对照关系",不应并入相似表达簇

被问"为什么复杂"时:

> 因为表达资产不是一条收藏记录。它后续要支持相似表达、对照表达、表达簇、复习和统计,所以保存时必须把关系和资产结构一起维护。

### 14.10 接口分层怎么讲得更具体

可以按这条链路讲:

- **接口入口层**:身份校验、参数解析、调用业务层、返回统一响应
- **可测试动作层**:部分模块把接口里的动作抽出来,可以脱离 HTTP 单独测
- **业务层**:业务语义、状态推进、跨表副作用
- **数据访问层**:Supabase 查询
- **共享逻辑层**:纯函数、选择器、规范化和可单测规则

可以这样答:

> 我没有做非常重的领域驱动设计,但也避免接口入口堆所有逻辑。接口入口主要处理 HTTP 边界,业务层承担业务语义,数据访问层处理查询,共享逻辑放纯函数。这样测试时能绕开 HTTP 直接测,也能在需要时补真实 HTTP 验证。

### 14.11 测试怎么证明链路不是手测

回答结构:

1. 纯规则用单元测试(提示词构造、业务逻辑、缓存钥匙、去重)
2. 页面交互用组件测试
3. 高风险入口用接口测试,验证错误结构和请求编号
4. 维护类脚本检查文字乱码、规范一致性、护栏
5. 不是默认跑全量,而是按改动风险跑最小相关集

可以这样答:

> 我会按风险选测试,不是无脑全量。改 TTS 就跑音频接口、预加载、服务端 TTS、重新生成接口;改场景练习就跑练习生成、场景练习视图、练习题;改表达保存就跑保存契约、手动组合、焦点辅助和页面交互。

面试时可以强调:

> 测试策略和架构拆分是配套的。逻辑拆到业务层 / 共享层后,才能用轻量测试覆盖核心语义;交互测试只覆盖真正需要 DOM 的路径。

### 14.12 复习调度怎么消费 5 类正式信号

这是项目里比较深的一条链路。面试官追问"复习不只是对错怎么实现"时可以展开:

回答结构:

1. **UI 收集**:复习页面收集 5 类信号 —— 识别状态、输出信心、完整输出状态、变体改写状态、完整输出覆盖度
2. **数据库落库**:专门的表保留每次复习完整事件 + 主反馈(再次 / 困难 / 良好)
3. **排序消费**:把 5 类信号映射成 5 级"焦点"(比如"输出信心低""目标表达未覆盖"),决定到期列表的优先级
4. **节奏消费**:在保留主方向的前提下,根据信号细调下次复习时间(12 小时到 10 天)
5. **解释回流**:"焦点"字段透传到 UI,映射成稳定文案("仍需输出训练""尚未覆盖目标表达"),用户能看到"为什么这条优先出现"
6. **历史兼容**:所有空值走保守中性解释,新字段按新增记录逐步生效,不要求回填

可以这样答:

> 复习不是简单的间隔重复算法。我们已经把"用户识别 / 主动回忆 / 变体迁移 / 完整输出" 4 类阶段信号正式入库,并且让排序和节奏都按这些信号细调。比如"完整输出未覆盖目标表达"的条目,即使用户提交"良好",下次也不会被排到很远;提交"再次"但同时未完成迁移训练的,会进一步收紧到 12 小时。这让复习不止"判断对错",而是基于"在哪个训练深度卡住"的细粒度调度。

被追问的点:

- **为什么不直接覆盖再次 / 困难 / 良好**:这 3 个是用户最直接反馈,新信号是补充维度;保留主方向兼容前端 UI 和历史数据
- **为什么新信号有空值兼容**:历史复习没有这些字段,强制回填会伪造结果;保守中性解释比假数据更安全
- **当前最大缺口**:有信号 + 有消费 + 有 UI 解释,但没有量化指标(7 / 14 / 30 天表达回忆率)来验证"这套调度真的让用户记住更多"

### 14.13 可观测性的多个入口

被问"出事故了怎么定位"时按入口讲,不要只答错误监控:

| 入口 | 用途 | 时效 |
|---|---|---|
| 错误监控平台 | 生产 5xx 自动聚合 + 请求编号标签 + 按业务模块自动分桶 | 分钟级告警 |
| 启动日志 | 服务一启动就打印一行,看 10 个关键配置是否就绪 | 部署后第一时间 |
| 管理状态页 | 实时观察限流后端、当日高成本用量、额度 | 即时 |
| 管理事件页 | 浏览器本地回看最近业务事件(继续学习、复习提交、场景完成、音频失败) | 会话内 |
| 用量快照脚本 | 当日高成本能力用量(3 段汇总 + 明细),可被定时任务采集 | 可定时 |
| 请求编号贯穿 | 把请求路由 → 业务层 → 错误响应 → 错误监控串成同一线索 | 任意时刻 |
| 预加载生命周期事件 | 回答"预加载到底有没有命中" + "失败回退到第几档" | 会话内 |

可以这样答:

> 我没把可观测性当成"上一个错误监控就够了",而是按事故类型分入口。生产 5xx 走错误监控告警 + 按业务模块自动分桶;部署后看启动日志确认配置没漂;实时排查走管理状态页和管理事件页;成本异常用用量快照看趋势;预加载是不是被环境抑制或者失败进了冷却第几档,看预加载事件;用户报某个请求编号时,所有日志都能用同一个编号串起来。

### 14.14 超重页面多轮拆分:工程方法论的真实案例

被问"项目里有哪些能体现工程方法论的东西"时,这是一条很有说服力的链路 —— 有量化数据,有"失败 → 反思 → 改策略 → 验证"的完整闭环。

#### 背景

表达管理页一开始是 2368 行单文件,是仓库里最大的前端文件。直接重写有两个问题:一是一次性大改容易出回归 bug;二是不知道哪种拆法收益最高。所以走"按规范化变更多轮迭代"的路线,每一轮都记录拆法和减幅,给下一轮策略提供数据。

#### 时间线和量化

| 轮次 | 起始行数 | 收口行数 | 减幅 | 关键策略 |
|---|---|---|---|---|
| 第一轮 | 2368 | 2125 | -243(-10.3%) | 抽样式常量 + 3 个 hook + 顶部组件 |
| 第二轮 | 2125 | 2102 | -23(-1.1%) | 抽 3 个 hook + 一个区域组件 |
| 第三轮 | 2108 | 2041 | -67(-3.2%) | 抽"属性很多的子树"(一次抽 41 个属性的列表组件) |

场景详情页同模式应用一次:**1326 → 849 行(-36%)**,抽出 4 块逻辑。

#### 关键发现(写进第二轮规范,直接指导第三轮)

第二轮拆完发现减幅远低于预期。逐对象算账才发现:

| 抽离对象 | 移走代码 | 页面新增"装配代码" | 净减 |
|---|---|---|---|
| 表达图相关 hook | ~10 行 | ~20 行解构 | **+10**(反而增长) |
| 句子表达保存 hook | ~26 行 | ~8 行解构 | -18 |
| 焦点关系标签 hook | ~10 行 | ~13 行解构 | +3 |
| 焦点模式区域组件 | ~55 行 JSX | ~42 行属性透传 | -13 |

**关键发现**:抽小 hook 时,"页面里写的装配代码"和"抽走的代码"差不多抵消,页面几乎不瘦身。

**对第三轮的指导**:第三轮应该优先抽"属性很多的子树",而不是继续抽小 hook。

#### 第三轮验证发现成立

第三轮按第二轮策略只抽一个"属性多的列表子树",把和这个子树强耦合的辅助函数一并迁入,减幅 **-67 行,大约是第二轮的 3 倍**。策略验证成立。

#### 怎么讲

> 这条线最值得讲的不是"我把大页面拆小了",而是"我把'怎么拆'变成了一个可量化、可迭代的工程问题"。第一轮收口后我没立刻跳进下一轮,而是先看减幅;第二轮减幅低于预期,我把每个抽离对象的"移走代码 vs 新增装配开销"拆开算账,发现"装配代码"和"抽走代码"近似抵消;这个发现直接写进规范,指导第三轮改抽"属性多的子树",实际减幅立刻翻 3 倍。核心是把策略验证留在规范里,下一个维护者(或 AI)能直接复用结论,不用从头试错。

#### 容易被追问

- **为什么不一次重写**:重写成本高、风险大,且不知道目标值;多轮迭代每次都有规范变更把策略和量化留档
- **怎么保证不破回归**:每轮拆分都要求页面渲染结果字节级兼容(相同的 className、属性);交互测试不弱化断言;每一轮跑完整测试
- **为什么把"关键发现"写进规范而不是开发日志**:规范是给"未来推第四、第五轮的人"看的硬约束,开发日志是给当下的人看的实施记录

### 14.15 线上事故复盘:第三方 TTS 库的隐藏 bug 导致进程反复崩

这是非常新鲜的真实生产事故,完整的"症状 → 根因 → 第一次修复失败 → 重修 → 验证"闭环。面试官问"最难的技术问题""线上事故复盘""怎么处理第三方库的 bug"都能直接讲。

#### 症状

用户反馈"网站时好时坏,刷新几次能进去"。Chrome 报"连接已重置",看网络面板请求大约 300 毫秒就被打断,0 字节响应 —— 是网络层被强制关闭,**不是后端返回了 5xx 错误**。

服务器上看进程状态,主应用的"重启计数"已经 102 次,但运行时间只有 5 分钟。说明进程在反复崩 + 自动拉起,拉起的窗口里 nginx 拿不到后端响应 → 用户看到的就是"连接被打断"。

#### 根因

进程日志反复出现:

```
未捕获异常: TypeError: Cannot read properties of undefined (reading 'audio')
```

源头是 `msedge-tts` 这个 npm 包 —— 我们用来生成 TTS 的库。

它在 WebSocket(一种长连接)的回调里,**裸读了一个对象的 `audio` 属性**。当我们应用层主动关掉连接后,微软 Edge 服务器还会继续推几帧残余数据,这些回调拿到的对象已经被清掉了,访问 `.audio` 就是访问"未定义.audio" → 报错。

这个错误**发生在长连接回调里,完全在我们 `await` 边界之外**,调用方的 `try / catch` 拦不到 → 直接冒到 Node 的"未捕获异常" → 进程崩。

#### 第一次修复失败,真正原因

直觉做法:加一个"进程级守卫",**只**白名单这一类已知错误(其它异常仍然让进程退出,保留出错及时崩的原则)。

第一版用"错误栈包含 msedge-tts"这个条件匹配。本地开发环境的栈长这样:

```
at WebSocket.onmessage (.../node_modules/msedge-tts/dist/MsEdgeTTS.js:142:42)
```

部署生产后,栈变成:

```
at ignore-listed frames
```

—— **Next.js 生产模式会把 `node_modules` 里的栈折叠成这一行**。我的"栈包含 msedge-tts"永远拿不到匹配,守卫没生效,进程继续崩。

这是一个**只在生产复现、本地永远查不到**的坑。教训:排查时要把"开发环境符号丢了没"和"生产环境符号丢了没"分开看。

#### 最终方案

改成只用**错误消息特征匹配**:

```typescript
const isKnownMsEdgeTtsWebSocketRace = (error: unknown): boolean => {
  if (!(error instanceof TypeError)) return false;
  const message = error.message ?? "";
  return (
    message.includes("Cannot read properties of undefined") &&
    (message.includes("'audio'") || message.includes('"audio"'))
  );
};
```

三层约束保证白名单不会误伤:

- 必须是 TypeError(这条消息只 TypeError 才会发)
- 消息同时命中 "undefined" 和 "audio"(单双引号变体都覆盖)
- **前置条件**:应用层所有读 `.audio` 的地方都加了"可选链"(`?.`)保护,**这才是主防线**,进程级守卫只做兜底

#### 测试覆盖

6 条单元测试,正反两侧都覆盖:

- 命中:开发模式完整栈
- 命中:**生产折叠栈**(防止后人改时退化回栈匹配又踩同样的坑)
- 命中:双引号变体
- **不**命中:同形态但属性名不是 audio 的 TypeError
- **不**命中:同消息但不是 TypeError(普通 Error 仍然让进程退出)
- **不**命中:`null` / `undefined` / 字符串等非 Error 拒绝原因

#### 验证生效

- 进程"重启计数"从 102 降到 0,运行时间持续增长
- 日志出现"已吞下 msedge-tts 长连接竞态"这条记录,**证明守卫确实在拦,不是 race 自己消失了**
- 用户反馈"刷新几次才能进"的现象消失

#### 怎么讲

> 我们用一个第三方 TTS 库,它的长连接回调里有时序竞态:我们应用层主动关连接后,远端服务器还会继续推几帧,触发对已清理对象的裸读 → TypeError → 未捕获异常 → 进程崩,生产累计了 102 次重启。这种异常出现在 await 边界外,应用层拦不到。
>
> 我加了一个进程级守卫,**只**白名单这一类特定错误,其它异常仍然让进程退出保留出错及时崩的原则。但第一版踩了个生产专有的坑 —— 我用错误栈包含库名来匹配,本地测都通过,生产没生效。后来才发现 Next.js 生产模式会把 `node_modules` 的栈折叠成 "at ignore-listed frames",我的匹配条件永远不命中。改成只用错误消息特征匹配,同时把"生产折叠栈"的情况写进单元测试防止后人退化。
>
> 这件事让我对"白名单异常"的纪律更严格:必须有具体到错误类型 + 关键消息特征的匹配;必须有覆盖生产符号丢失情况的测试;**只在应用层已经主动防护之后才用进程级白名单兜底**,而不是反过来。

#### 容易被追问

- **为什么不 fork 这个库修源码**:技术上可行,但维护一个 fork 的长期成本高于"应用层防护 + 进程级窄白名单"两层防护,后者在生产已经稳定
- **为什么不直接 catch 所有异常不让进程崩**:未捕获异常之后 Node 进程状态可能不一致(资源没关、事件循环残留),catch 所有会把"短崩快重启"变成"半死长崩",反而更难排查。窄白名单 + 其它一律退出才安全
- **怎么定位到是这个库的问题**:我们应用层所有读 `.audio` 的地方都有可选链保护,但日志反复出现同一个消息 → 只有这个库的源码里有裸读 `.audio` → 唯一嫌疑源
- **怎么验证不是误判**:不是看"崩溃消失了",而是看"已吞下"这条日志出现了 → 证明守卫确实在干活,race 还在发生只是被吞了。**如果消失了反而要怀疑是不是没生效**

## 15. 分模块讲解卡

面试前快速过一遍每个模块讲什么、做到哪里、边界是什么。

### 15.1 今日任务

核心讲法:

> 今日任务不是入口列表,而是把"继续学习、复习、表达沉淀和进度"重新编排成今天的优先任务。

可展开:

- 为什么推荐这一步
- 如何承接关闭浏览器后再打开的恢复
- 如何把复习和进度的结果重新组织成下一步入口

边界:

- 当前不是复杂推荐系统
- 个性化还能继续增强

### 15.2 场景学习

核心讲法:

> 场景是主学习工作台,承载真实语境、音频、表达提取、练习和变体学习。

可展开:

- 页面只负责编排,复杂判断下沉到下层
- 学习状态通过服务端"开始 / 进度 / 暂停 / 完成"推进
- 音频分段落、整场景、句子三层兜底

边界:

- 整场景音频失败后不自动一句一句串播,只给提示和替代入口
- 不把所有交互状态都写服务端

### 15.3 表达管理

核心讲法:

> 表达管理不是收藏夹,而是表达资产工作台。

可展开:

- 保存后可以补全翻译、例句、用法说明、语义焦点
- 相似 / 对照表达和表达簇让表达形成关系
- 复习和今日任务后续能消费这些资产

边界:

- 当前仍偏重型工作台,后续可以简化操作路径
- AI 补全不是最终事实,仍需要业务结构承接和用户维护

### 15.4 复习

核心讲法:

> 复习不只是提交对错,而是把表达重新拉回回忆和输出训练,并产生后续调度信号。

可展开:

- 到期 / 提交 / 总结 分入口
- 复习结果写回服务端,影响后续回流
- 队列清空后有收束反馈和回今日任务的下一步

边界:

- 当前输出质量分析还不算很深
- 调度模型仍能继续细化

### 15.5 管理后台 / 可观测性

核心讲法:

> 管理后台不只是后台管理入口,也承接音频重新生成、缓存排障和最小业务事件回看。

可展开:

- TTS 浏览器缓存面板能辅助排查音频缓存
- 业务事件回看能看到最近的关键动作和失败摘要
- 管理按钮、列表、筛选逐步统一,降低误操作

边界:

- 当前不是正式 BI 或全量埋点平台
- 业务事件主要服务本地验收和排障

技术面试也可能追问产品判断。回答时不要把产品讲成"功能列表",要回到产品北极星:**让每一次场景学习沉淀为未来真实场景中能回忆、能使用、能迁移的表达资产**。

### 16.1 这个产品解决的真实问题

回答结构:

1. 用户问题:学过很多内容,但离开学习页后很难主动用出来
2. 产品判断:问题不只是"内容不够",而是学习入口、表达沉淀、复习回流和长期状态没有串起来
3. 方案:今日任务编排入口,场景提供真实语境,表达管理沉淀资产,复习做回忆训练,进度给长期反馈

可以这样答:

> 我想解决的不是"再做一个课程页",而是"学过的表达不能变成自己资产"的问题。很多学习产品停在"看懂内容",这个项目更关注后续能不能回忆、使用和迁移。所以它把场景学习、表达保存、复习训练和每日编排串成闭环。

### 16.2 为什么从场景开始,而不是从单词或题库

回答结构:

1. 单词和题库更适合局部训练,但容易脱离真实使用语境
2. 场景能提供人物、语气、目的和上下文,用户更容易理解表达为什么这样说
3. 表达从场景中被提取后,再进入表达管理和复习,既保留语境,又能沉淀成资产

可以这样答:

> 因为这个产品的目标不是让用户记住孤立词条,而是让用户未来在相似场景里能说出来。场景提供语境,表达管理把有价值的表达抽出来,复习再把它拉回主动回忆。这样输入、沉淀和输出之间是连续的。

容易被追问的点:

- **单词是不是不重要**:不是不重要,而是不作为产品主入口;词汇可以服务表达,但不替代表达
- **题库是不是不重要**:题目用于训练和反馈,但不能替代真实语境输入

### 16.3 今日任务的产品价值

回答结构:

1. 用户每天打开学习产品时,最大成本之一是"不知道先做什么"
2. 今日任务不是功能入口集合,而是任务编排层
3. 它把"继续学习、复习、表达沉淀和进度"重新组织成今天的优先任务
4. 推荐解释能降低用户不信任感

可以这样答:

> 今日任务的价值是降低启动成本。用户不需要每天重新判断要学新场景、继续上次中断、还是先复习。系统用服务端聚合状态把这些任务编排出来,并解释推荐原因。它不是内容生产层,而是学习闭环的调度入口。

### 16.4 表达管理为什么是资产层,不是收藏夹

回答结构:

1. 收藏夹只是保存,资产层要能维护、关联、复习和回流
2. 表达管理支持翻译、例句、用法说明、语义焦点、相似 / 对照表达、表达簇
3. 这些后续能被复习和今日任务消费
4. 资产化的目标是迁移,不是保存数量

可以这样答:

> 如果只是收藏夹,用户保存完就结束了。表达管理的定位是表达资产工作台,保存后的表达还能补全、关联、聚类和复习。它让一次场景学习产生的表达,变成后续可以继续维护和调用的个人资产。

### 16.5 复习和普通背卡片有什么区别

回答结构:

1. 普通卡片常围绕"记得 / 不记得"
2. 这个项目的复习更强调回忆、熟悉度、输出信心和完整输出
3. 复习不是孤立训练,它消费场景和表达管理的沉淀结果,也把信号写回后续调度
4. 队列清空后给收束反馈和下一步入口,避免训练结束后断掉

可以这样答:

> 复习不只是判断对错,而是把表达重新拉回主动回忆和输出训练。它的价值在于产生正式训练信号,再回流到今日任务和进度。这样复习不是一个孤立页面,而是学习闭环的一环。

### 16.6 怎么判断一个新需求该不该做

回答结构:

1. 先过产品北极星:是否强化回忆、使用、迁移
2. 再看是否补主链路短板:进入学习、表达沉淀、复习回流、状态承接
3. 再看成本和复杂度:是否需要改变数据模型、状态流、权限或缓存
4. 最后决定是立即做、降级做,还是暂缓

可以这样答:

> 我不会只看一个功能酷不酷,而是先问它是否让用户更容易在未来真实场景中回忆、使用或迁移表达。如果只是增加一个展示入口,但不进入沉淀、复习或回流链路,我会降低优先级。当前阶段更优先补今日编排、复习反馈、音频稳定性和表达资产维护。

### 16.7 核心指标会怎么设计

不要一上来讲日活。可以从学习闭环指标讲起:

- **进入**:今日任务首要点击率、继续学习点击率
- **学习**:场景完成率、场景中断后恢复率
- **沉淀**:每个场景保存表达数、表达补全率、表达簇组织率
- **回忆**:复习完成率、熟悉度提升、完整输出比例
- **回流**:复习后回今日任务的下一步点击率、次日继续学习率
- **稳定性**:整场景音频兜底率、预加载命中率、关键接口错误率

可以这样答:

> 我会把指标围绕闭环设计,而不是只看访问量。比如用户是否从今日任务进入学习,是否完成场景,是否保存并维护表达,是否回来做复习,复习信号是否让下一次推荐更准。技术指标也要配合产品指标,比如音频兜底率会直接影响听熟体验。

### 16.8 竞品已经有背单词、AI 对话,这个项目差异在哪

回答结构:

1. 背单词强在词汇覆盖,但弱在真实表达迁移
2. AI 对话强在即时互动,但生成内容如果不沉淀容易变成一次性体验
3. 这个项目的差异是把真实语境、表达资产、复习回流、每日编排和工程治理串起来

可以这样答:

> 我不把它定位成背单词工具,也不定位成单纯 AI 对话。它更像一个表达资产闭环:先在场景里遇到表达,再把表达沉淀下来,后续复习、回流和继续学习。AI 是生产和辅助能力,但核心价值是生成内容进入学习生命周期。

### 16.9 为什么现在不做完整 AI 口语对话

回答结构:

1. AI 口语对话很有价值,但如果没有表达资产和复习回流,容易变成一次性聊天
2. 当前阶段先建立"输入 → 沉淀 → 回忆 → 再进入"的基础闭环
3. 后续可以把口语对话作为输出训练层,消费已经沉淀的表达

可以这样答:

> 我不是认为 AI 口语不重要,而是它更适合建立在表达资产层之后。否则用户聊完一次,内容很难沉淀和复用。当前先把场景学习、表达资产和复习跑稳,后续口语对话可以作为更强的输出训练入口。

### 16.10 这个产品现在最大的产品短板

可以答得具体一点:

- 今日推荐还可以更个性化,目前更偏基于当前学习状态的任务编排
- 复习输出质量分析还不够深,还能进一步判断用户是否真的会迁移使用
- 表达管理是重型工作台,长期需要简化成更自然的表达维护路径
- 本地事件还只是本地回看,不是正式跨设备埋点和分析平台

一句话收束:

> 当前产品已经有闭环骨架,短板主要不在"有没有功能",而在推荐、反馈、资产维护和观测体系还可以继续细化。

## 17. 技术怎么服务产品目标

用于回答"你做的技术优化到底服务了什么产品目标"。

### 17.1 音频缓存服务什么

不要只答性能优化。更好的讲法:

> 音频缓存服务的是"听熟"和"重复学习"。英语场景学习里,用户会反复听同一句、同一个表达、同一个完整场景。如果每次都重新生成或等待远端,学习节奏会被打断。所以我做多层缓存、预加载和循环播放,本质是降低重复听的摩擦。

产品收益:

- 降低重复播放等待
- 支持碎片化回到同一场景
- 让循环复习更接近"离开页面也能继续听熟"的体验

### 17.2 服务端状态闭环服务什么

> 它服务的是"系统能接住用户"。用户今天中断了,明天打开还能继续;用户完成复习,后续推荐能消费这个结果。如果状态只在前端,产品很难做到长期回流。

产品收益:

- 今日任务能解释为什么推荐当前任务
- 复习能从真实学习行为里拿信号
- 进度能聚合长期表现
- 重复点击和网络重试不容易破坏学习事实

### 17.3 接口治理服务什么

> 接口治理不是纯后端洁癖,它服务的是"学习体验稳定"。TTS、AI 生成、批量导入都是高成本能力,如果没有限流、去重和错误追踪,用户会遇到重复提交、长时间等待、失败不可解释的问题。治理能力让产品从"能跑"变成"可持续维护"。

产品收益:

- 高成本接口不容易被重复请求打爆
- 用户数据访问边界更清楚
- 出问题时能通过请求编号和事件摘要定位

### 17.4 管理后台 / 可观测性服务什么

> 管理后台和可观测性的目标不是做后台花活,而是让这个学习产品可维护。音频失败、复习中断、继续学习点击这些事情,维护者看不到就很难判断用户在哪里掉出闭环。

## 18. 常见追问回答

### 18.1 这个项目最大的技术亮点

> 最大亮点是它不是单点页面项目,而是一个学习闭环型全栈应用。学习状态由服务端维护,表达能沉淀成资产,音频有生成、存储、缓存、预加载和循环播放链路,接口层也补了限流、去重、追踪和权限边界。

### 18.2 你最有把握讲深的是哪块

优先三块:

- 学习状态闭环:开始 / 进度 / 暂停 / 完成 如何支撑今日任务、复习、进度
- TTS 音频链路:文件存储、临时链接、浏览器本地缓存、预加载、循环播放
- 接口治理:限流、去重、请求编号、来源校验、数据库行级权限

### 18.3 循环播放为什么不是逐个音频播放

> 因为目标是后台或锁屏后尽量继续播放。逐个音频播放需要 JS 在上一段结束时切下一段,手机后台不稳定,而且还可能继续发网络请求。循环播放先把多个场景组成一个整体音频,再交给浏览器原生循环,连续性更好。

### 18.4 如果音频候选场景变化怎么办

> 当前策略是同一天稳定打包。当天候选变化不会频繁打乱已经准备好的资源;用户刷新或重新准备时可以生成新的打包。这样牺牲一点实时性,换缓存命中和播放稳定性。

### 18.5 这个项目还有哪些没做完

> 还没有做完整离线应用、平台级熔断、全面强制执行的内容安全策略和更细粒度的推荐模型。当前阶段重点是先把学习闭环、表达资产、音频链路和接口治理跑稳。

## 19. 简历版表达

### 19.1 项目描述

- 基于 Next.js 16 + React 19 + TypeScript + Supabase 搭建单仓全栈英语学习系统,覆盖今日任务、场景学习、表达资产、复习训练、进度聚合等核心模块
- 设计并落地服务端主导的学习状态闭环,支持学习进度回写、关闭浏览器后恢复、复习信号沉淀和进度聚合
- 实现 TTS 音频生成、文件存储持久化、浏览器本地缓存、空闲预加载、循环播放打包和播放兜底,降低重复请求、二次播放延迟和后台切歌失败概率
- 建设接口治理基线:请求编号追踪、统一错误响应、服务端注册准入、邮箱验证码与邮箱验证闭环、高成本接口限流和每日额度、关键写接口去重、来源校验、用户数据权限边界收紧
- 建设可观测性与上线治理:接入 Sentry 自动捕获 5xx 并注入请求编号标签,启动期检查 10 个关键配置,落地内容安全策略上报模式,搭建 GitHub Actions 自动检查工作流,产出灾备 / 事故响应剧本和可直接套用的 Nginx 反向代理 + 限流模板
- 主导接口错误响应一致性收口:先对 58 个接口做盘点分级,再一次性收口高优问题,全量错误响应统一为"错误码 + 详情 + 请求编号",7 处高成本接口补上按业务模块打日志的能力
- 主导超重页面多轮拆分:表达管理页经三轮拆分(2368 → 2041 行)、场景详情页经一轮(1326 → 849 行,-36%);第二轮收口后建立"装配开销 vs 抽走开销"量化算账方法,把"高属性子树优先"策略写进规范指导第三轮,第三轮减幅是第二轮的 3 倍

### 19.2 优化点表达

- 通过场景预加载、列表 / 详情缓存回填和空闲资源预热,优化场景进入速度与重复访问体验
- 通过音频网址缓存、浏览器本地缓存、播放地址复用和同日稳定循环打包,减少 TTS 重复生成、远端请求和后台播放切歌依赖
- 通过共享限流、注册 IP 频率控制、每日额度、管理员紧急开关、去重和统一日志追踪,提升公开注册与高成本 AI 接口的稳定性与可排障性
- 通过用户数据查询和写入逐步切换到数据库行级权限模型,提升后端最小权限边界与安全纵深
- 通过 Sentry 5xx 聚合 + 启动期自检 + 内容安全策略上报 + GitHub Actions 自动检查,把事故响应从"等用户截图"降到分钟级,把配置漂移检测前置到部署后第一时间

- 通过跨场景取消 + 失败冷却阶梯 + 共享存储签名链接缓存,避免后台空跑、反复打上游、多进程命中率打折三类高成本资源失败放大问题
- 通过接口错误响应契约一次性收口,全量错误响应带"错误码 + 详情 + 请求编号",前端能稳定解析,后端高成本路径自动按模块进入错误监控分桶
- 通过进程级窄白名单守卫修复第三方 TTS 库 WebSocket 回调时序竞态导致的未捕获异常,把生产 102 次进程重启降到 0,守卫包含 6 条单元测试覆盖开发栈 + 生产折叠栈两种符号场景
