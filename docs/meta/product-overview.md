# AbandonClaw 产品说明总览

## 1. 产品定位

AbandonClaw 是一个面向中文学习者的英语场景学习系统。

它不是传统的“单词卡片”产品，也不是只做内容浏览的课程页，而是围绕真实语境、表达沉淀、复习回流和每日任务编排构建的学习闭环产品。

一句话概括：

> 这是一个以场景学习为主线、以表达资产沉淀为核心、以 today 每日编排为入口的英语学习系统。

产品北极星：

> 让每一次场景学习，都沉淀为用户在未来真实场景中能回忆、能使用、能迁移的表达资产。

这句话比 `today -> scene -> chunks -> review` 更接近产品目标本身。后者是实现机制，前者是用户结果。我们优化的不是学习体验本身，而是表达被用户在未来再次使用的概率。

后续判断新需求优先级时，应先看它是否强化这三个维度：

- 回忆：离开原学习场景后，用户还能不能主动想起这个表达。
- 使用：遇到相似表达需求时，用户能不能说出、写出或选择它。
- 迁移：换一个人、主题、语气或场景后，用户能不能改写并继续使用它。

具体落到产品动作上，应优先看它是否帮助用户：

- 更快进入一次真实语境学习。
- 更好理解并提取值得长期学习的表达。
- 把表达沉淀成可维护、可复习、可关联的个人资产。
- 从“看懂”推进到“能回忆、能使用、能迁移”。
- 在下一次学习时被系统自然接住。

## 2. 产品想解决的问题

当前英语学习产品常见的问题是：

- 学习入口碎片化，用户不知道今天该先做什么。
- 学过的内容只停留在浏览层，没有沉淀成自己的表达资产。
- 复习和输出训练割裂，用户很难把“看懂”推进到“能主动说出来”。
- 学习状态依赖临时前端体验，缺少长期可恢复、可聚合、可回流的闭环。

AbandonClaw 关注的是这条连续链路：

1. 今天从哪里开始学
2. 在真实语境里学什么
3. 学到的表达如何沉淀
4. 沉淀后的内容如何复习和回流
5. 系统如何在第二天继续接住用户

## 3. 目标用户

当前产品更适合：

- 希望通过真实场景，而不是纯词表学习英语的中文学习者。
- 已经有一定基础，但缺少持续输入、输出和复习闭环的人。
- 希望逐步建立个人表达库，而不是只完成题目的人。
- 需要碎片化进入，但又希望系统记住自己学到哪的人。

## 4. 核心产品理念

### 4.1 场景驱动，而不是词库驱动

用户的学习起点不是孤立单词，而是一个具体场景。

用户先进入语境，再在语境中完成：

- 阅读内容
- 听音频
- 识别重点表达
- 保存表达
- 做练习
- 看变体

这让学习更接近真实使用环境，而不是脱离语境的硬背。

### 4.2 表达资产化，而不是只读不存

场景中的 chunk / phrase 不只是“看过”，还可以进入用户自己的表达资产库。

这些资产后续可以继续：

- 补全翻译、例句和用法说明
- 建立相似表达 / 对照表达
- 组织成 expression cluster
- 加入 review 复习链路

学习结果不是一次性消耗，而是可持续积累。

### 4.3 每日编排，而不是功能平铺

产品不是把所有能力直接摆给用户，而是通过 `today` 页面把底层学习数据重新编排成“今天该做什么”。

`today` 现在不仅能给出任务入口，还会解释当前为什么优先推荐这一步，例如继续上次中断的场景、处理今日 review，或进入后续练习。

这能降低用户决策成本，让系统承担任务组织角色。

### 4.4 服务端学习闭环，而不是本地瞬时状态

项目中的学习状态不是简单的前端点击记录，而是由服务端维护完整闭环：

- 开始
- 推进
- 暂停
- 完成
- 复习写回
- 聚合回流

这保证了：

- 学习可以恢复
- 进度可以聚合
- today / review / progress 能消费稳定结果
- 重复点击、网络重试和多端操作不容易破坏状态

## 5. 当前版本新增亮点

最近几轮迭代后，产品从“主闭环跑通”推进到了“更接近可持续使用和上线验证”的状态。

### 5.1 Today 编排更像学习教练

`today` 不再只是入口列表，而是开始承担任务解释和承接角色：

- 显示当前首要任务
- 解释为什么推荐这一步
- 展示继续学习、review、表达沉淀等任务状态
- 在用户完成一轮学习后，继续用稳定聚合字段承接结果

### 5.2 Review 从“提交结果”升级为“阶段反馈”

`review` 不再只是提交一次复习结果，而是围绕回忆、熟悉度、输出信心和完整输出形成更清晰的训练节奏。

当前已经支持：

- 普通表达复习
- 场景回补练习
- 来源场景可用性判断
- 正式 review 信号写回
- 提交后的差异化反馈
- 队列清空后的“这轮回忆先收住了”收束反馈
- 返回 `today` 的下一步入口

### 5.3 音频体验更稳

音频链路已经从“能播放”继续往“失败可控”推进：

- 支持句子 / chunk / scene full 等不同粒度音频
- 有音频预热和缓存策略，减少重复请求
- scenes 列表支持循环复习，会提前把一组已学场景准备成 review pack，播放开始后尽量由单个循环音频持续承接后台播放；这不是实时切歌，而是优先保证离开页面或锁屏后的连续听熟
- scene full 播放失败时，会给出受控提示
- 失败后可切到逐句跟读，而不是让用户卡在错误状态
- 管理端批量重生成音频采用有界并发，降低上游压力

### 5.4 后端接口治理更完整

当前已经补齐一批基础治理能力：

- 请求级 `requestId`
- 统一错误响应
- 结构化日志
- 服务端受控注册入口
- 注册邮箱验证码、邮箱验证回调与重发验证邮件入口
- 邀请码注册、注册 IP 频控和后台注册模式切换
- 高成本接口限流
- 高成本接口每日额度、调用前预占和管理员紧急开关
- 账号状态处置：`disabled`、`generation_limited`、`readonly`
- Origin 校验
- 关键写接口幂等保护
- 用户态数据访问逐步收紧到 RLS / 用户上下文
- 真实 HTTP baseline 和数据库策略脚本验证

这些能力不直接改变学习功能，但显著提升了线上稳定性、安全边界和排查效率。

### 5.5 公开注册与小范围开放准备更完整

产品已经从“内部账号使用”推进到“小范围公开注册可控试放”的准备状态：

- 注册入口支持关闭注册、邀请注册和公开注册三种模式。
- 管理员可以在 `/admin/invites` 生成、停用和调整邀请码，并查看使用记录与最小账号摘要。
- 邮箱注册使用注册页验证码前置校验；项目内验证回调继续承接 Supabase 邮件链接确认，未验证用户不能进入主应用或受保护 API，并可在 `/verify-email` 重发验证邮件。
- 管理员可以通过 `/admin/users` 快速限制异常账号，通过 `/admin` 紧急关闭单个高成本能力。
- 公开前验证已经有统一 runbook 和真实 HTTP baseline 入口。

这不是完整增长或运营后台，但已经覆盖小范围试放最需要的准入、验证、止损和留证路径。

### 5.6 最小业务事件可回看

系统现在可以记录关键客户端业务事件和失败摘要，例如：

- 点击 today continue
- 打开 review
- 提交 review
- 完成 scene 学习
- scene full 音频失败
- 点击音频失败后的替代 CTA

管理端新增 `/admin/observability`，可以在本地浏览器回看最近事件，用于开发、验收和发布前排查。

这不是完整埋点平台，但已经能帮助维护者判断用户在哪一步中断。

### 5.7 上线治理与可观测性收口

最近一轮针对"主流程跑通后"的边际收口已经把上线治理推到内测可放行水准：

- **错误聚合到位**：`@sentry/nextjs` 在生产捕获 5xx，自动附带 requestId tag，事故响应从"等用户截图"降到分钟级
- **启动期 env 自检**：服务启动时打印 `[boot]` 行,覆盖 sentry / upstash / resend / email / origin / IP 限流 / daily quota / registration mode / CSP mode 10 个关键就绪状态,不打印 secret,运维侧一眼可见漂移
- **CSP 双轨**:`Content-Security-Policy-Report-Only` 已上,违规通过 `/api/csp-report` 收集并送 Sentry;通过 `CSP_ENFORCE=true` env 一键切正式 enforce,无需改代码
- **CI 工作流**:GitHub Actions 在 PR / push main 跑全套护栏(lint / mojibake / unit + scripts test / openspec validate / maintenance:check),防止规范漂移
- **合规占位**:`/privacy` / `/terms` 占位页面 + 注册同意条款 checkbox,小范围邀请熟人内测的合规底线已有
- **部署可重复性**:`deploy/nginx.example.conf` + `proxy_common` + 启用步骤文档,生产部署直接 cp 即用,降低首次部署出错率
- **灾备 / 事故响应文档**:`disaster-recovery.md` 覆盖 RPO / RTO / 4 类事故响应;`incident-response-runbook.md` 含 Nginx / 腾讯云 WAF / Cloudflare 启用顺序与 4 类剧本
- **当日用量出口**:`pnpm run usage:snapshot` 把高成本能力用量(3 段:汇总 / 明细 / JSON)从 `/admin/status` 主动暴露出来,可被 PM2 cron 调度

这些能力的目标不是"做平台",而是把"主流程跑通"和"出事能止住"之间的距离缩短到内测阶段可接受的水平。

### 5.8 错误响应更可解释,接口契约更一致

最近一轮针对 API 错误响应做了一次系统性收口(参考 `docs/dev/api-error-response-audit-2026-05-16.md`),把分散在 58 个 route + admin server action 里的 3 处响应契约破坏(P0)+ 10 处高成本路径可观测性缺失(P1)一次性补齐:

- **全量错误响应统一三件套**:所有 API 错误响应都带 `code + details + requestId`,前端可以稳定解析错误结构,遇到 401 / 403 / 429 / 500 都能拿到稳定字段判定下一步行为
- **`auth/logout` 失败语义修正**:Supabase signOut 失败时从 500 `INTERNAL_ERROR` 改为 401 `AUTH_UNAUTHORIZED`,前端能更准确识别 "session 已失效,需要重新登录" 而不是 "服务器故障";响应仍包含 `requestId` 便于维护者排查
- **高成本路径异常按业务模块进入 Sentry**:scene 生成 / 表达 AI 补全 / 相似表达生成 / 场景 mutate 等 7 处高成本路径的未知异常都会通过 `logApiError(module, error, { request })` 注入 module 关联,事故后能按业务模块过滤,而不是只看到一堆没有上下文的 5xx
- **CSP 违规上报端点错误返回也带 code / requestId**:内部工具排查时不再撞到孤立的 plain `{error}` 响应,与其它 API 错误格式对齐

这些修复对成功路径无影响,但显著提升了前端错误处理逻辑的稳定性和后端事故定位的速度。

### 5.9 TTS 预热与后台播放更稳

针对推荐场景首句要"现场加载"和后台预热"切场景不打断"两个老问题做了一轮收口:

- **Today 推荐场景后台预热**:Today 页加载完后会在后台预热前 2 个推荐场景的第 1 句音频;用户点开推荐卡片时通常能直接开始播放,不再需要等待远端 TTS 生成
- **切场景自动取消旧预热**:切换场景时旧场景排队和正在加载的预热会被自动打断(`cancelWarmupsBySceneSlug`),节省用户带宽,避免在新场景里和当前播放抢资源
- **失败 cooldown 阶梯**:sentence / chunk 音频失败后采用 `5 秒 → 1 分钟 → 5 分钟 → 30 分钟` 的冷却阶梯,避免疑难内容反复打上游 TTS;scene full 仍走自己的 45 秒独立冷却
- **签名 URL 缓存跨 worker 共享**:服务端 TTS 签名 URL 缓存升级为 Upstash Redis 优先 + 内存 fallback;之前是进程内 Map,PM2 多 worker 模式下每个 worker 各自一份,命中率被打折;Upstash 不可用时退化保持本地零配置

### 5.10 会话健壮性

- **旧 session 自动清理**:切换 Supabase 项目、轮换密钥或被后台手动登出之后,浏览器残留的旧 session cookie 不再让主应用崩在登录态判断,应用会自动清掉旧 cookie 并把用户引导回登录页。这避免了一类典型的 "上次能用今天打不开" 体验。

### 5.11 匿名试用 + 分享链接灰度入口

注册环节(邀请码 + 邮箱验证码 + 设密码)对潜在用户是高摩擦动作,首次访问流失明显;同时分享场景链接给非用户,对方只能看到登录墙,无法感知场景化学习的核心价值。本轮在试用与分享路径开了一个受控匿名试用口,让访客不注册也能体验"场景列表 -> 场景详情 -> 听句子 -> 看短语详情 -> 固定练习 / 固定变体"的受控闭环:

- **入口前缀严格隔离**:`/trial` 渲染公开试用场景列表,`/trial/scene/[slug]` 与 `/share/scene/[slug]` 渲染匿名只读详情;主入口 `today / scenes / scene / review / chunks / progress` 在 middleware 显式守护,匿名访问被强制重定向到 `/login`,主链路语义零变化
- **身份与配额四道防线**:localStorage UUID + `X-Anonymous-Id` 头识别访客身份;`SHA256(ip + 每日轮换 salt)` 做 IP 维度防绕过;Redis 命名空间 `anon:quota:*` 跟已登录用户配额完全分离;实时 AI 默认注册阻断,TTS 预生成播放 30 次/天 单访客
- **可读但不可写**:Supabase RLS 公开内容表(scenes / scene_variants / chunks / phrases is_public/builtin)显式 anon SELECT;所有用户态表(profiles / user_*) 默认 deny 兜住,加 SQL parser audit test 守护这一不变量
- **三层注册引导漏斗**:L1 顶栏常驻配额条 + L2 内联卡片(完成首场景后) + L3 阻断弹窗(触发禁用功能 / 配额用尽);8 个漏斗事件按 server / client 两路上报,daily cron 聚合到 `daily_anon_cost_report`,匿名 → 注册转化率和单转化成本可量化
- **TTS 接匿名走独立 quota**:`tts_play` 只读 Storage 已存在的 mp3,无上游付费成本,不进 `HighCostCapability` 体系,新建独立 `tts-playback-quota` 模块,配额体系完全分家
- **AI / 写入只做样子**:固定练习 / 固定变体可本地体验;导入、生成、重新生成、提交、保存、加入复习、实时 AI 解释等入口只展示外观,点击统一弹注册阻断,不触发上游模型或业务写入
- **一键止血**:`ALLOW_ANONYMOUS_TRIAL=false` 切回 → 所有 `/trial/*` 与 `/share/*` 路径退回 `/login`,既有登录用户主链路零影响

这套能力的目标不是"做匿名学习",而是把转化漏斗第一步从"看到登录墙"改成"先试用再决定",同时让匿名流量的 TTS 成本可控、可观测、可紧急关闭。

## 6. 产品整体结构

当前产品可以理解为五层结构。

### 6.1 Today：每日任务入口层

负责回答：

- 今天最应该先做什么
- 当前学习卡在什么位置
- 今天还有哪些下游任务可做
- 为什么当前优先推荐这一步

它本质上是任务编排层，不是内容生产层。

### 6.2 Scene：主学习流程层

负责承载一次完整的场景学习。

用户在这里完成：

- 阅读和听熟场景
- 查看重点表达
- 保存表达
- 进入练习
- 解锁变体
- 推进学习进度

这是产品的主流程层。

### 6.3 Chunks / Phrases：表达资产层

负责把用户在学习过程中沉淀出的表达转成长线资产。

核心能力包括：

- 搜索和筛选表达
- 查看详情
- AI 补全
- 相似 / 对照表达维护
- expression cluster 管理
- 进入复习链路

这是产品的资产中台。

### 6.4 Review：复习和输出训练层

负责把沉淀后的表达重新拉回复习和输出链路。

当前 `review` 的价值不只是“记得 / 不记得”，而是逐步收集：

- 是否识别
- 是否有主动输出信心
- 是否完成完整输出
- 是否需要更早回流

这些信号会影响后续调度。

### 6.5 Progress / Admin：反馈与维护层

`progress` 负责展示长期学习表现和概览统计。当前页面重点展示学习节奏、表达资产、复习正确率、最近 7 天学习趋势、学习热力图和成长足迹，强调“持续沉淀”而不是单纯完成任务数量。

`admin` 负责支撑维护、音频重生成、缓存观察和业务事件回看，让产品不仅能用，也能被持续维护。管理端当前已经形成较统一的内容维护工作台体验，首页入口、列表、筛选、菜单、表单、时间展示和操作按钮保持同一套交互层级。

最近几轮后，`admin` 也承担了公开注册前的最小运营处置：邀请码管理、注册模式切换、用户状态处置、高成本紧急开关和运行状态检查。这些能力的目标不是替代完整运营后台，而是让小范围开放时能快速发放、观察、降级和关闭入口。

### 6.6 Anonymous Trial / Share：匿名灰度入口层（可选）

`/trial`、`/trial/scene/[slug]` 与 `/share/scene/[slug]` 是匿名灰度路径，跟主应用其它路径**完全隔离**：

- `/trial` 展示公开试用场景列表，由访客自己选择进入详情
- `/trial/scene/[slug]` 与 `/share/scene/[slug]` 不要求注册即可读到场景内容
- 提供"看 + 听句子 + 看短语详情"只读闭环，但禁用所有写入与实时 AI 能力
- 由 `ALLOW_ANONYMOUS_TRIAL` env 总开关控制开放；关闭时退回 `/login`
- 不进入 today / 主推荐 / 进度聚合；匿名访客的学习状态走 sessionStorage，关浏览器即清

它的作用是"转化漏斗第一步"——把"看到登录墙"改成"先试用再决定"，同时保持主入口边界不变。

## 7. 当前核心学习闭环

当前主闭环可以概括为：

1. 用户进入 `today`
2. 系统推荐继续哪个 `scene` 或今天先做什么
3. 用户进入 `scene` 做主学习
4. 在场景中保存表达，沉淀到 `chunks / phrases`
5. 表达进入 `review`，产生正式训练信号
6. 服务端聚合学习状态、复习结果和长期概览
7. 用户回到 `today`，系统继续重新组织入口

一句话表示：

> Today 分发任务 -> Scene 完成主学习 -> Phrases 沉淀资产 -> Review 写回信号 -> Learning Service 聚合 -> Today 再次编排入口

## 8. 主要页面与用户价值

### 8.1 Today 页面

用户价值：

- 快速开始今天的学习
- 找到上次中断的位置
- 看到 review 和表达沉淀任务
- 理解当前为什么优先做这一步

### 8.2 Scenes 列表页

用户价值：

- 新用户可以从 Start Here 入门路径直接开始第一个日常场景
- 选择、筛选和排序要进入的学习场景
- 快速继续上次中断的学习内容
- 在需要时导入或生成新的学习场景
- 对已学到一定进度的场景进行循环复习，在后台或锁屏场景下尽量保持连续听熟

### 8.3 Scene 详情页

用户价值：

- 在真实语境中完成主学习
- 从阅读走向练习与变体
- 在学习中提取值得反复学习的表达
- 音频失败时仍能通过逐句跟读继续推进

### 8.4 Chunks / Phrases 页面

用户价值：

- 管理自己的表达资产
- 继续补全、整理、分类和关联表达
- 把零散学习痕迹变成长期资产

### 8.5 Review 页面

用户价值：

- 把已经学过的表达重新拉回回忆链路
- 形成正式训练信号
- 根据掌握情况调整后续复习节奏
- 完成一轮后得到清晰反馈和下一步入口

### 8.6 Progress 页面

用户价值：

- 看到近期学习表现
- 理解自己是否保持持续学习
- 感知长期积累是否在增长
- 通过热力图、最近 7 天趋势和成长足迹理解自己的学习节奏

### 8.7 Admin 页面

维护价值：

- 管理 TTS 音频和批量重生成
- 查看浏览器端业务事件和失败摘要
- 支撑发布前验收和问题排查
- 通过统一列表、筛选和操作按钮降低维护操作的误用成本
- 慢请求或跳转时通过全局 loading 和按钮处理中状态降低重复操作风险

### 8.8 匿名试用 / Share 灰度入口页

用户价值（针对未注册访客）：

- 从首页试用或分享链接进来不被登录墙挡住，先看到场景列表或直接读到场景内容（标题、句子、中英对照）
- 点句子可以听预生成 TTS 音频，体会真实语境的听感
- 点击句子可以查看和真实场景详情一致的短语详情面板
- 知道注册后可以解锁什么（顶栏配额条 + 内联卡片 + 阻断弹窗三层引导）

边界：

- 固定练习 / 固定变体可本地体验；任何写入、实时生成、导入、重新生成、提交或实时 AI 动作都禁用，触发即弹"注册解锁"
- AI 表达解释默认匿名禁用，直接弹 L3 注册阻断
- TTS 30 次/天 单访客上限，到上限弹 L3 阻断
- 学习态不持久化，关浏览器即清；注册后从零开始

## 10. 当前优化点

### 10.1 体验优化

- `today` 提供更明确的推荐原因，减少用户选择成本。
- `/scenes` 从管理列表升级为学习入口，新用户优先看到 Start Here 和可直接执行的学习 CTA。
- `review` 提交后给出更具体的结果反馈，而不是只提示成功。
- `review` 队列清空后提供收束反馈和返回 `today` 的下一步。
- `scene full` 音频失败后提供逐句跟读替代入口。
- 场景、音频和学习入口尽量通过预热和缓存降低等待感。
- scenes 循环复习优先播放已准备的 review pack，让用户离开页面或锁屏后仍尽量能连续听熟已学场景；页面会说明本次循环包含的场景，避免把整体音频包误解成实时切换的单场景播放。

### 10.2 性能与稳定性优化

- 场景进入前预取关键数据。
- 音频使用缓存、预热和失败降级。
- scenes 列表复习使用 review pack，把一组场景的可播放内容合并为单个循环音频资源，减少后台播放对页面 JS 切换下一段的依赖。
- 高成本接口加入限流，避免短时间重复请求放大成本。
- 高成本接口加入每日额度、预占和管理员紧急开关，异常时能在上游调用前止损。
- 注册入口加入邀请准入、邮箱验证码、邮箱验证、IP 频控和后台模式切换，避免公开试放直接变成无门槛开放。
- 关键写接口加入幂等保护，降低重复点击和网络重试风险。
- 真实 HTTP baseline 用于验证入口链路，而不仅是 handler 层自测。

### 10.3 安全与可维护性优化

- 请求带 `requestId`，方便排查。
- 错误响应结构更统一。
- 写接口有 Origin 校验。
- 用户态数据访问逐步回到用户上下文和 RLS。
- 后台权限入口显式白名单化。
- 管理后台提供最小邀请码、用户状态和高成本能力处置入口。
- 发布前检查覆盖安全头、限流、Origin、TTS 异常和主链路验收。

## 11. 当前产品优势

- 主线清晰：`today -> scene -> phrases -> review/progress`
- 学习不只停留在浏览，而是有沉淀和回流
- 表达资产不是平铺存储，而是有关系和 cluster 组织
- 学习状态由服务端维护，支持恢复和长期聚合
- today 承担学习编排角色，降低用户每天开始学习的成本
- 音频链路有缓存、预热和失败降级，体验更稳
- 后端治理和发布检查逐步补齐，产品更接近可持续上线状态
- 文档和 OpenSpec 流程已经能支撑后续迭代可追踪
- 上线治理已具备可观测性 / 自检 / CI / 灾备 / 事故响应五个最小入口,内测可放行

## 12. 当前产品边界

当前产品已经形成闭环，但仍处于“主链路可用，体验持续增强”的阶段。

当前边界包括：

- `today` 的推荐策略仍可以更个性化。
- `review` 的训练反馈还可以继续细化，例如更强的输出质量分析。
- 业务事件目前是本地可回看,**生产侧已经接入 Sentry 错误聚合**,但跨设备业务事件聚合和正式 retention 类用户结果指标仍未平台化。
- 音频链路已具备降级，但上游连接稳定性仍需要持续观察。
- `chunks / phrases` 是重型工作台，后续仍有继续简化和运营化空间。
- 公开注册已有小范围试放所需的准入和处置基线，但完整风控、邮件投递监控、WAF、增长漏斗和运营后台仍未平台化。
- 安全治理已有基线**(含 CSP report-only / 启动自检 / CI 护栏)**,但完整熔断、全量 OpenAPI、BI 看板、CSP enforce 的平台化能力暂未做。
- 匿名试用仅在 `/trial`、`/trial/scene/[slug]` 与 `/share/scene/[slug]` 灰度，主入口 today / scenes / scene / review / chunks / progress 仍要求登录；匿名 → 注册数据迁移（学习态、表达保存关联到新账号）属于本期不收项，等灰度数据验证后再起 V2。

所以它当前最准确的定位是：

> 一个已经跑通主学习闭环、具备表达资产沉淀能力，并开始形成复习、反馈、音频体验和上线治理体系的英语学习产品。

## 13. 后续演进方向

优先方向：

- 继续强化 `today` 的推荐策略和任务解释能力（当前主要基于 starter path / continue learning,可引入用户表达资产维度推荐相关场景,直接强化"迁移")。
- 增强 `review` 的训练反馈细粒度和调度智能度(调度算法本身已经消费 `recognitionState` / `outputConfidence` / `fullOutputStatus` / `variantRewriteStatus` / `fullOutputCoverage` 5 类正式信号,下一步是加输出质量评估)。
- 让业务事件从本地回看 + Sentry 错误聚合升级到 retention 类用户结果指标(7/14/30 天表达回忆率 / 主动使用率)。
- 继续优化 `scene` 中训练颗粒度、音频稳定性和完成反馈。
- 强化 `chunks / phrases` 的表达资产运营能力。

可以暂缓：

- 全量 BI 看板
- 完整 OpenAPI 平台化
- 完整离线 / PWA 能力
- 复杂多实验推荐系统
- 重型埋点平台

## 14. 一页版总结

AbandonClaw 的核心不是“做了几个学习页面”，而是把英语学习里的几个关键环节串成了一条连续链路：

- 用 `today` 降低每天开始学习的决策成本。
- 用 `scene` 提供真实语境中的主学习流程。
- 用 `phrases / chunks` 把表达沉淀成长期资产。
- 用 `review` 把资产重新拉回复习和输出训练。
- 用 `progress` 和聚合服务给出长期反馈。
- 用音频缓存、失败降级和接口治理保证体验稳定。
- 用业务事件回看和验收清单支撑持续维护。

当前最有价值的地方不是单一页面或单一 AI 能力，而是：

> 它已经具备“场景学习 -> 表达沉淀 -> 正式回写 -> 聚合反馈 -> 再次进入”的完整产品骨架，并开始补齐体验稳定性、后端治理和可观测性。

## 15. 北极星实现进度

回到 §1 的产品北极星：

> 让每一次场景学习，都沉淀为用户在未来真实场景中能回忆、能使用、能迁移的表达资产。

按三个维度看当前实现：

### 15.1 回忆（用户离开场景后能主动想起）

**实现度:80%**

- 表达资产沉淀链路完整(`Scene -> Chunks -> Review`)
- Review 已经按 SM-2 类调度 + 5 类正式信号(`recognitionState` / `outputConfidence` / `fullOutputStatus` / `variantRewriteStatus` / `fullOutputCoverage`)综合计算 due 优先级和下次复习时间
- 调度算法已经能"低输出信心 / 未完成完整输出 / 仅识别" 优先,`schedulingFocus` 字段透传到 UI 解释"为什么这条优先出现"
- 还差:**retention 类用户结果指标**(7/14/30 天表达回忆率),用于量化"回忆"是否真的发生

### 15.2 使用(遇到相似需求能说出/写出/选择)

**实现度:50%**

- Review 已经按阶段分:识别 → 主动回忆 → 变体迁移 → 完整输出
- 完整输出目标表达覆盖结果(`fullOutputCoverage`)已经作为正式信号写回
- Today / Chunks 能呈现"已沉淀的表达"
- 还差:**输出质量评估**(用户写出来的句子是否真贴合表达,目前依赖用户自评)、**AI 口语对话**作为输出训练入口

### 15.3 迁移(换人/主题/语气/场景能改写)

**实现度:30%**

- `expression cluster` + `相似表达` 提供组织手段
- 变体改写完成状态(`variantRewriteStatus`)已纳入正式信号 + 调度
- Scene variant 能力存在但生成成本高
- 还差:**主动驱动的迁移训练**(在新语境中改写应用、跨场景表达调用、推荐"含相关表达的新场景")— 这是当前最大短板,也是 `Today` 推荐引入"用户表达资产维度"后能直接强化的方向

### 15.4 北极星 vs 实现的总结

| 维度 | 架构骨架 | 信号收集 | 信号消费 | 用户结果量化 |
| --- | --- | --- | --- | --- |
| 回忆 | ✅ | ✅ | ✅ | ❌ |
| 使用 | ✅ | ✅ | 部分 | ❌ |
| 迁移 | 部分 | 部分 | ❌ | ❌ |

**判断**:产品骨架已经把"为达成北极星该有的能力"建好了,但"我们达成了多少"还没有指标系统来回答。下一阶段最高 ROI 不是再加新功能,是 **a)** 加 retention 类用户结果埋点,**b)** 把 Today 推荐接入用户表达资产维度补齐"迁移"短板。
