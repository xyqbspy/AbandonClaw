# Domain Rules

本目录用于维护“领域规则”文档。

领域规则回答的是：

- 系统为什么这样判定
- 哪些条件会改变学习语义
- 哪些字段才算稳定证据或正式信号
- 哪些消费方必须共用同一套解释

如果你要理解“规则边界”和“判定标准”，先看这里；如果你要理解字段来自哪里、落在哪、页面如何映射，再看 `docs/system-design/`。

## 核心术语

- `稳定证据`
  - 能被服务端正式落库、聚合并被多个页面稳定消费的学习信号
- `正式信号`
  - 已进入后端正式记录模型的字段，不等于前端临时阶段或草稿
- `聚合摘要`
  - 服务端基于正式信号生成的稳定消费字段，供 `today` / `progress` / `review` 等页面使用
- `调度信号`
  - 会影响复习排序、间隔或优先级的正式字段或聚合结果

## 推荐使用方式

- 判断某个状态是否算“真正完成”
  - 先看这里
- 判断页面能不能直接消费某个字段
  - 先确认它是不是稳定证据、正式信号或聚合摘要
- 如果问题已经变成“这些字段具体落在哪里”
  - 转去 `docs/system-design/`

## 使用原则

出现这些情况时，应优先补或改 `domain-rules`：

- 学习完成判定变化
- 稳定证据边界变化
- 正式信号语义变化
- 调度优先级或状态推进规则变化

以下内容通常不应写在这里：

- 页面字段映射
- 服务端 fallback 结构
- 数据表位置或迁移脚本说明
- 组件拆分和实现文件组织

## 与其它文档的边界

- `domain-rules/`
  - 解释判定标准、稳定证据和正式语义边界
- `system-design/`
  - 解释这些规则最终如何被服务端、数据表和页面实现消费
- `feature-flows/`
  - 解释这些规则嵌入主链路后，实际在哪些节点生效

## 建议正文模板

新增或重写 `domain-rules` 文档时，优先按这组章节组织：

1. 目标
2. 基本原则
3. 当前规则定义
4. 消费边界
5. 维护约束
6. 改动时一起检查
